# 이벤트 스토밍 전시 개발팀 가이드

## 1. 개요

### 1.1 이 가이드의 목적

전시 개발팀을 대상으로 이벤트 스토밍 워크샵을 진행할 때, **데이터 소비자/조합자로서의 특수한 역할**에서 발생하는 판단 난제에 대한 실전 가이드입니다.

```
┌─────────────────────────────────────────────────────────────┐
│              전시팀 워크샵의 핵심 도전 5가지                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. API 어그리게이션이 이벤트인가?                         │
│     "상품+추천+배너 조합하는 건 이벤트인가요?"             │
│                                                             │
│  2. 캐시 갱신/배치 처리의 이벤트 여부                      │
│     "Redis 캐시 갱신은 이벤트인가요?"                      │
│                                                             │
│  3. 데이터 소비자의 이벤트 소유권                          │
│     "상품팀 API 호출 결과는 누구 이벤트인가요?"            │
│                                                             │
│  4. 개인화 vs 공통 전시 데이터 구분                        │
│     "사용자별 추천은 별도 이벤트인가요?"                   │
│                                                             │
│  5. 렌더링 준비 vs 비즈니스 이벤트                         │
│     "화면 데이터 조합 완료는 이벤트인가요?"                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 전시팀의 특수성

> 전시팀은 다른 개발팀과 달리 **데이터 생산자가 아닌 데이터 소비/조합자**입니다.
> 이 특성을 이해해야 이벤트 스토밍에서 올바른 판단을 할 수 있습니다.

```
┌─────────────────────────────────────────────────────────────┐
│                 전시팀 vs 다른 팀 비교                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  일반적인 개발팀 (상품팀, 주문팀 등):                      │
│  ┌────────────────────────────────────────────────────┐    │
│  │  데이터 생산 → 비즈니스 로직 처리 → 이벤트 발행   │    │
│  │  예: "상품이 등록되었다", "주문이 접수되었다"       │    │
│  └────────────────────────────────────────────────────┘    │
│                                                             │
│  전시 개발팀:                                              │
│  ┌────────────────────────────────────────────────────┐    │
│  │  데이터 수신 → 조합/변환 → 화면용 데이터 구성      │    │
│  │  예: 상품+추천+배너+프로모션 → 홈 화면 데이터      │    │
│  └────────────────────────────────────────────────────┘    │
│                                                             │
│  ─────────────────────────────────────────────────────────  │
│                                                             │
│  전시팀의 주요 역할:                                       │
│  • 화면 출력을 위한 API 어그리게이션 (여러 서비스 조합)   │
│  • 전시 데이터 구성 (상품 목록, 배너, 레이아웃)           │
│  • 개인화/추천 노출 (사용자별 맞춤 데이터)                │
│  • 캐싱 및 성능 최적화                                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 2. 데이터 소비자 관점의 이벤트 스토밍

### 2.1 데이터 생산자 vs 소비자의 이벤트 소유권

```
┌─────────────────────────────────────────────────────────────┐
│               이벤트 소유권 핵심 원칙                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  핵심 질문: "누가 데이터의 원본을 관리하는가?"             │
│                                                             │
│  ─────────────────────────────────────────────────────────  │
│                                                             │
│  데이터 생산자 (원본 소유팀):                              │
│  • 상품팀: "상품이 등록되었다" 발행                        │
│  • 검색팀: "검색 결과가 반환되었다" 발행                   │
│  • 추천팀: "추천 목록이 생성되었다" 발행                   │
│  • 프로모션팀: "배너가 활성화되었다" 발행                  │
│                                                             │
│  ─────────────────────────────────────────────────────────  │
│                                                             │
│  데이터 소비자 (전시팀):                                   │
│  • 원본 데이터 이벤트 → 발행하지 않음 ❌                   │
│  • 조합/변환 작업 → 대부분 이벤트 아님 ❌                  │
│  • 전시 구성 변경 → 비즈니스 의미 있으면 이벤트 ✅        │
│                                                             │
│  ─────────────────────────────────────────────────────────  │
│                                                             │
│  전시팀이 이벤트를 발행하는 경우:                          │
│  • 🟧 "메인 홈 전시 구성이 변경되었다" (MD가 레이아웃 변경)│
│  • 🟧 "개인화 추천 배치가 완료되었다" (배치 처리 결과)    │
│  • 🟧 "품절 상품이 전시에서 제외되었다" (비즈니스 규칙)   │
│  • 🟧 "프로모션 배너 노출이 시작되었다" (예약 시간 도달)  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 소비자 관점 이벤트 판단 기준

```
┌─────────────────────────────────────────────────────────────┐
│              전시팀 이벤트 판단 3단계 체크                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  □ 1. 다른 팀/시스템이 이 변화를 알아야 하는가?           │
│       → 아니오: 내부 기술 처리 (이벤트 아님)               │
│       → 예: 다음 질문으로                                  │
│                                                             │
│  □ 2. 비즈니스적으로 의미 있는 상태 변화인가?             │
│       (기술적 캐싱/조합이 아닌 비즈니스 결과)              │
│       → 아니오: 기술적 처리 (이벤트 아님)                  │
│       → 예: 다음 질문으로                                  │
│                                                             │
│  □ 3. 고객/운영자에게 영향이 있는가?                      │
│       (화면에 보이는 변화, 운영 정책 반영)                 │
│       → 아니오: 🩷 핫스팟 표시 (논의 필요)                 │
│       → 예: 도메인 이벤트! 🟧                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.3 전시팀 데이터 흐름 이해

```
┌─────────────────────────────────────────────────────────────┐
│                   전시팀 데이터 흐름                        │
└─────────────────────────────────────────────────────────────┘

        [데이터 원본팀들]                    [전시팀]

    ┌─────────────┐
    │  🟩 상품팀   │──┐
    │  상품 정보   │  │
    └─────────────┘  │
                     │      ┌─────────────────────────┐
    ┌─────────────┐  │      │                         │
    │  🟩 검색팀   │──┼──────│   API 어그리게이션      │
    │  검색 결과   │  │      │   (데이터 조합/변환)    │
    └─────────────┘  │      │                         │
                     │      │   ❌ 대부분 이벤트 아님  │
    ┌─────────────┐  │      │                         │
    │  🟩 추천팀   │──┼──────│   ─────────────────── │
    │  추천 목록   │  │      │                         │
    └─────────────┘  │      │   ✅ 비즈니스 의미 있는 │
                     │      │      결과만 이벤트      │
    ┌─────────────┐  │      │                         │
    │ 🟩 프로모션팀│──┘      └─────────────────────────┘
    │  배너/할인   │                    │
    └─────────────┘                    │
                                       ▼
                              ┌─────────────────┐
                              │   화면 데이터    │
                              │   (고객에게 전시)│
                              └─────────────────┘
```

## 3. API 어그리게이션 이벤트 가이드

### 3.1 기본 원칙

```
┌─────────────────────────────────────────────────────────────┐
│  API 어그리게이션 = 여러 서비스 데이터를 조합하여           │
│                     화면에 필요한 형태로 변환하는 것        │
│                                                             │
│  ❌ API 호출 자체는 이벤트가 아님                           │
│  ❌ 데이터 캐싱/저장은 이벤트가 아님                        │
│  ✅ 비즈니스적으로 의미 있는 조합 결과는 이벤트가 될 수 있음│
└─────────────────────────────────────────────────────────────┘
```

### 3.2 어그리게이션 이벤트 판단 플로우차트

```
┌─────────────────────────────────────────────────────────────┐
│              API 어그리게이션 이벤트 판단 흐름              │
└─────────────────────────────────────────────────────────────┘

                    [API 어그리게이션 작업]
                            │
                            ▼
                ┌───────────────────┐
                │ 다른 팀/시스템이  │──Yes──▶ ❌ 이벤트 아님
                │ 알아야 하는가?    │         (내부 기술 처리)
                └───────────────────┘
                        │ No
                        ▼
                ┌───────────────────┐
                │ 비즈니스 상태가   │──No───▶ ❌ 이벤트 아님
                │ 변경되었는가?     │         (단순 데이터 조회)
                └───────────────────┘
                        │ Yes
                        ▼
                ┌───────────────────┐
                │ 고객/사용자에게   │──No───▶ 🩷 핫스팟으로 표시
                │ 영향이 있는가?    │         (논의 필요)
                └───────────────────┘
                        │ Yes
                        ▼
                    ✅ 이벤트!
```

### 3.3 어그리게이션 유형별 이벤트 처리

```
┌─────────────────────────────────────────────────────────────┐
│              어그리게이션 유형별 이벤트 판단                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  유형              │ 설명                 │ 이벤트 여부    │
│  ─────────────────────────────────────────────────────────  │
│  실시간 조합       │ 요청마다 여러 API    │ ❌ 이벤트 아님 │
│                    │ 호출하여 조합        │ (기술 처리)    │
│                    │ 예: 상품상세 페이지   │                │
│  ─────────────────────────────────────────────────────────  │
│  캐시 조합         │ 캐시된 데이터 기반   │ ❌ 이벤트 아님 │
│                    │ 조합                 │ (기술 처리)    │
│                    │ 예: 홈 화면 섹션 조합 │                │
│  ─────────────────────────────────────────────────────────  │
│  배치 전처리       │ 사전에 조합하여 저장 │ ⚠️ 조건부      │
│                    │ 예: 추천 목록 사전   │ 비즈니스 결과면│
│                    │     생성             │ 이벤트         │
│  ─────────────────────────────────────────────────────────  │
│  개인화 조합       │ 사용자별 맞춤        │ ⚠️ 조건부      │
│                    │ 데이터 조합          │ 배치 결과면    │
│                    │ 예: 맞춤 추천 생성   │ 이벤트         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.4 이벤트가 되는 경우 vs 아닌 경우

```
┌─────────────────────────────────────────────────────────────┐

├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ❌ 이벤트 아님 (내부 기술 처리)                           │
│  ├── "상품 API 호출됨"                                     │
│  ├── "추천 API 응답 수신됨"                                │
│  ├── "Redis 캐시 저장됨"                                   │
│  ├── "화면 데이터 조합됨"                                  │
│  ├── "JSON 변환 완료됨"                                    │
│  ├── "CDN 캐시 저장됨"                                     │
│  └── "Elasticsearch 조회 완료됨"                           │
│                                                             │
│  ─────────────────────────────────────────────────────────  │
│                                                             │
│  ✅ 이벤트 (비즈니스 의미 있음)                            │
│  ├── "개인화 추천 목록이 생성되었다" (배치 처리 결과)      │
│  ├── "메인 홈 전시 구성이 변경되었다" (운영자 설정 변경)   │
│  ├── "프로모션 배너 노출이 시작되었다" (예약 시간 도달)    │
│  ├── "상품 전시 순서가 변경되었다" (MD가 순서 조정)        │
│  ├── "품절 상품이 전시에서 제외되었다" (재고 연동 결과)    │
│  ├── "신상품 섹션이 갱신되었다" (배치 처리 결과)           │
│  └── "전시 데이터 조합이 실패했다" (장애 상황)             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.5 구체적 시나리오별 판단

#### 시나리오 A: 메인 홈 화면 로딩

```
┌─────────────────────────────────────────────────────────────┐
│              시나리오 A: 메인 홈 화면 로딩                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  사용자가 앱 홈 화면에 접속하면:                           │
│  1. 상품팀 API → 인기 상품 목록 조회                       │
│  2. 추천팀 API → 맞춤 추천 목록 조회                       │
│  3. 프로모션팀 API → 배너/이벤트 조회                      │
│  4. 검색팀 API → 인기 검색어 조회                          │
│  5. 데이터 조합 → 화면 렌더링                              │
│                                                             │
│  ─────────────────────────────────────────────────────────  │
│                                                             │
│  판단:                                                     │
│  ❌ 1~5번 모두 이벤트 아님                                 │
│     → 사용자 요청에 대한 실시간 데이터 조합                │
│     → 다른 팀이 알 필요 없음                               │
│     → 비즈니스 상태 변경 없음                              │
│                                                             │
│  ─────────────────────────────────────────────────────────  │
│                                                             │
│  예외 - 이벤트가 되는 경우:                                │
│  ✅ "메인 홈 레이아웃이 변경되었다" (MD가 섹션 순서 변경)  │
│  ✅ "홈 배너가 교체되었다" (예약된 배너 자동 교체)         │
│  ✅ "긴급 공지가 노출되었다" (운영자 긴급 설정)            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 시나리오 B: 캐시 갱신

```
┌─────────────────────────────────────────────────────────────┐
│              시나리오 B: 캐시 갱신                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
```
