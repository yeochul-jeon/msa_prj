### 7.1 워크샵 오프닝 스크립트

```
┌─────────────────────────────────────────────────────────────────┐
│             물류 시스템 워크샵 오프닝 스크립트                    │
│             (협력사/계열사 담당자 대상 — 이벤트 스토밍 간략 설명) │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  "안녕하세요, 오늘 물류 시스템 이벤트 스토밍 워크샵을            │
│  진행할 [이름]입니다.                                           │
│                                                                 │
│  먼저 이벤트 스토밍이 뭔지 간단히 설명드리겠습니다.             │
│  이벤트 스토밍은 비즈니스 프로세스에서 일어나는                 │
│  '중요한 사건(이벤트)'을 포스트잇에 적어서                      │
│  시간 순서대로 벽에 붙이는 워크샵입니다.                        │
│                                                                 │
│  포스트잇 색깔마다 의미가 다릅니다:                             │
│  • 🟧 주황색: 이벤트 (일어난 사건, 과거형)                     │
│    예: '출하지시가 생성되었다'                                  │
│  • 🟦 파란색: 커맨드 (명령, 누가 뭘 했는지)                    │
│    예: '출하지시 생성하기'                                      │
│  • 🟨 노란색: 애그리게잇 (변경되는 데이터 묶음)                │
│    예: '출하지시', '배송정보'                                   │
│  • 💜 보라색: 정책 (자동으로 적용되는 규칙)                     │
│    예: '센터배송여부에 따라 WMS/택배사 분기'                    │
│  • 🩷 분홍색: 핫스팟 (논의 필요, 모르겠는 부분)                │
│  • 🟩 초록색: 외부 시스템 (우리 팀 밖의 시스템)                │
│    예: '기간계(OR)', '택배사', 'SAP'                            │
│                                                                 │
│  시작하기 전에 몇 가지 부탁드릴 점이 있습니다:                  │
│                                                                 │
│  ① 오늘은 6개 서비스를 다루는데, 각 서비스별로 이벤트를         │
│     구분해주세요. '이 이벤트는 배송관리인가, 택배사연동인가?'   │
│     라고 생각해보시면 됩니다.                                   │
│                                                                 │
│  ② 기간계(주문/상품)의 이벤트는 초록색으로 표시합니다.          │
│     우리(물류)의 이벤트와 구분하는 게 중요합니다.               │
│     '주문이 완료되었다'는 기간계 → 초록색                       │
│     '출하지시가 생성되었다'는 물류 → 오렌지색                   │
│                                                                 │
│  ③ EAI 인터페이스명이 아닌 비즈니스 용어로 적어주세요.          │
│     'IF_FU_SHIP_ORD_001 호출'이 아니라                          │
│     '출하지시가 생성되었다'로 적어주세요.                       │
│                                                                 │
│  ④ 모르겠는 건 핫스팟(분홍색)으로 표시하면 됩니다.              │
│     특히 기간계와 물류의 경계가 애매한 부분은                   │
│     핫스팟으로 표시하고 함께 논의합시다.                        │
│                                                                 │
│  완벽한 정답은 없습니다. 편하게 진행해주세요!"                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

```
┌─────────────────────────────────────────────────────────────────┐
│     상황: EAI 인터페이스와 도메인 이벤트 혼동                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  대응:                                                          │
│  "좋은 질문입니다! EAI 인터페이스와 이벤트는 다릅니다.          │
│                                                                 │
│  EAI는 '어떻게 전달하느냐'이고,                                 │
│  이벤트는 '무슨 일이 일어났느냐'입니다.                         │
│                                                                 │
│  비유하면, 택배(EAI)는 배송 수단이고,                           │
│  편지 내용(이벤트)은 '무슨 소식'인지입니다.                     │
│                                                                 │
│  예시:                                                          │
│  • IF_FU_SHIP_ORD_001 (EAI) → '출하지시가 생성되었다' (이벤트) │
│  • IF_WMS_OUT_CONFIRM (EAI) → '출고처리가 완료되었다' (이벤트)  │
│  • IF_SAP_POSTING (EAI) → 'SAP 전표 posting' (외부 시스템)     │
│                                                                 │
│  하나의 이벤트가 여러 EAI를 트리거할 수 있고,                   │
│  하나의 EAI가 여러 이벤트의 결과일 수도 있습니다.               │
│                                                                 │
│  비즈니스 관점에서 '무슨 일이 일어났다'를 적어주세요."          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### "WMS 출고확정은 우리 이벤트인가 WMS 이벤트인가?"

```
┌─────────────────────────────────────────────────────────────────┐
│     상황: WMS 외부 솔루션과의 이벤트 경계 질문                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  대응:                                                          │
│  "WMS 솔루션 내부에서 피킹/패킹하는 건 WMS 내부입니다.          │
│                                                                 │
│  하지만 WMS에서 '출고 완료'를 우리 시스템에 전달하는 순간,      │
│  그건 우리(물류 시스템)의 이벤트가 됩니다.                       │
│                                                                 │
│  원칙은 '경계를 넘는 시점'입니다:                                │
│  • WMS 내부 (피킹, 패킹, 적재) → 이벤트 아님                   │
│  • WMS → 물류시스템 전달 → 🟧 WMS 이벤트                       │
│  • 물류시스템 → WMS 전달 → 🟧 FU 이벤트                        │
│                                                                 │
│  정리하면:                                                      │
│  🟧 '출고지시가 접수되었다' — FU가 WMS에 요청한 시점            │
│  🟧 '출고처리가 완료되었다' — WMS가 완료를 전달한 시점          │
│  이 두 가지만 이벤트로 적어주세요."                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### "반품접수와 회수지시의 경계는?"

```
┌─────────────────────────────────────────────────────────────────┐
│     상황: 반품접수(기간계)와 회수지시(물류)의 경계 질문          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  대응:                                                          │
│  "이건 물류 워크샵에서 가장 중요한 경계입니다.                   │
│                                                                 │
│  반품접수: 고객이 '반품하겠다'고 신청하는 것                    │
│  → 기간계(OR)가 처리 → 🟩 초록색                               │
│                                                                 │
│  회수지시: 물류가 '택배사에 가서 가져와라' 하는 것              │
│  → 물류(RTN)가 처리 → 🟧 오렌지색                              │
│                                                                 │
│  비유하면:                                                      │
│  반품접수 = 고객센터에서 '반품하겠습니다' 접수                  │
│  회수지시 = 물류센터에서 '택배기사님, 수거해주세요'             │
│                                                                 │
│  두 개는 다른 시스템, 다른 이벤트입니다:                        │
│  🟩 '반품이 접수되었다' (기간계 OR)                              │
│     ↓                                                           │
│  🟧 '회수지시가 생성되었다' (물류 RTN)                          │
│     ↓                                                           │
│  🟧 '회수가 확정되었다' (물류 RTN)                              │
│     ↓                                                           │
│  🟩 '환불이 처리되었다' (기간계 OR)"                             │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 9.1 자주 묻는 질문

#### 워크샵 준비 관련

```
Q1: "협력사 담당자가 이벤트 스토밍을 처음 접하는데 괜찮은가요?"
A: 워크샵 오프닝에서 10분 정도 간략 설명을 포함합니다.
   포스트잇 색상과 의미만 알면 됩니다.
   사전에 간단한 소개 자료와 6개 서비스 개요 1장짜리를 공유하면
   더 원활하게 진행할 수 있습니다.

Q2: "6개 서비스를 한 번에 다 해야 하나요?"
A: Option A(3일, 프로세스축별 분할)를 권장합니다.
   배송→반품→정산 프로세스 흐름을 따라가면서
   자연스럽게 서비스 간 연동도 잡을 수 있습니다.

Q3: "기간계(주문/상품) 담당자도 참석해야 하나요?"
A: Day 1(배송관리) 세션에 기간계 담당자 1명이 참석하면 좋습니다.
   기간계→물류 경계(주문완료→배송정보전달)가 가장 중요한 시작점이기
   때문입니다. 참석이 어려우면 사전에 연동 포인트를 정리해두세요.

Q4: "WMS 솔루션 업체도 참석해야 하나요?"
A: Day 1 WMS 세션에 참석하면 좋지만 필수는 아닙니다.
   WMS 솔루션과의 경계(물류시스템 ↔ WMS 내부)를 미리 정리해두면
   WMS 담당자만으로도 진행 가능합니다.

Q5: "사전에 어떤 자료를 준비해야 하나요?"
A: ① 6개 서비스 개요 1장짜리 요약 (참석자 공유)
   ② 배송 프로세스 흐름도 (주문→배송→반품→정산)
   ③ EAI 인터페이스 주요 목록 (물류 관련 30개)
   ④ 이벤트 스토밍 간략 소개 자료
   ⑤ 택배사 목록 및 연동 방식 정리
```

#### 진행 중 관련

```
Q6: "72개 EAI 인터페이스를 다 다뤄야 하나요?"
A: 아닙니다. EAI는 전송 수단이지 이벤트가 아닙니다.
   비즈니스 관점에서 '무슨 일이 일어났는가'를 도출합니다.
   EAI 목록은 참고용으로 비치하되,
   이벤트 도출의 주 재료는 프로세스 흐름입니다.

Q7: "참석자가 EAI 인터페이스명으로 이벤트를 적고 있어요."
A: "비즈니스 용어로 바꿔볼까요?"라고 안내합니다.
   'IF_FU_SHIP_ORD_001 호출' → '출하지시가 생성되었다'
   "이 인터페이스가 호출되면 비즈니스 관점에서
    무슨 일이 일어난 건가요?" 라는 질문이 효과적입니다.

Q8: "택배사별로 이벤트를 나누고 있어요."
A: 택배사 차이는 COU 서비스 내부 처리입니다.
   "택배사가 달라도 '배송 지시를 보낸다'는 같은 일이죠.
    하나의 이벤트로 적고, 택배사는 속성으로 구분합시다."

Q9: "참석자가 CRUD 수준으로 너무 세밀하게 적고 있어요."
A: "이벤트 스토밍에서는 '비즈니스 관점'이 중요합니다.
    '배송정보 레코드가 INSERT되었다'가 아니라,
    '배송정보가 전달되었다'입니다.
    다른 사람이 들어도 이해할 수 있는 말로 적어주세요."

Q10: "한 서비스의 이벤트가 너무 많이 나왔어요."
A: 기준을 다시 확인합니다:
   "이 이벤트로 다른 서비스가 반응해야 하나요?"
   아니라면 내부 처리이므로 제외합니다.
   서비스당 핵심 이벤트 5~10개가 적정 수준입니다.
```

#### 결과 정리 관련

```
Q11: "기간계→물류(FU) 연동이 가장 중요한데, 어떻게 강조하나요?"
A: 전체 흐름도에서 이 연동을 ⭐ 표시하고,
   별도로 "핵심 연동 포인트"로 문서화합니다.
   주문완료→배송정보전달은 물류 프로세스의 시작점이므로
   장애격리 및 동시성 처리 전략도 함께 논의합니다.

Q12: "바운디드 컨텍스트를 6개로 확정해도 되나요?"
A: MSA 후보 서비스가 6개로 이미 식별되어 있으므로,
   워크샵에서는 각 서비스가 독립 컨텍스트인지 "검증"합니다.
   독립배포/독립DB/독립팀/프로세스축 기준으로 확인하면 됩니다.
   TRK/STL은 팀 독립성이 조건부이므로 핫스팟으로 표시할 수 있습니다.

Q13: "워크샵 결과를 어떤 형태로 정리하나요?"
A: 결과물 템플릿(8.2절)을 활용하세요.
   핵심은: ① 서비스별 이벤트 목록
           ② 서비스 간 연동 매트릭스 (6×6)
           ③ 핫스팟 목록 (기간계 경계, 동시성 이슈 등)
           ④ 기간계/외부 시스템 연동 정리

Q14: "이벤트 스토밍 결과를 개발에 바로 쓸 수 있나요?"
A: 이벤트 스토밍은 도메인 이해와 서비스 경계 확인이 목적입니다.
   개발 설계(API, DB, EAI 리팩토링 등)는 별도 단계에서 진행하며,
   이벤트 스토밍 결과를 입력으로 활용합니다.
```
