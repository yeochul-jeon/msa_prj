# 이벤트 스토밍 검색 서비스 개발팀 가이드

## 1. 개요

### 1.1 이 가이드의 목적

검색 서비스 개발팀을 대상으로 이벤트 스토밍 워크샵을 진행할 때, **퍼실리테이터와 참석자가 자주 마주하는 특수한 상황**에 대한 실전 가이드입니다.

```
┌─────────────────────────────────────────────────────────────┐
│              검색팀 워크샵의 핵심 도전 4가지                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 기술 이벤트와 비즈니스 이벤트의 경계 모호              │
│     "인덱싱 완료도 이벤트인가요?"                          │
│                                                             │
│  2. 배치 처리와 실시간 처리의 구분                         │
│     "야간 배치 색인은 어떻게 표현하죠?"                    │
│                                                             │
│  3. 외부 검색 엔진 의존성 처리                             │
│     "Elasticsearch 내부 동작도 이벤트인가요?"              │
│                                                             │
│  4. 다른 팀에 서비스 제공하는 역할                         │
│     "상품팀이 우리 API를 호출하는 건 누구 이벤트?"         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 2. 기술 이벤트 vs 비즈니스 이벤트 구분

### 2.1 핵심 판단 기준

```
┌─────────────────────────────────────────────────────────────┐
│                    핵심 판단 기준                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  "이 변화가 비즈니스 담당자에게 의미가 있는가?"            │
│                                                             │
│  예 → 비즈니스 이벤트 가능성 높음 🟧                       │
│  아니오 → 기술/인프라 이벤트 (제외)                        │
│                                                             │
│  ─────────────────────────────────────────────────────────  │
│                                                             │
│  추가 판단 기준:                                           │
│  • 운영팀이나 기획팀이 알아야 하는가?                      │
│  • 사용자 경험에 직접적 영향을 주는가?                     │
│  • 다른 팀/시스템에 알려야 하는가?                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 판단 체크리스트 (3단계)

이벤트 후보가 도메인 이벤트인지 판단할 때 아래 질문을 순서대로 확인하세요.

```
□ 1. 비즈니스적 의미가 있는가?
     → 아니오: 기술 이벤트 (제외)
     → 예: 다음 질문으로

□ 2. 다른 팀/시스템이 알아야 하는가?
     → 아니오: 내부 처리 (제외 또는 검토 필요)
     → 예: 다음 질문으로

□ 3. 사용자 또는 운영자가 인지할 수 있는가?
     → 아니오: 🩷 핫스팟으로 표시 (추가 논의)
     → 예: 도메인 이벤트! 🟧
```

## 3. 이벤트인지 결정하기 어려운 상황 TOP 10

> **검색팀 워크샵의 핵심 가이드**: 가장 자주 발생하는 판단 난제 상황과 그 해결책

### 3.1 요약 테이블

| # | 상황 | 결론 | 판단 기준 |
|---|------|------|-----------|
| 1 | 인덱싱 완료 | 조건부 | 전체 색인 → ✅, 개별 문서 → ❌ |
| 2 | 검색 쿼리 실행 | ❌ | 커맨드의 중간 과정일 뿐 |
| 3 | 검색 결과 반환 | ✅ | 사용자에게 의미 있음 |
| 4 | 캐시 갱신 | ❌ | 기술 최적화 |
| 5 | 동의어 사전 갱신 | ✅ | 검색 품질 변화 |
| 6 | 검색 순위 변경 | ✅ | 비즈니스 의미 있음 |
| 7 | 자동완성 후보 갱신 | ✅ | 사용자 경험 영향 |
| 8 | 검색 로그 기록 | ❌ | 내부 처리 |
| 9 | 무검색 결과 발생 | ✅ | 비즈니스 알림 필요 |
| 10 | 스펠체크 동작 | ❌ | 기능 동작일 뿐 |

### 3.2 상황별 상세 가이드

#### 상황 1: 인덱싱 관련 이벤트 판단

```
┌─────────────────────────────────────────────────────────────┐
│              상황 1: 인덱싱 완료가 이벤트인가요?            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  질문: "상품 1건이 인덱싱되었습니다"는 이벤트인가요?       │
│                                                             │
│  ─────────────────────────────────────────────────────────  │
│                                                             │
│  ❌ 개별 문서 인덱싱 (이벤트 아님)                          │
│  • "문서 1건이 인덱싱되었다"                               │
│  • "상품 ID:12345가 색인되었다"                            │
│  → 이유: 기술적 처리 단위, 비즈니스 의미 없음              │
│                                                             │
│  ─────────────────────────────────────────────────────────  │
│                                                             │
│  ✅ 전체/대량 색인 완료 (이벤트 맞음)                       │
│  • "신규 상품 배치 색인이 완료되었다"                      │
│  • "전체 재색인이 완료되었다"                              │
│  • "상품 카테고리별 색인이 갱신되었다"                     │
│  → 이유: 운영팀이 알아야 함, 서비스 상태 변화              │
│                                                             │
│  ─────────────────────────────────────────────────────────  │
│                                                             │
│  🩷 핫스팟으로 표시할 경우                                  │
│  • "실시간 색인 vs 배치 색인 기준이 불명확"                │
│  • "색인 지연 시 알림 기준이 없음"                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**판단 공식**:
- 개별 건 → 이벤트 아님
- 단위 작업 완료 (배치, 전체, 카테고리 등) → 이벤트

#### 상황 4: 배치 vs 실시간 처리 판단

```
┌─────────────────────────────────────────────────────────────┐
│          상황 4: 배치 작업 전체를 하나의 이벤트로?          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  질문: "야간 배치가 실행되었습니다"는 이벤트인가요?        │
│                                                             │
│  ─────────────────────────────────────────────────────────  │
│                                                             │
│  ❌ 너무 추상적 (이벤트로 부적합)                           │
│  • "배치가 실행되었다"                                     │
│  • "스케줄러가 동작했다"                                   │
│  • "크론잡이 시작되었다"                                   │
│                                                             │
│  ─────────────────────────────────────────────────────────  │
│                                                             │
│  ✅ 배치 결과를 구체적으로 표현 (이벤트 맞음)               │
│  • "인기 검색어 순위가 갱신되었다"                         │
│  • "동의어 사전이 갱신되었다"                              │
│  • "자동완성 후보가 갱신되었다"                            │
│  • "검색 랭킹 모델이 업데이트되었다"                       │
│  • "상품 색인이 전체 갱신되었다"                           │
│                                                             │
│  ─────────────────────────────────────────────────────────  │
│                                                             │
│  교정 질문: "이 배치로 무엇이 변경되었나요?"               │
│                                                             │
│  배치 = 트리거(커맨드 🟦에 가까움)                         │
│  결과물 = 이벤트(🟧)                                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**배치 처리 이벤트 도출 공식**:

```
배치 작업 → "무엇이 변경되었나요?" → 구체적 이벤트

예시:
야간 배치 → 인기 검색어가 갱신되었다
           동의어 사전이 업데이트되었다
           검색 로그가 집계되었다
```

```
│  • "외부에서 검색 요청이 들어왔다"                         │
│  → 이유: 호출은 커맨드, 결과가 이벤트                      │
│                                                             │
│  ─────────────────────────────────────────────────────────  │
│                                                             │
│  검색팀 관점의 이벤트:                                     │
│  ┌──────────────────────────────────────────────────────┐  │
│  │  🟩 상품팀     🟦 검색 요청     🟧 검색 결과가       │  │
│  │  (외부)   →   (커맨드)     →   반환되었다 (이벤트)  │  │
│  └──────────────────────────────────────────────────────┘  │
│                                                             │
│  상품팀 관점의 이벤트:                                     │
│  ┌──────────────────────────────────────────────────────┐  │
│  │  🟦 상품 검색하기  →  🟩 검색팀  →  🟧 상품이        │  │
│  │  (커맨드)             (외부)       검색되었다        │  │
│  └──────────────────────────────────────────────────────┘  │
│                                                             │
│  ─────────────────────────────────────────────────────────  │
│                                                             │
│  핵심: 각 팀이 자기 관점에서 이벤트를 정의                 │
│  통합 워크샵에서 연동 지점을 맞춤                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 6.2 상황별 개입 스크립트

#### "인덱싱 완료가 이벤트인가요?"

```
대응:
"좋은 질문입니다. 개별 문서 1건의 인덱싱은 이벤트가 아닙니다.

하지만 '신규 상품 배치 색인이 완료되었다'처럼
비즈니스 의미 있는 단위라면 이벤트입니다.

질문해볼게요:
'이 색인 완료를 운영팀이 알아야 하나요?'
'이 색인으로 사용자 검색 결과가 달라지나요?'

YES라면 이벤트, NO라면 제외합니다."
```

#### "캐시 갱신도 이벤트인가요?"

```
대응:
"캐시 갱신은 성능 최적화이지 비즈니스 이벤트가 아닙니다.
제외해주세요.

다만, 캐시 문제로 사용자에게 영향이 있다면
분홍색 핫스팟으로 표시해두면 됩니다:
🩷 '캐시 불일치로 오래된 검색 결과 노출 이슈'

이런 식으로 문제점은 기록하되,
캐시 동작 자체는 이벤트로 적지 않습니다."
```

#### "검색 쿼리 분석이 이벤트인가요?"

```
대응:
"검색 쿼리 분석, 형태소 분석, 불용어 제거...
이런 것들은 '검색하기' 커맨드의 내부 처리 과정입니다.

이벤트는 결과입니다:
- '검색 결과가 반환되었다' ✅
- '무검색 결과가 발생했다' ✅

중간 과정이 아니라 최종 결과를 적어주세요."
```

#### "배치 작업은 어떻게 표현하나요?"

```
대응:
"'야간 배치가 실행됨'은 너무 추상적입니다.

배치로 무엇이 변경되었는지 구체적으로 적어주세요:
- '인기 검색어 순위가 갱신되었다'
- '동의어 사전이 업데이트되었다'
- '검색 로그가 집계되었다'

배치는 트리거(커맨드)이고,
그 결과물이 이벤트입니다."
```

#### "API 호출을 이벤트로 적어도 되나요?"

```
대응:
"API 호출은 커맨드입니다. 결과가 이벤트예요.

❌ '검색 API가 호출되었다'
✅ '검색 결과가 반환되었다'

'~가 호출되었다'를
'~가 반환되었다' 또는 '~가 완료되었다'로
바꿔보세요."
```

## 8. FAQ 및 트러블슈팅

### 8.1 자주 묻는 질문

**Q: 검색 서비스팀 워크샵에서 검색 엔진(ES, Solr 등) 상세 논의도 해야 하나요?**

A: 이벤트 스토밍에서는 **비즈니스 이벤트**에 집중합니다.
- 검색 엔진은 외부 시스템(🟩)으로 표시
- "ES 샤드 리밸런싱"이 아니라 "검색 색인이 갱신됨"으로 추상화
- 검색 엔진 관련 기술 이슈는 🩷 핫스팟으로 표시
- 상세 인프라 논의는 이후 설계 단계에서 진행

---

**Q: 실시간 색인과 배치 색인은 다른 이벤트로 적어야 하나요?**

A: 비즈니스 관점에서 결과가 같다면 **하나의 이벤트**로 표현하고, 차이점은 핫스팟으로 표시합니다.
- 🟧 "상품 색인이 갱신되었다" (공통 이벤트)
- 🩷 "실시간 vs 배치 처리 기준 정의 필요"

단, 비즈니스 요구사항이 다르다면 분리:
- 🟧 "실시간 상품 색인이 반영되었다" (즉시 반영 필요)
- 🟧 "배치 색인이 완료되었다" (야간 처리)

---

**Q: 검색 로그 기록도 이벤트인가요?**

A: **내부 처리용 로깅은 이벤트가 아닙니다.**
- ❌ "검색 로그가 기록되었다" (내부 처리)

단, 비즈니스 분석을 위한 집계 결과는 이벤트:
- ✅ "검색어 통계가 집계되었다"
- ✅ "무검색 결과 키워드가 식별되었다"

---

**Q: 무검색 결과(Zero Result)는 어떻게 다루나요?**

A: 무검색 결과는 **중요한 비즈니스 이벤트**입니다.
- ✅ "무검색 결과가 발생했다"
- 이유: 운영팀/기획팀이 알아야 하고, 개선 액션이 필요함

관련 정책도 함께 도출:
- 💜 "무검색 결과 발생 시 유사 상품 추천"
- 💜 "무검색 키워드 3회 이상 발생 시 알림"

---

### 8.2 트러블슈팅

**상황 1: 참석자들이 계속 검색 엔진 용어를 사용할 때**

```
대응:
1. 부드럽게 리다이렉트:
   "좋습니다. 그 ES 작업이 끝나면 결과적으로
    사용자나 다른 팀에게 무슨 일이 일어나나요?"

2. 예시로 변환:
   "예를 들어, 'ES 인덱스 리프레시 완료'는
    '검색 색인이 갱신되었다'로 표현하면 됩니다."

3. 기술 논의는 핫스팟으로:
   "검색 엔진 관련 기술 결정은 핫스팟으로 표시해두고,
    나중에 설계 단계에서 자세히 다루겠습니다."
```

**상황 2: 실시간 vs 배치 논쟁이 길어질 때**

```
대응:
1. 핫스팟으로 표시하고 넘어가기:
   "두 의견 모두 타당합니다. 일단 핫스팟으로 표시해두죠.
    🩷 '실시간/배치 처리 기준 정의 필요'"

2. 비즈니스 요구사항 확인:
   "비즈니스 관점에서, 즉시 반영이 필수인가요?
    아니면 몇 분 지연되어도 괜찮은가요?"

3. 나중으로 미루기:
   "이 부분은 설계 단계에서 SLA를 정의할 때
    함께 논의하면 좋겠습니다."
```

**상황 3: "우리는 검색 API를 제공하기만 하는데요?"**

```
대응:
검색팀이 플랫폼 역할을 한다면:

1. 검색팀 내부 이벤트 도출:
   "다른 팀이 사용하더라도, 검색팀 내부에서
    일어나는 이벤트를 먼저 적어봅시다."

2. 연동 지점 명확화:
   "어떤 팀이 검색 API를 사용하나요?
    그 팀과의 연동 지점을 🟩으로 표시합시다."

3. 서비스 관점 이벤트:
   "API를 제공하는 입장에서,
    '검색 서비스가 정상화되었다',
    '검색 서비스가 점검 중이다' 같은
    서비스 상태 이벤트도 중요합니다."
```

**상황 4: 이벤트가 너무 기술적으로만 나올 때**

```
대응:
유도 질문 활용:

1. "이 기술 작업으로 사용자가 뭘 할 수 있게 되나요?"
   → 인덱싱 완료 → 사용자가 검색 가능

2. "이 작업이 안 되면 운영팀에서 뭐라고 하나요?"
   → "검색이 안 돼요" → "검색 서비스 장애 발생"

3. "기획팀에게 이 작업을 설명한다면?"
   → "신규 상품이 검색에 반영됩니다"

4. 비즈니스 언어로 바꿔 적기:
   "형태소 분석 완료" → (제외, 중간 과정)
   "검색 결과 반환" → 그대로 사용
```
