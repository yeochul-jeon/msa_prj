## 1. 전체 시간축 흐름

### 1.1 방송 라이프사이클 전체 흐름도

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                       방송 라이프사이클 전체 흐름                            │
└─────────────────────────────────────────────────────────────────────────────┘

  시간축 무관                방송 전                   방송 중              방송 후
  (기반 데이터)         (비실시간 계획)             (실시간 운영)         (정산/분석)
 ─────────────────────────────────────────────────────────────────────────────

  ┌─────────┐          ┌─────────────┐          ┌─────────┐          ┌──────────┐
  │  PDSS   │ ──────→  │    편성      │ ──────→ │  온에어  │ ──────→ │ PGM실적  │
  │ (BSS)   │          │   (BCS)      │          │  (BOM)   │          │  /정산   │
  │ 기반    │          └──────┬──────┘          │  실시간  │          │  (BCS)   │
  │ 데이터  │                 │                  │  Read    │          └──────────┘
  └────┬────┘          ┌──────┴──────┐          └────┬─────┘
       │               │ 방송제작    │               │              ┌──────────┐
       │               │  기획       │               ├────────────→ │   매체   │
       │               │ (BCS)       │               │              │  (RFM)   │
       │               └─────────────┘               │              └──────────┘
       │                                              │
       │           ┌──────────────┐                   │
       └─────────→ │ 타사 모니터  │ ←─── (비교 데이터)
                   │   (BCM)      │
                   └──────────────┘

  ─────────────────────────────────────────────────────────────────────────────
  범례:
  ──→  이벤트 흐름 (주방향)
  ←──  참조/비교 데이터 흐름
```

### 1.3 이벤트 발행/구독 매트릭스

> 7개 서비스 간 이벤트 발행(Publish)과 구독(Subscribe) 관계를 매트릭스로 표현합니다.

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    이벤트 발행/구독 매트릭스 (7×7)                            │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  발행 ＼ 구독    PDSS   편성   기획   온에어  타사   매체   PGM실적         │
│  ──────────────────────────────────────────────────────────────────         │
│  PDSS          │  -     ●      ●      ●       ●      ●      ●              │
│  편성          │  -     -      ●      ●       -      -      ●              │
│  기획          │  -     ●      -      ●       -      -      -              │
│  온에어        │  -     -      -      -       -      ●      ●              │
│  타사모니터    │  -     ○      -      -       -      -      -              │
│  매체          │  -     -      -      -       -      -      -              │
│  PGM실적      │  -     -      -      -       -      -      -              │
│                                                                             │
│  ● = 주요 이벤트 구독 (필수 연동)                                           │
│  ○ = 참조 데이터 구독 (선택적 연동)                                         │
│                                                                             │
│  핵심 흐름:                                                                 │
│  PDSS → (전체)   : 기반 데이터 변경 시 모든 서비스에 영향                   │
│  편성 → 온에어   : 편성 확정 → 방송 준비 (가장 중요한 연동)                │
│  온에어 → PGM실적: 방송 종료 → 실적 집계 시작                              │
│  온에어 → 매체   : 방송 데이터 → 매체 비용 산출                            │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 2. 서비스 간 이벤트 흐름 상세

### 2.1 PDSS → 전체 서비스

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    PDSS(기반 데이터) → 전체 서비스 흐름                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  PDSS는 채널 시스템의 "마스터 데이터"를 관리합니다.                          │
│  PDSS의 변경은 모든 하위 서비스에 전파됩니다.                                │
│                                                                             │
│                    ┌──────────┐                                             │
│                    │   PDSS   │                                             │
│                    │ 기반데이터│                                             │
│                    └────┬─────┘                                             │
│                         │                                                   │
│     🟧 채널이 등록되었다 / 채널정보가 변경되었다                             │
│     🟧 프로그램유형이 정의되었다                                             │
│     🟧 시간대 정책이 변경되었다                                              │
│                         │                                                   │
│         ┌───────┬───────┼───────┬───────┬───────┐                          │
│         ▼       ▼       ▼       ▼       ▼       ▼                          │
│     ┌──────┐┌──────┐┌──────┐┌──────┐┌──────┐┌──────┐                     │
│     │ 편성 ││ 기획 ││온에어││ 타사 ││ 매체 ││PGM   │                     │
│     │      ││      ││      ││모니터││      ││실적  │                     │
│     └──────┘└──────┘└──────┘└──────┘└──────┘└──────┘                     │
│                                                                             │
│  전파 방식:                                                                 │
│  • 이벤트 기반: 채널 추가/삭제 → 관련 서비스에 알림                         │
│  • API 참조: 채널 기본 정보 조회 → 동기 호출                                │
│                                                                             │
│  🩷 핫스팟:                                                                 │
│  • PDSS 데이터 변경 시 전체 서비스 동시 반영 순서                           │
│  • AS-IS 시스템에서 분리 시 데이터 마이그레이션 범위                        │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2.2 편성 ↔ 방송제작 기획

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                       편성 ↔ 방송제작 기획 흐름                              │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  편성과 기획은 방송 전 단계에서 양방향 연동합니다.                           │
│                                                                             │
│  ┌──────────────┐                        ┌──────────────┐                  │
│  │    편성       │                        │  방송제작    │                  │
│  │   (BCS)       │                        │   기획       │                  │
│  └──────┬───────┘                        └──────┬───────┘                  │
│         │                                        │                          │
│         │ 🟧 편성항목이 배치되었다                │                          │
│         │ 🟧 편성표가 확정되었다                  │                          │
│         │────────────────────────────────────→   │                          │
│         │                                        │ 기획서와 편성 매칭       │
│         │                                        │                          │
│         │    🟧 기획서가 승인되었다               │                          │
│         │    🟧 출연자가 확정되었다               │                          │
│         │   ←────────────────────────────────────│                          │
│         │ 편성항목에 기획 정보 반영               │                          │
│                                                                             │
│  연동 포인트:                                                               │
│  • 편성 → 기획: "이 시간대에 방송할 프로그램이 편성되었다"                  │
│  • 기획 → 편성: "이 프로그램의 기획이 확정되었다"                           │
│                                                                             │
│  💜 정책:                                                                   │
│  • "기획서 미승인 프로그램은 편성 확정 불가"                                │
│  • "편성 확정 후 기획 변경 시 편성팀 승인 필요"                             │
│                                                                             │
│  🩷 핫스팟:                                                                 │
│  • 편성 변경과 기획 변경의 우선순위 충돌                                    │
│  • 긴급 편성 시 기획서 후속 작성 프로세스                                   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2.3 편성 → 온에어

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                          편성 → 온에어 흐름                                  │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  ⭐ 채널 시스템에서 가장 중요한 연동 포인트                                  │
│  비실시간(편성) → 실시간(온에어)으로의 전환점                                │
│                                                                             │
│  ┌──────────────┐                        ┌──────────────┐                  │
│  │    편성       │                        │   온에어     │                  │
│  │ (비실시간)    │                        │  (실시간)    │                  │
│  └──────┬───────┘                        └──────┬───────┘                  │
│         │                                        │                          │
│         │ 🟧 편성표가 확정되었다                  │                          │
│         │────────────────────────────────────→   │                          │
│         │                                        │ 📖 편성정보 읽기모델     │
│         │                                        │    생성/갱신             │
│         │                                        │                          │
│         │ 🟧 편성변경이 확정되었다                │                          │
│         │   (긴급 편성 변경)                      │                          │
│         │────────────────────────────────────→   │                          │
│         │                                        │ 📖 읽기모델 즉시 갱신    │
│                                                                             │
│  온에어의 데이터 소비 방식:                                                  │
│  • 편성 데이터를 직접 수정하지 않음 (Read Only)                              │
│  • 편성 확정 이벤트를 받아 자체 읽기 모델로 변환                            │
│  • 방송 중 편성 변경 시에도 온에어 읽기 모델만 갱신                          │
│                                                                             │
│  장애 격리:                                                                 │
│  • 편성 서비스 장애 시에도 온에어는 기존 읽기 모델로 운영 가능               │
│  • 편성 변경 이벤트 유실 시 재처리 메커니즘 필요                            │
│                                                                             │
│  💜 정책:                                                                   │
│  • "편성 확정 후 온에어 읽기 모델 반영 지연 최대 5분"                       │
│  • "방송 시작 30분 전까지 편성 변경 가능"                                   │
│                                                                             │
│  참고: 온에어의 읽기 모델 패턴은                                             │
│  [이벤트스토밍_읽기모델_가이드.md](./이벤트스토밍_읽기모델_가이드.md) 참조    │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2.4 온에어 ↔ 내부 시스템

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                      온에어 ↔ 내부 시스템(상품/주문) 흐름                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  온에어는 방송 중 상품/주문 정보를 실시간으로 참조합니다.                    │
│  데이터 소유는 내부 시스템에 있고, 온에어는 읽기만 합니다.                   │
│                                                                             │
│  ┌───────────────────────────────────────────────────────┐                  │
│  │                     온에어 (BOM)                       │                  │
│  │                                                       │                  │
│  │  📖 방송상품정보 읽기모델                               │                  │
│  │  📖 실시간주문현황 읽기모델                              │                  │
│  │  📖 실시간재고 읽기모델                                  │                  │
│  └──────────┬───────────────────────┬────────────────────┘                  │
│             │                       │                                       │
│        (상품 정보 참조)         (주문 현황 참조)                             │
│             │                       │                                       │
│             ▼                       ▼                                       │
│  ┌──────────────┐         ┌──────────────┐                                 │
│  │  🟩 상품     │         │  🟩 주문     │                                 │
│  │  서비스      │         │  서비스      │                                 │
│  │  (내부)      │         │  (내부)      │                                 │
│  └──────────────┘         └──────────────┘                                 │
│                                                                             │
│  이벤트 흐름:                                                               │
│  • 🟩 상품팀 → 온에어: "상품가격이 변경되었다" → 읽기모델 갱신             │
│  • 🟩 주문팀 → 온에어: "주문이 접수되었다" → 실시간 현황 갱신              │
│  • 온에어 → 🟩 상품팀: 방송 중 상품 교체 알림 (커맨드 발행)                │
│                                                                             │
│  소유권 원칙:                                                               │
│  • 상품/주문 데이터의 원본 = 내부 시스템                                    │
│  • 온에어는 읽기 모델로 복제본을 유지                                       │
│  • 데이터 불일치 시 내부 시스템이 진실의 원천(Source of Truth)               │
│                                                                             │
│  🩷 핫스팟:                                                                 │
│  • 내부 시스템 장애 시 온에어 읽기 모델의 유효 기간                         │
│  • 실시간 재고 동기화 지연 허용 범위                                        │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2.5 온에어 → PGM 실적/정산

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                       온에어 → PGM 실적/정산 흐름                            │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  방송 종료 후 실적 데이터가 PGM 실적/정산 서비스로 전달됩니다.              │
│                                                                             │
│  ┌──────────────┐                        ┌──────────────┐                  │
│  │   온에어     │                        │  PGM 실적    │                  │
│  │  (실시간)    │                        │  /정산       │                  │
│  └──────┬───────┘                        └──────┬───────┘                  │
│         │                                        │                          │
│         │ 🟧 방송이 종료되었다                    │                          │
│         │   - 채널ID, 프로그램ID                  │                          │
│         │   - 방송시간, 판매상품목록               │                          │
│         │────────────────────────────────────→   │                          │
│         │                                        │ 실적 집계 시작           │
│         │                                        │                          │
│         │                                        │ 🟧 방송실적이            │
│         │                                        │    집계되었다            │
│         │                                        │                          │
│         │                                        │ 🟧 정산이               │
│         │                                        │    완료되었다            │
│                                                                             │
│  데이터 특성:                                                               │
│  • 대량 데이터 전달 (방송당 수천~수만 건 주문 데이터)                       │
│  • 배치 처리 가능 (실시간 필요 없음)                                        │
│  • 보안 민감 데이터 포함 (매출, 수수료 등)                                  │
│                                                                             │
│  💜 정책:                                                                   │
│  • "방송 종료 후 24시간 이내 1차 실적 집계 완료"                            │
│  • "정산 확정 전 검증 절차 필수"                                            │
│                                                                             │
│  🩷 핫스팟:                                                                 │
│  • Fade out 가능성에 따른 서비스 범위 설정                                  │
│  • 보안 데이터 접근 권한 관리                                               │
│  • 정산 기준 변경 시 소급 적용 여부                                         │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 3. 바운디드 컨텍스트 후보 확인

### 3.1 7개 서비스 = 7개 컨텍스트 검증

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                 7개 MSA 후보 = 7개 바운디드 컨텍스트 검증                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  검증 기준: 각 서비스가 독립된 바운디드 컨텍스트인지 확인                    │
│                                                                             │
│  서비스       │ 독립배포 │ 독립DB │ 독립팀  │ 시간축  │ 컨텍스트 ✓         │
│  ─────────────────────────────────────────────────────────────────────────  │
│  PDSS         │  ✅      │  ✅    │ ✅     │ 무관    │ ✅ 기반 데이터     │
│  편성         │  ✅      │  ✅    │ ✅     │ 방송전  │ ✅ 편성 계획       │
│  방송제작기획 │  ✅      │  ✅    │ ✅     │ 방송전  │ ✅ 기획 관리       │
│  온에어       │  ✅      │  ✅    │ ✅     │ 방송중  │ ✅ 실시간 운영     │
│  타사모니터   │  ✅      │  ✅    │ 조건부 │ 독립    │ ✅ 경쟁사 분석     │
│  매체         │  조건부  │  ✅    │ 조건부 │ 방송후  │ ⚠️ Fade out 검토  │
│  PGM실적/정산 │  ✅      │  ✅    │ ✅     │ 방송후  │ ⚠️ Fade out 검토  │
│                                                                             │
│  ⚠️ 주의 사항:                                                              │
│  • 매체/PGM실적: Fade out 가능성이 있어 독립 서비스 유지 비용 대비          │
│    이점을 워크샵에서 논의해야 함                                             │
│  • 타사모니터: 이관 가능성이 있어 소유권을 명확히 해야 함                   │
│  • 편성/기획: 같은 BCS 약어지만 별도 컨텍스트로 분리 확정                   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3.2 컨텍스트 맵 (Upstream/Downstream)

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    채널 시스템 컨텍스트 맵                                    │
└─────────────────────────────────────────────────────────────────────────────┘

                         [Upstream]
                              │
                    ┌─────────┴─────────┐
                    │       PDSS        │
                    │  (기반 데이터)     │
                    │  Conformist(준수자)│ ← 하위 서비스는 PDSS 모델 준수
                    └─────────┬─────────┘
                              │
          ┌───────────────────┼───────────────────┐
          │                   │                   │
          ▼                   ▼                   ▼
  ┌──────────────┐   ┌──────────────┐   ┌──────────────┐
  │    편성       │   │ 타사 모니터  │   │   (기타)     │
  │   (BCS)       │   │   (BCM)      │   │              │
  │ Partnership  │   │ Separate Ways│   │              │
  └──────┬───────┘   │ (독립 운영)  │   └──────────────┘
         │           └──────────────┘
         │
    ┌────┴──────────────┐
    │                   │
    ▼                   ▼
┌──────────────┐  ┌──────────────┐
│ 방송제작기획 │  │   온에어     │
│   (BCS)       │  │  (BOM)       │
│ Partnership  │  │ ACL          │ ← Anti-Corruption Layer
└──────────────┘  │ (부패방지계층)│     편성 모델을 온에어 모델로 변환
                  └──────┬───────┘
                         │
              ┌──────────┼──────────┐
              │                     │
              ▼                     ▼
       ┌──────────────┐     ┌──────────────┐
       │  PGM 실적    │     │    매체      │
       │  /정산       │     │   (RFM)      │
       │ Customer/    │     │ Customer/    │
       │ Supplier     │     │ Supplier     │
       └──────────────┘     └──────────────┘
                   [Downstream]

  ─────────────────────────────────────────────────────────────────────────
  컨텍스트 매핑 패턴:
  • Conformist (준수자)    : 하위 서비스가 상위 모델을 그대로 사용
  • Partnership (파트너십) : 양방향 협력 (편성 ↔ 기획)
  • ACL (부패방지계층)     : 모델 변환 계층 (편성 → 온에어)
  • Separate Ways (독립)   : 느슨한 결합 (타사 모니터)
  • Customer/Supplier      : 공급자-소비자 관계 (온에어 → 정산/매체)
```

### 3.3 내부 시스템 컨텍스트와의 관계

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                  채널 시스템 ↔ 내부 시스템 컨텍스트 관계                      │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  ┌─────────────────────────────────────────────────────┐                   │
│  │              채널 시스템 (외부 협력사)                │                   │
│  │                                                     │                   │
│  │  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐              │                   │
│  │  │ PDSS │ │ 편성 │ │ 기획 │ │온에어│              │                   │
│  │  └──────┘ └──────┘ └──────┘ └──┬───┘              │                   │
│  │  ┌──────┐ ┌──────┐ ┌──────┐    │                  │                   │
│  │  │ 타사 │ │ 매체 │ │ PGM  │    │                  │                   │
│  │  └──────┘ └──────┘ └──────┘    │                  │                   │
│  └────────────────────────────────┼──────────────────┘                   │
│                                   │                                       │
│                              ACL (부패방지계층)                            │
│                                   │                                       │
│  ┌────────────────────────────────┼──────────────────┐                   │
│  │              내부 시스템 (자사)  │                  │                   │
│  │                                │                  │                   │
│  │  ┌───────────┐   ┌───────────┐│  ┌───────────┐   │                   │
│  │  │ 상품 서비스│   │ 주문 서비스││  │ 전시 서비스│   │                   │
│  │  │ (상품정보) │   │ (주문/결제)││  │ (노출/검색)│   │                   │
│  │  └───────────┘   └───────────┘│  └───────────┘   │                   │
│  │                                │                  │                   │
│  └────────────────────────────────┴──────────────────┘                   │
│                                                                             │
│  연동 원칙:                                                                 │
│  • 채널 시스템은 내부 시스템의 데이터를 "참조"만 함 (Read)                  │
│  • 내부 시스템은 채널 시스템의 방송 이벤트를 "구독"함                       │
│  • 양쪽 모두 상대방의 모델을 직접 사용하지 않음 (ACL 적용)                 │
│                                                                             │
│  이벤트 소유권 경계:                                                        │
│  ┌──────────────────────┬────────────────────────┐                         │
│  │ 채널 시스템 소유      │ 내부 시스템 소유        │                         │
│  ├──────────────────────┼────────────────────────┤                         │
│  │ 편성표가 확정되었다   │ 상품가격이 변경되었다   │                         │
│  │ 방송이 시작되었다     │ 주문이 접수되었다       │                         │
│  │ 방송이 종료되었다     │ 재고가 변경되었다       │                         │
│  │ 방송실적이 집계됨     │ 결제가 완료되었다       │                         │
│  └──────────────────────┴────────────────────────┘                         │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```
