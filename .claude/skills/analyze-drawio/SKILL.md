# draw.io 이벤트 스토밍 분석 스킬

## 입력

`$ARGUMENTS`에서 draw.io 파일 경로를 파싱한다.
- 예: `/workshop-prep analyze-drawio ./files/search-team.drawio`
- 파일이 없으면 사용자에게 경로를 요청한다.

## 실행 단계

### Step 1: draw.io XML 읽기

Read 도구로 draw.io 파일을 읽는다. draw.io 파일은 XML 형식이다.

### Step 2: 색상 기반 요소 추출

`references/color-mapping.md`의 매핑 규칙을 참조하여 각 `mxCell`의 `style` 속성에서 `fillColor`를 추출한다.

XML에서 추출해야 할 정보:
- `value` 속성: 포스트잇에 적힌 텍스트 (HTML 태그 제거 필요)
- `style` 속성 중 `fillColor`: 색상 코드
- 위치 정보 (`mxGeometry`의 x, y): 타임라인 순서 추정에 활용

### Step 3: 요소 분류 및 집계

색상→요소 매핑으로 분류한 후:
1. 요소 유형별 개수 집계
2. 각 유형별 목록 정리
3. 타임라인 순서 추정 (x 좌표 기준 정렬)

### Step 4: 오분류 감지

`references/color-mapping.md`의 오분류 감지 규칙을 적용한다:
- 색상과 이름 패턴이 불일치하는 항목을 경고로 표시
- 예: 노란색(애그리게이트)인데 이름이 "~됨"으로 끝남 → 이벤트 오분류 가능성

### Step 5: 결과 보고서 출력

아래 형식으로 구조화된 마크다운 보고서를 출력한다:

```markdown
# draw.io 분석 결과: {파일명}

## 요소 현황 요약

| 요소 | 개수 | 비고 |
|------|------|------|
| 도메인 이벤트 🟧 | N개 | |
| 커맨드 🟦 | N개 | |
| 애그리게이트 🟨 | N개 | |
| 정책 💜 | N개 | |
| 핫스팟 🩷 | N개 | |
| 외부 시스템 🟩 | N개 | |
| 읽기 모델 📖 | N개 | |

## 요소별 목록

### 도메인 이벤트 🟧 (N개)
1. 이벤트명1
2. 이벤트명2
...

### 커맨드 🟦 (N개)
...

## ⚠️ 오분류 의심 항목

| # | 현재 분류 | 이름 | 의심 사유 | 제안 |
|---|----------|------|----------|------|
| 1 | 애그리게이트 🟨 | "주문이 취소됨" | 이름이 "~됨"으로 끝남 | 이벤트로 재분류 |
...

## 흐름 영역 추정

x 좌표 기반으로 추정한 타임라인 영역:
1. 영역 1: ...
2. 영역 2: ...
```

## 주의사항

- draw.io 파일이 여러 페이지를 포함할 수 있다. 각 `<diagram>` 태그가 하나의 페이지이다.
- HTML 엔티티(`&lt;`, `&gt;`, `<br>` 등)를 적절히 처리한다.
- 색상 코드는 대소문자를 무시하고 비교한다.
- 매핑되지 않는 색상의 요소는 "미분류"로 표시한다.
