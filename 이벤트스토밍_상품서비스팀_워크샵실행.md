## 7. 퍼실리테이터 실전 스크립트

### 7.1 워크샵 오프닝 스크립트

```
┌─────────────────────────────────────────────────────────────┐
│             상품 서비스팀 워크샵 오프닝 스크립트            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  "안녕하세요, 오늘 상품 서비스팀 이벤트 스토밍 워크샵을    │
│  진행할 [이름]입니다.                                      │
│                                                             │
│  시작하기 전에 오늘 워크샵에서 몇 가지 부탁드릴 점이       │
│  있습니다.                                                 │
│                                                             │
│  ① '상품명 변경', '설명 수정' 같은 단순 데이터 변경은      │
│     별도 이벤트로 적지 마시고,                             │
│     '상품 판매 시작', '품절', '가격 변경'처럼              │
│     다른 팀이 알아야 하는 비즈니스 상태 변화를 적어주세요. │
│                                                             │
│  ② 재고 관련 이벤트는 '누가 재고를 관리하는가'에 따라     │
│     소유권이 달라집니다. 상품팀이 재고 데이터를 관리하면   │
│     '재고가 차감되었다'는 우리 이벤트입니다.               │
│     주문팀은 '재고 차감 요청'을 보내고 결과를 받죠.       │
│                                                             │
│  ③ 검수 단계를 '접수→1차→2차→최종'으로 세분화하지 말고,  │
│     판매자가 알아야 하는 결과인 '검수 승인/반려'만         │
│     적어주세요.                                            │
│                                                             │
│  ④ 가격은 '정가', '판매가', '할인가'를 구분해주세요.      │
│     정가/판매가는 상품팀, 할인가는 프로모션팀 영역입니다.  │
│                                                             │
│  완벽한 정답은 없습니다. 모르는 건 핫스팟(분홍색)으로      │
│  표시하면 됩니다. 편하게 진행해주세요!"                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 7.2 상황별 개입 스크립트

#### "상품명 변경도 이벤트인가요?"

```
┌─────────────────────────────────────────────────────────────┐
│     상황: 단순 데이터 변경이 이벤트인지 질문               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  대응:                                                     │
│  "좋은 질문입니다! 기준은 '다른 팀이 즉시 반응해야        │
│   하는가'입니다.                                           │
│                                                             │
│  상품명 변경은?                                            │
│  • 검색팀이 재색인해야 하지만, 배치로 처리 가능            │
│  • 주문팀이나 프로모션팀이 즉시 뭔가를 할 필요 없음        │
│  → 별도 이벤트로 적지 않아도 됩니다.                       │
│                                                             │
│  반면에, 가격 변경은?                                      │
│  • 프로모션팀: 할인율 재계산 필요                          │
│  • 주문팀: 주문 시 가격 확인 필요                          │
│  • 검색팀: 가격 표시 업데이트 필요                         │
│  → 다른 팀이 반응해야 하므로 이벤트입니다. 🟧              │
│                                                             │
│  정리하면:                                                 │
│  • 상품명/설명/이미지 변경 → 제외 (배치 처리)             │
│  • 가격/재고/판매상태 변경 → 이벤트 🟧"                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### "재고 차감은 상품팀 이벤트인가요?"

```
┌─────────────────────────────────────────────────────────────┐
│     상황: 재고 관련 이벤트 소유권에 대한 질문               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  대응:                                                     │
│  "재고 데이터를 누가 관리하느냐에 따라 달라집니다.         │
│                                                             │
│  우리(상품팀)가 재고 데이터를 관리한다면:                  │
│  • '재고가 차감되었다' → 상품팀 이벤트 🟧                 │
│  • '재고가 예약되었다' → 상품팀 이벤트 🟧                 │
│  • '재고가 복원되었다' → 상품팀 이벤트 🟧                 │
│                                                             │
│  주문팀은 '요청'을 보내고 '결과'를 받습니다:              │
│  • 주문팀: '재고 예약이 요청되었다' → 주문팀 발행         │
│  • 상품팀: '재고가 예약되었다' → 상품팀 발행              │
│                                                             │
│  이렇게 표현해볼까요?                                      │
│  ┌──────────────────────────────────────────────────────┐  │
│  │  🟩 주문팀           상품팀                          │  │
│  │                                                       │  │
│  │  재고 예약 요청  ──→  재고 확인/처리                 │  │
│  │  (주문팀 발행)              │                        │  │
│  │                              ▼                        │  │
│  │  재고 예약됨     ←──  재고 예약됨                   │  │
│  │  (수신)              (상품팀 발행)                   │  │
│  └──────────────────────────────────────────────────────┘  │
│                                                             │
│  소유권 원칙: '데이터를 관리하는 팀이 이벤트를 발행한다'" │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### "검수 단계를 다 적어야 하나요?"

```
┌─────────────────────────────────────────────────────────────┐
│     상황: 검수 단계 세분화에 대한 질문                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  대응:                                                     │
│  "검수 내부 단계는 적지 않아도 됩니다.                     │
│                                                             │
│  기준은 '판매자가 알아야 하는 상태인가?'입니다.            │
│                                                             │
│  판매자가 보는 상태:                                       │
│  • '검수 진행중' - 내부적으로 1차/2차가 있어도 모름        │
│  • '검수 완료' - 판매 가능                                 │
│  • '검수 반려' - 수정 후 재신청 필요                       │
│                                                             │
│  그래서 이벤트는 이렇게:                                   │
│  ❌ 접수됨 → 1차 완료 → 2차 완료 → 최종 승인             │
│  ✅ 검수 요청됨 → 검수 승인됨 (또는 반려됨)               │
│                                                             │
│  내부 처리 단계(1차/2차 검수)는 우리 시스템 내부 로직이고, │
│  비즈니스 관점에서는 '요청 → 결과'만 중요합니다.          │
│                                                             │
│  예외적으로, 판매자에게 각 단계를 알려줘야 한다면          │
│  그때는 단계별로 이벤트를 나눌 수 있어요.                  │
│  그런 경우가 있나요?"                                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### "가격 변경과 할인 적용이 헷갈려요"

```
┌─────────────────────────────────────────────────────────────┐
│     상황: 가격 vs 할인 구분에 대한 질문                     │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  대응:                                                     │
│  "가격 종류를 먼저 구분해봅시다.                           │
│                                                             │
│  상품팀이 관리하는 가격:                                   │
│  • 정가: 원래 가격 (RRP)                                   │
│  • 판매가: 실제 판매 가격 (정가보다 낮을 수 있음)          │
│                                                             │
│  프로모션팀이 관리하는 가격:                               │
│  • 할인가: 프로모션 기간 동안의 일시적 할인                │
│  • 쿠폰 적용가: 고객이 쿠폰 적용 시 가격                   │
│                                                             │
│  이벤트 구분:                                              │
│  • '정가가 변경되었다' → 상품팀 이벤트 🟧                 │
│  • '판매가가 변경되었다' → 상품팀 이벤트 🟧               │
│  • '할인가가 적용되었다' → 프로모션팀 이벤트 🟧           │
│                                                             │
│  예를 들어:                                                │
│  정가 100,000원, 판매가 80,000원인 상품에                  │
│  20% 할인 프로모션이 적용되면                              │
│  할인가 64,000원 (판매가의 20% 할인)                       │
│                                                             │
│  이 경우 프로모션팀이 '할인가가 적용되었다'를 발행하고,    │
│  상품팀은 이를 받아서 표시 가격을 업데이트합니다.          │
│                                                             │
│  지금 말씀하시는 건 어떤 가격인가요?"                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```
┌─────────────────────────────────────────────────────────────┐
│     상황: 판매자 관점과 플랫폼 관점 구분 질문               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  대응:                                                     │
│  "판매자 행동과 플랫폼 처리는 별개의 이벤트입니다.         │
│                                                             │
│  판매자 행동 (셀러센터에서):                               │
│  • '상품 등록이 요청되었다' → 판매자가 등록 버튼 클릭     │
│  • '가격 변경이 요청되었다' → 판매자가 가격 수정 제출     │
│  → 판매자의 행동 결과                                     │
│                                                             │
│  플랫폼/관리자 처리:                                       │
│  • '상품 등록이 승인되었다' → 검수 통과 후 시스템 처리    │
│  • '가격 변경이 반려되었다' → 관리자 검토 후 결정         │
│  → 플랫폼의 처리 결과                                     │
│                                                             │
│  흐름으로 보면:                                            │
│  ┌──────────────────────────────────────────────────────┐  │
│  │  🟩 판매자                                           │  │
│  │     │                                                 │  │
│  │     │ '상품 등록이 요청되었다'                       │  │
│  │     ▼                                                 │  │
│  │  상품팀 (검수)                                       │  │
│  │     │                                                 │  │
│  │     ├──→ '상품 등록이 승인되었다' → 판매자 알림     │  │
│  │     │                                                 │  │
│  │     └──→ '상품 등록이 반려되었다' → 판매자 알림     │  │
│  └──────────────────────────────────────────────────────┘  │
│                                                             │
│  '상품이 등록되었다' 하나로 뭉치지 말고,                   │
│  요청/승인/반려를 구분해주세요."                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### "품절과 판매중지가 다른가요?"

```
┌─────────────────────────────────────────────────────────────┐
│     상황: 품절과 판매중지 구분 질문                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  대응:                                                     │
│  "네, 다릅니다. 후속 처리가 다르거든요.                    │
│                                                             │
│  품절:                                                     │
│  • 원인: 재고가 0이 됨 (자동 감지)                         │
│  • 결과: 주문 불가, 하지만 입고 시 자동 복구               │
│  • 표시: '품절' 라벨, 입고 알림 신청 가능                  │
│  • 이벤트: '상품이 품절되었다' 🟧                         │
│                                                             │
│  판매중지:                                                 │
│  • 원인: 관리자/판매자의 의사결정                          │
│  • 결과: 주문 불가, 수동 해제 필요                         │
│  • 표시: 검색에서 제외 가능, 장바구니에서 제거             │
│  • 이벤트: '상품 판매가 중지되었다' 🟧                    │
│                                                             │
│  장바구니 처리도 다릅니다:                                 │
│  • 품절: 장바구니 유지, '품절' 표시, 입고 시 주문 가능    │
│  • 판매중지: 장바구니에서 제거, 재판매 전까지 주문 불가   │
│                                                             │
│  그래서 둘 다 별개의 이벤트로 적어야 합니다.               │
│  '판매 재개'도 품절 해제와 판매중지 해제가 다를 수 있어요.│
│  이 부분 구분이 필요하면 핫스팟으로 표시해둡시다."         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 7.3 이벤트 도출 유도 질문 리스트

#### 상품 등록 관련 질문

```
□ "판매자가 상품을 처음 등록하면 무슨 일이 일어나나요?"
□ "등록된 상품이 바로 판매되나요, 검수가 필요한가요?"
□ "검수 결과는 어떤 상태들이 있나요?"
□ "등록 반려 시 판매자에게 어떤 알림이 가나요?"
□ "상품 정보 수정 시에도 검수가 필요한가요?"
□ "대량 등록은 어떻게 처리하나요?"
```

#### 재고 관련 질문

```
□ "재고 데이터는 누가 관리하나요?"
□ "주문 시 재고는 어떻게 처리하나요?"
□ "재고 예약과 확정 차감이 다른가요?"
□ "재고 예약 유효 시간이 있나요?"
□ "재고가 0이 되면 자동으로 품절 처리되나요?"
□ "물류팀에서 입고 완료 알림이 오나요?"
□ "멀티 창고 재고는 어떻게 관리하나요?"
```

#### 가격 관련 질문

```
□ "가격 종류가 몇 가지 있나요? (정가/판매가/할인가)"
□ "판매자가 가격을 변경하면 바로 적용되나요?"
□ "급격한 가격 변동 시 검토가 필요한가요?"
□ "프로모션 할인가는 누가 관리하나요?"
□ "정가 변경 시 진행 중인 프로모션에 영향이 있나요?"
□ "가격 변경 이력을 기록하나요?"
```

#### 상품 상태 관련 질문

```
□ "상품이 가질 수 있는 상태는 무엇인가요?"
□ "품절과 판매중지의 차이가 있나요?"
□ "판매중지 사유는 어떤 것들이 있나요?"
□ "단종 처리는 어떻게 되나요?"
□ "품절 상품이 입고되면 자동으로 판매 재개되나요?"
□ "불량 신고가 들어오면 자동으로 판매중지 되나요?"
```

#### 검색/노출 관련 질문

```
□ "상품 등록 후 검색에 바로 노출되나요?"
□ "실시간 색인과 배치 색인의 기준은 무엇인가요?"
□ "광고 상품 노출은 어떻게 처리하나요?"
□ "판매중지 상품은 검색에서 어떻게 처리하나요?"
□ "색인 실패 시 어떻게 처리하나요?"
```

#### 판매자 연동 관련 질문

```
□ "판매자가 셀러센터에서 할 수 있는 것은 무엇인가요?"
□ "판매자 요청에 대한 승인 프로세스가 있나요?"

│     "주문팀은 '재고 차감 요청'을 발행하고,                 │
│      상품팀은 처리 후 '재고가 차감됨'을 발행합니다.        │
│      두 개는 다른 이벤트예요."                             │
│                                                             │
│  3. 연동 방향 명확화:                                      │
│     ┌──────────────────────────────────────────────────┐  │
│     │  주문팀 → 상품팀: 재고 차감 요청                 │  │
│     │  상품팀 → 주문팀: 재고 차감 완료 (결과)          │  │
│     │  물류팀 → 상품팀: 출고 완료 (입력)               │  │
│     │  상품팀 → 물류팀: 출고 요청 (필요 시)            │  │
│     └──────────────────────────────────────────────────┘  │
│                                                             │
│  4. 핫스팟으로 표시하고 넘어가기:                          │
│     "소유권이 불명확하면 핫스팟으로 표시해두고,            │
│      통합 워크샵에서 다른 팀과 함께 정하면 됩니다."        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 상황 2: 모든 데이터 변경을 이벤트로 적으려고 할 때

```
┌─────────────────────────────────────────────────────────────┐
│  상황 2: 모든 데이터 변경을 이벤트로 적으려고 할 때        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  증상:                                                     │
│  "상품명 변경, 설명 변경, 이미지 변경... 다 이벤트 아닌가요?"
│  "필드마다 이벤트를 만들어야 할 것 같은데요"              │
│                                                             │
│  대응:                                                     │
│                                                             │
│  1. 판단 기준 제시:                                        │
│     "이 변경이 일어나면 다른 팀이 즉시 뭔가를 해야 하나요? │
│      아니라면 별도 이벤트가 필요 없습니다."                │
│                                                             │
│  2. 구분 예시 제공:                                        │
│     "상품명 변경 → 검색팀이 배치로 재색인하면 됨 → 제외   │
│      가격 변경 → 프로모션팀이 할인 재계산 필요 → 이벤트"  │
│                                                             │
│  3. 그룹화 제안:                                           │
│     "비슷한 데이터 변경은 하나로 묶을 수도 있어요.         │
│      '상품 기본 정보가 수정되었다'로 묶거나,               │
│      아예 이벤트로 적지 않고 배치 색인으로 처리하거나."    │
│                                                             │
│  4. 핵심 이벤트 집중:                                      │
│     "먼저 핵심 비즈니스 이벤트에 집중합시다:               │
│      등록/검수/가격변경/재고변경/판매상태변경              │
│      나머지는 필요하면 나중에 추가하면 됩니다."            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 상황 3: 가격 이벤트 혼란 (정가/판매가/할인가)

```
┌─────────────────────────────────────────────────────────────┐
│  상황 3: 가격 관련 이벤트가 혼란스러울 때                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  증상:                                                     │
│  "'가격이 변경되었다'라고 적었는데 뭐가 변경된 건가요?"    │
│  "프로모션 할인도 가격 변경 아닌가요?"                     │
│                                                             │
│  대응:                                                     │
│                                                             │
│  1. 가격 종류 명확화:                                      │
│     "가격이 세 가지가 있습니다:                            │
│      • 정가: 상품의 원래 가격 (상품팀 관리)               │
│      • 판매가: 실제 판매 가격 (상품팀 관리)               │
│      • 할인가: 프로모션 중 가격 (프로모션팀 관리)"        │
│                                                             │
│  2. 이벤트 분리:                                           │
│     "각각 다른 이벤트로 적어주세요:                        │
│      • '정가가 변경되었다' (상품팀)                       │
│      • '판매가가 변경되었다' (상품팀)                     │
│      • '할인가가 적용되었다' (프로모션팀)"                │
│                                                             │
│  3. 연동 관계 설명:                                        │
│     "정가가 변경되면 프로모션팀이 할인율을 재계산해야 해요.│
│      할인가가 적용되면 상품팀이 표시 가격을 업데이트하고요.│
│      서로 알림을 주고받는 거죠."                           │
│                                                             │
│  4. 시각화:                                                │
│     ┌──────────────────────────────────────────────────┐  │
│     │  정가 100,000원 (상품팀)                         │  │
│     │       ↓                                           │  │
│     │  판매가 80,000원 (상품팀, 정가의 20% 할인)       │  │
│     │       ↓                                           │  │
│     │  할인가 64,000원 (프로모션팀, 판매가의 20% 할인) │  │
│     └──────────────────────────────────────────────────┘  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```
