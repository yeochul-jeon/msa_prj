# 이벤트 스토밍 부서별 워크샵 가이드

## 1. 개요

### 1.1 부서별 워크샵이란?

전체 참석자가 한자리에 모이기 어려운 상황에서, **부서 단위로 이벤트 스토밍을 진행**한 후 결과를 통합하는 방식입니다.

```
┌─────────────────────────────────────────────────────────────┐
│                     부서별 워크샵 방식                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   [부서A 워크샵]   [부서B 워크샵]   [부서C 워크샵]           │
│        │               │               │                   │
│        └───────────────┼───────────────┘                   │
│                        ▼                                    │
│                 [통합 워크샵]                               │
│                        │                                    │
│                        ▼                                    │
│              [바운디드 컨텍스트 확정]                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 언제 부서별 방식을 선택하는가?

| 상황 | 권장 방식 |
|------|-----------|
| 전 부서 일정 조율이 가능 | 통합 워크샵 (기존 방식) |
| 6시간 확보가 어려움 | **부서별 워크샵** |
| 참석 인원이 20명 이상 | **부서별 워크샵** |
| 부서 간 물리적 거리가 멀음 | **부서별 워크샵** |
| 도메인이 매우 복잡함 | **부서별 워크샵** (심층 탐구) |

### 1.3 장단점 비교

| 구분 | 장점 | 주의점 |
|------|------|--------|
| **일정** | 짧은 시간(2.5시간) 확보 용이 | 총 소요 시간은 더 길어질 수 있음 |
| **참여도** | 소규모로 집중도 높음 | 부서 간 시각 차이 발견이 어려움 |
| **깊이** | 해당 부서 도메인 심층 탐구 | 부서 간 연동 지점 누락 가능 |
| **결과물** | 부서별 명확한 산출물 | 통합 시 불일치 발생 가능 |

## 3. 부서별 워크샵 상세 가이드

### 3.1 사전 준비 (D-7일 전)

#### 체크리스트

```
□ 회의실 예약 (벽면 활용 가능한 공간)
□ 참석자 확정 (3-6명)
  - 팀장/파트장 (필수)
  - 선임 개발자 2-3명 (필수)
  - 기획/운영 담당자 1명 (권장)
□ 준비물 확보
  - 🟧 오렌지 포스트잇 (이벤트용) - 100장
  - 🟦 파란색 포스트잇 (커맨드용) - 50장
  - 🟨 노란색 포스트잇 (애그리게잇용) - 30장
  - 🩷 분홍색 포스트잇 (핫스팟용) - 20장
  - 🟩 연두색 포스트잇 (외부 연동용) - 30장  ← 부서별 워크샵 필수!
  - 마커펜 (검정, 굵은 것) - 참석자 수만큼
  - 마스킹테이프
□ 사전 안내 메일 발송
  - 워크샵 목적
  - 준비해올 것: 담당 업무 프로세스 생각해오기
□ 현재 시스템 자료 수집 (해당 부서 범위)
```

#### 참석자 구성 가이드

| 역할 | 인원 | 필수 여부 |
|------|------|-----------|
| 팀장/파트장 | 1명 | 필수 |
| 선임 개발자 | 2-3명 | 필수 |
| 기획/운영 담당자 | 1명 | 권장 |
| 다른 부서 참관자 | 0-1명 | 선택 |

> **Tip**: 다른 부서에서 1명이 참관하면, 연동 지점 식별이 훨씬 수월해집니다.

### 3.2 부서별 워크샵 진행 (2.5시간)

#### 타임라인 요약

| 시간 | Phase | 활동 | 소요 |
|------|-------|------|------|
| 00:00-00:15 | 오프닝 | 목표 공유 및 그라운드 룰 | 15분 |
| 00:15-00:25 | 개념 설명 | 이벤트 스토밍 핵심 요소 | 10분 |
| 00:25-01:00 | 이벤트 도출 | 도메인 이벤트 브레인스토밍 | 35분 |
| 01:00-01:10 | - | 휴식 | 10분 |
| 01:10-01:35 | 타임라인 정렬 | 시간 순서대로 배치 | 25분 |
| 01:35-01:55 | 요소 추가 | 커맨드/액터 식별 | 20분 |
| 01:55-02:10 | 요소 추가 | 애그리게잇/정책 추가 | 15분 |

#### Phase 1: 오프닝 (15분)

**목표**: 워크샵 목적 공유 및 편안한 분위기 조성

**스크립트 예시**:
```
"안녕하세요, 오늘 2시간 30분 동안 [부서명] 팀의 이벤트 스토밍을
진행하겠습니다.

오늘의 목표는 두 가지입니다:
1. 우리 부서에서 일어나는 주요 업무 이벤트를 시각화하기
2. 다른 부서와 주고받는 데이터/이벤트를 명확히 파악하기

정답은 없습니다. 평소 업무하시면서 느끼셨던 것들을
자유롭게 공유해 주세요."
```

**그라운드 룰**:
- 비판 없이 모든 의견 수용
- 포스트잇 1장 = 1개 이벤트
- 모르는 것은 핫스팟(분홍색)으로 표시

---

#### Phase 2: 개념 설명 (10분)

**핵심만 빠르게 전달**:

```
┌─────────────────────────────────────────────────────────────┐
│                    포스트잇 색상 가이드                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  🟧 오렌지: 도메인 이벤트                                   │
│     "~되었다" (과거형)                                      │
│     예: "주문이 접수되었다", "결제가 완료되었다"            │
│                                                             │
│  🟦 파란색: 커맨드 (명령)                                   │
│     "~하기" (명령형)                                        │
│     예: "주문하기", "결제하기"                              │
│                                                             │
│  🟨 노란색: 애그리게잇 (데이터 묶음)                        │
│     명사형                                                  │
│     예: "주문서", "상품", "회원"                            │
│                                                             │
│  🩷 분홍색: 핫스팟 (불확실/문제)                            │
│     논의가 필요한 부분                                      │
│                                                             │
│  🟩 연두색: 외부 시스템/연동                                │
│     다른 부서나 외부 API                                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### Phase 3: 도메인 이벤트 도출 (35분)

**진행 방식**:

1. **개별 작업 (15분)**:
   - 각자 담당 업무에서 발생하는 이벤트를 포스트잇에 작성
   - 말하지 않고 조용히 작성

2. **붙이기 (5분)**:
   - 화이트보드에 순서 상관없이 붙이기

3. **공유 및 보완 (15분)**:
   - 각자 붙인 포스트잇 설명
   - 중복 제거 및 누락 이벤트 추가

**퍼실리테이터 질문**:
```
- "고객이 처음 우리 서비스를 접할 때 무슨 일이 일어나나요?"
- "이 작업이 끝나면 다음에 어떤 일이 발생하나요?"
- "실패하거나 취소될 때는 어떤 이벤트가 발생하나요?"
- "정기적으로 발생하는 업무가 있나요? (배치, 정산 등)"
```

---

#### Phase 4: 타임라인 정렬 (25분)

**목표**: 이벤트를 시간 순서대로 배치

```
┌─────────────────────────────────────────────────────────────┐
│                    화이트보드 레이아웃                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ←────────────── 시간 흐름 ──────────────→                  │
│                                                             │
│  [시작]                                             [종료]  │
│                                                             │
│  🟧 ──→ 🟧 ──→ 🟧 ──→ 🟧 ──→ 🟧 ──→ 🟧                    │
│                                                             │
│  ↑                                                          │
│  처음 발생하는                                              │
│  이벤트                                                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**진행 방식**:
1. "가장 먼저 발생하는 이벤트는?" → 왼쪽에 배치
2. "그 다음에는?" → 순서대로 배치
3. 병렬로 발생하는 이벤트는 위아래로 배치
4. 불확실한 순서는 핫스팟(🩷) 표시

---

#### Phase 5: 커맨드/액터 추가 (20분)

**커맨드 추가 (10분)**:
- 각 이벤트를 발생시키는 **행동**을 파란색 포스트잇에 작성
- 이벤트 왼쪽에 배치

```
  🟦           🟧
[주문하기] → [주문이 접수되었다]
```

**액터 식별 (10분)**:
- 커맨드를 실행하는 **주체**를 작은 포스트잇이나 마커로 표시
- 예: 고객, 관리자, 시스템, 배치

---

#### Phase 6: 애그리게잇/정책 추가 (15분)

**애그리게잇 (10분)**:
- 관련 이벤트들을 묶는 핵심 데이터 객체
- 노란색 포스트잇으로 이벤트 위에 배치

```
        🟨
      [주문서]
         │
  🟧 ─── 🟧 ─── 🟧
  주문    결제    배송
  접수    완료    시작
```

**정책 (5분)**:
- "만약 ~하면, ~한다" 규칙
- 연보라색 포스트잇 (없으면 다른 색으로 표시)

---

#### Phase 7: 외부 연동 지점 식별 (15분) ⭐ 핵심

> **이 단계가 부서별 워크샵의 가장 중요한 차별점입니다!**

**목표**: 다른 부서와 주고받는 이벤트/데이터를 명확히 식별

**진행 방식**:

```
┌─────────────────────────────────────────────────────────────┐
│                  외부 연동 지점 식별                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  [입력 영역]              [부서 내부]           [출력 영역]  │
│  ← 받는 것                                      보내는 것 → │
│                                                             │
│  ┌──────────┐                               ┌──────────┐   │
│  │ 🟩       │                               │ 🟩       │   │
│  │ 주문팀에서│                               │ 배송팀에 │   │
│  │ 주문접수 │     🟧──🟧──🟧──🟧           │ 출고요청 │   │
│  │ 전달받음 │                               │ 전달     │   │
│  │ ←        │                               │       → │   │
│  └──────────┘                               └──────────┘   │
│                                                             │
│  ┌──────────┐                               ┌──────────┐   │
│  │ 🟩       │                               │ 🟩       │   │
│  │ PG사에서 │                               │ 알림서비스│   │
│  │ 결제승인 │                               │ 에 푸시  │   │
│  │ 수신     │                               │ 요청     │   │
│  │ ←        │                               │       → │   │
│  └──────────┘                               └──────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**퍼실리테이터 질문**:

```
입력 (다른 곳에서 받는 것):
- "이 작업을 시작하려면 다른 부서에서 뭘 받아야 하나요?"
- "외부 시스템에서 들어오는 데이터가 있나요?"
- "다른 팀의 어떤 작업이 완료되어야 우리가 시작할 수 있나요?"

출력 (다른 곳으로 보내는 것):
- "이 작업이 끝나면 다른 부서에 알려줘야 하는 게 있나요?"
- "외부 시스템으로 전송하는 데이터가 있나요?"
- "우리 작업 결과를 기다리는 다른 팀이 있나요?"
```

**연두색 포스트잇 작성법**:

```
┌────────────────────────────────┐
│  🟩 연두색 포스트잇            │
├────────────────────────────────┤
│                                │
```

**목표**: 용어, 프로세스, 데이터 정의 통일

**진행 방식**:
1. 사전 분석에서 식별된 불일치 항목 제시
2. 관련 부서 담당자 간 논의

## 6. FAQ 및 트러블슈팅

### 6.1 부서별 워크샵 FAQ

**Q: 한 부서 인원이 2명밖에 안 되는데 진행 가능한가요?**

A: 가능하지만 권장하지 않습니다. 최소 3명은 되어야 다양한 관점이 나옵니다.
인원이 부족하면:
- 관련 부서에서 1명 참관 요청
- 해당 부서 출신 다른 팀원 초대
- 또는 유사 부서와 합동 워크샵 진행

---

**Q: 부서별 워크샵에서 바운디드 컨텍스트까지 확정해도 되나요?**

A: "잠정 후보"까지만 도출하세요.
최종 확정은 통합 워크샵에서 전체 맥락을 보고 해야 합니다.
부서별로 확정하면 나중에 조정이 어려워집니다.

---

**Q: 외부 연동 지점이 너무 많으면 어떻게 하나요?**

A: 우선순위를 정하세요:
1. 매일 발생하는 핵심 연동
2. 장애 시 영향도가 큰 연동
3. 가끔 발생하지만 중요한 연동

워크샵에서는 1, 2번에 집중하고, 3번은 나중에 보완합니다.

---

**Q: 다른 부서와 연동 지점에 대해 의견이 다르면?**

A: 핫스팟(분홍색)으로 표시하고 통합 워크샵에서 해결합니다.
부서별 워크샵에서 무리하게 결론 내리지 마세요.

---

### 6.2 통합 워크샵 FAQ

**Q: 부서별 결과물의 용어가 다 다른데 어떻게 통일하나요?**

A: 통합 워크샵 전에 퍼실리테이터가 사전 분석에서 불일치를 식별합니다.
통합 워크샵에서:
1. 불일치 목록 제시
2. 관련 부서 담당자 논의
3. 다수결 또는 도메인 전문가 의견 존중
4. 결정된 용어를 "용어 사전"에 기록

---

**Q: 통합 워크샵에 참석 못하는 부서가 있으면?**

A: 해당 부서 결과물을 대신 발표할 사람을 지정하세요.
가능하면 해당 부서와 가장 많이 연동하는 부서 담당자가 대리 발표.
중요한 결정 사항은 나중에 해당 부서와 재확인.

---

**Q: 바운디드 컨텍스트가 부서 구조와 완전히 다르게 나오면?**

A: 정상입니다. 기술적 최적 구조와 조직 구조가 다른 경우가 많습니다.

선택지:
1. 조직 구조를 컨텍스트에 맞게 재편 (장기적 권장)
2. 조직은 유지하되 서비스 경계는 컨텍스트 기준으로
3. 현실적 절충 (일부 조정)

---

### 6.3 트러블슈팅

**상황 1: 부서 간 책임 공방**

```
증상: "이건 우리 업무가 아니에요", "원래 저쪽에서 해야 하는데..."
대응:
1. 핫스팟으로 표시하고 일단 넘어가기
2. "책임이 어디인지보다, 현재 어떻게 되고 있는지에 집중하죠"
3. 통합 워크샵에서 전체 맥락 보고 결정
```

**상황 2: 너무 상세하게 들어가려는 참석자**

```
증상: 테이블 컬럼 수준까지 정의하려고 함
대응:
1. "오늘은 큰 그림을 그리는 시간입니다"
2. "그 수준의 상세화는 설계 단계에서 진행합니다"
3. 상세 논의가 필요한 항목은 별도 메모
```

**상황 3: 연동 지점이 하나도 안 나올 때**

```
증상: "우리 부서는 다른 팀이랑 연동하는 게 없어요"
대응:
질문 시도:
- "어떤 부서가 보내준 정보로 시작하나요?"
- "작업 결과를 누가 받아가나요?"
- "외부 API를 호출하는 게 있나요?"
- "배치로 데이터를 주고받는 게 있나요?"
```

**상황 4: 통합 워크샵에서 불일치가 너무 많을 때**

```
증상: 검증해야 할 불일치가 30개 이상
대응:
1. 우선순위 정하기 (임팩트 기준)
2. 상위 10개만 워크샵에서 해결
3. 나머지는 후속 회의로 분리
4. 핵심 연동부터 해결
```
