## 7. 퍼실리테이터 실전 스크립트

### 7.1 워크샵 오프닝 스크립트

```
┌─────────────────────────────────────────────────────────────────┐
│             채널 시스템 워크샵 오프닝 스크립트                   │
│             (협력사 담당자 대상 — 이벤트 스토밍 간략 설명 포함)  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  "안녕하세요, 오늘 채널 시스템 이벤트 스토밍 워크샵을           │
│  진행할 [이름]입니다.                                          │
│                                                                 │
│  먼저 이벤트 스토밍이 뭔지 간단히 설명드리겠습니다.            │
│  이벤트 스토밍은 비즈니스 프로세스에서 일어나는                │
│  '중요한 사건(이벤트)'을 포스트잇에 적어서                     │
│  시간 순서대로 벽에 붙이는 워크샵입니다.                       │
│                                                                 │
│  포스트잇 색깔마다 의미가 다릅니다:                            │
│  • 🟧 주황색: 이벤트 (일어난 사건, 과거형)                    │
│    예: '방송이 시작되었다'                                     │
│  • 🟦 파란색: 커맨드 (명령, 누가 뭘 했는지)                   │
│    예: '편성표 확정하기'                                       │
│  • 🟨 노란색: 애그리게잇 (변경되는 데이터 묶음)               │
│    예: '편성표', '방송'                                        │
│  • 💜 보라색: 정책 (자동으로 적용되는 규칙)                    │
│    예: '편성 확정 시 온에어에 자동 반영'                       │
│  • 🩷 분홍색: 핫스팟 (논의 필요, 모르겠는 부분)               │
│  • 🟩 초록색: 외부 시스템 (우리 팀 밖의 시스템)               │
│    예: '상품 서비스', '주문 서비스'                            │
│                                                                 │
│  시작하기 전에 몇 가지 부탁드릴 점이 있습니다:                 │
│                                                                 │
│  ① 오늘은 7개 서비스를 다루는데, 각 서비스별로 이벤트를        │
│     구분해주세요. '이 이벤트는 편성인가, 온에어인가?'          │
│     라고 생각해보시면 됩니다.                                  │
│                                                                 │
│  ② 온에어처럼 '조회만 하는' 서비스가 있습니다.                │
│     조회는 이벤트가 아니라 '읽기 모델'로 표현합니다.           │
│     자세한 건 해당 서비스 차례에 설명드리겠습니다.              │
│                                                                 │
│  ③ 내부 시스템(상품/주문)의 이벤트는 초록색으로 표시합니다.    │
│     우리(채널)의 이벤트와 구분하는 게 중요합니다.              │
│                                                                 │
│  ④ 모르겠는 건 핫스팟(분홍색)으로 표시하면 됩니다.             │
│     특히 Fade out 가능성이 있는 서비스 범위는                  │
│     핫스팟으로 표시하고 함께 논의합시다.                       │
│                                                                 │
│  완벽한 정답은 없습니다. 편하게 진행해주세요!"                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 7.2 상황별 개입 스크립트 (10개 상황)

#### "이 이벤트가 편성인가요, 온에어인가요?"

```
┌─────────────────────────────────────────────────────────────────┐
│     상황: 서비스 간 이벤트 소유권 혼동                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  대응:                                                         │
│  "좋은 질문입니다! 기준은 '누가 데이터를 변경하는가'입니다.    │
│                                                                 │
│  편성 서비스:                                                  │
│  • 편성표를 만들고, 확정하고, 변경하는 건 편성                 │
│  • '편성표가 확정되었다' = 편성의 이벤트                       │
│                                                                 │
│  온에어 서비스:                                                │
│  • 방송을 시작하고, 종료하는 건 온에어                         │
│  • '방송이 시작되었다' = 온에어의 이벤트                       │
│                                                                 │
│  헷갈리면 이렇게 생각해보세요:                                 │
│  '이 데이터가 저장되는 DB는 어느 서비스에 있나요?'             │
│  그 서비스가 이벤트의 주인입니다."                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### "온에어는 조회만 하는데 이벤트가 있나요?"

```
┌─────────────────────────────────────────────────────────────────┐
│     상황: Read 위주 서비스의 이벤트 표현 질문                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  대응:                                                         │
│  "맞습니다. 온에어는 대부분 '보여주기'만 합니다.               │
│                                                                 │
│  보여주기(조회)는 '읽기 모델'이라고 합니다.                    │
│  연두색 포스트잇에 적어주세요:                                 │
│  📖 '편성정보 읽기모델'                                        │
│  📖 '실시간주문현황 읽기모델'                                  │
│                                                                 │
│  하지만 온에어도 소수의 이벤트는 있습니다:                     │
│  🟧 '방송이 시작되었다' — 이건 온에어만 알 수 있죠            │
│  🟧 '방송이 종료되었다'                                       │
│  🟧 '방송 중 상품이 교체되었다'                               │
│                                                                 │
│  정리하면:                                                     │
│  • 보여주기 → 📖 읽기 모델 (연두색)                            │
│  • 상태 바꾸기 → 🟧 이벤트 (주황색)                           │
│  온에어는 읽기 모델이 많고, 이벤트가 적은 서비스입니다."       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### "PDSS 변경이 다른 서비스에 다 영향을 주나요?"

```
┌─────────────────────────────────────────────────────────────────┐
│     상황: PDSS 기반 데이터 전파 범위 질문                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  대응:                                                         │
│  "네, PDSS는 채널의 '마스터 데이터'를 관리합니다.              │
│                                                                 │
│  채널 코드가 바뀌면? → 편성, 온에어, PGM 다 영향받죠.          │
│  프로그램유형이 바뀌면? → 편성, 기획이 영향받습니다.           │
│                                                                 │
│  하지만 모든 PDSS 변경이 이벤트인 건 아닙니다:                 │
│  • 관리 화면 설정 변경? → 내부 처리 (이벤트 아님)             │
│  • 채널 코드 변경? → 전체 영향 (이벤트!)                      │
│                                                                 │
│  기준은 '다른 서비스가 이 변경을 알아야 하는가?'입니다.        │
│  알아야 하면 이벤트, 아니면 내부 처리입니다."                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### "편성을 여러 번 수정하는데 매번 이벤트인가요?"

```
┌─────────────────────────────────────────────────────────────────┐
│     상황: 편성 임시 저장/수정 빈도 질문                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  대응:                                                         │
│  "초안을 여러 번 수정하는 건 이벤트가 아닙니다.                │
│                                                                 │
│  비유하면, 회의록을 여러 번 고치는 건 중요하지 않지만,         │
│  '회의록이 배포되었다'는 중요하죠.                             │
│                                                                 │
│  편성도 마찬가지입니다:                                        │
│  • 초안 수정, 임시 저장 → 편성 내부 → 이벤트 아님             │
│  • 편성표 확정! → 온에어가 반응 → 이벤트 🟧                   │
│                                                                 │
│  '확정' 또는 '변경 확정'만 이벤트로 적어주세요."               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

```
└─────────────────────────────────────────────────────────────────┘
```

```
│                                                                 │

│  초록색은 '우리 팀 밖의 시스템'이라는 표시입니다.              │
│  우리가 소유하는 이벤트(주황색)와 구분해주세요."               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### "TV 채널과 모바일 라이브를 구분해야 하나요?"

```
┌─────────────────────────────────────────────────────────────────┐
│     상황: 채널 유형별 이벤트 구분 질문                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  대응:                                                         │
│  "프로세스가 같으면 구분하지 않아도 됩니다.                    │
│                                                                 │
│  TV든 모바일이든 '편성 → 방송 → 정산' 흐름이 같다면:          │
│  🟧 '편성표가 확정되었다' 하나로 충분합니다.                  │
│  채널 유형(TV/모바일)은 이벤트의 속성(데이터)에 포함하면 돼요. │
│                                                                 │
│  하지만 프로세스 자체가 다르다면 구분이 필요합니다:            │
│  예를 들어 TV는 '송출 시작'이고                                │
│  모바일은 '스트리밍 시작'이라면                                │
│  별도 이벤트로 나눌 수 있습니다.                               │
```

## 9. FAQ 및 트러블슈팅

### 9.1 자주 묻는 질문

#### 워크샵 준비 관련

```
Q1: "협력사 담당자가 이벤트 스토밍을 처음 접하는데 괜찮은가요?"
A: 워크샵 오프닝에서 10분 정도 간략 설명을 포함합니다.
   포스트잇 색상과 의미만 알면 됩니다.
   사전에 간단한 소개 자료를 공유하면 더 좋습니다.

Q2: "7개 서비스를 한 번에 다 해야 하나요?"
A: Option A(3일)를 권장합니다. 시간축별로 나누어 진행하면
   참석자 부담을 줄이면서 서비스 간 흐름도 잡을 수 있습니다.

Q3: "각 서비스 담당자가 다른데, 전체 워크샵에 다 참석해야 하나요?"
A: 자기 서비스 세션은 필수, 연동 서비스 세션은 권장입니다.
   최소한 Day 3(통합)에는 모든 서비스 대표가 참석해야 합니다.

Q4: "내부 시스템(상품/주문) 담당자도 참석해야 하나요?"
A: Day 2 온에어 세션에 내부 시스템 담당자 1명이 참석하면 좋습니다.
   참석이 어려우면, 사전에 연동 포인트를 정리해두세요.

Q5: "사전에 어떤 자료를 준비해야 하나요?"
A: ① 7개 서비스 개요 1장짜리 요약 (참석자 공유)
   ② 현재 시스템 아키텍처 다이어그램 (있으면)
   ③ 방송 프로세스 흐름도 (간략한 수준)
   ④ 이벤트 스토밍 간략 소개 자료
```

#### 진행 중 관련

```
Q6: "편성과 기획의 경계가 헷갈린다고 합니다."
A: "편성은 '언제, 어느 채널에 뭘 틀 것인가'이고,
    기획은 '무엇을 어떻게 만들 것인가'입니다.
    편성이 시간표라면, 기획은 수업 준비입니다."

Q7: "온에어에서 읽기 모델을 어떻게 표현하나요?"
A: 연두색 포스트잇에 📖 기호와 함께 적습니다.
   "편성정보 읽기모델 ← 편성표 확정 이벤트"처럼
   원본 이벤트를 화살표로 연결합니다.
   자세한 방법은 읽기모델 가이드를 참조하세요.

Q8: "Fade out 서비스 논의에 시간을 너무 많이 쓰고 있어요."
A: 핫스팟으로 표시하고 넘어가세요.
   "Fade out 여부는 오늘 결정할 사안이 아닙니다.
    핵심 입출력만 정리하고, 상세는 후속 논의로 남깁시다."

Q9: "참석자가 CRUD 수준으로 너무 세밀하게 적고 있어요."
A: "이벤트 스토밍에서는 '비즈니스 관점'이 중요합니다.
    '편성항목 레코드가 INSERT되었다'가 아니라,
    '편성표가 확정되었다'입니다.
    다른 사람이 들어도 이해할 수 있는 말로 적어주세요."

Q10: "한 서비스의 이벤트가 너무 많이 나왔어요."
A: 기준을 다시 확인합니다:
   "이 이벤트로 다른 서비스가 반응해야 하나요?"
   아니라면 내부 처리이므로 제외합니다.
   서비스당 핵심 이벤트 5~10개가 적정 수준입니다.
```

#### 결과 정리 관련

```
Q11: "편성 → 온에어 연동이 가장 중요한데, 어떻게 강조하나요?"
A: 시간축 흐름도에서 이 연동을 ⭐ 표시하고,
   별도로 "핵심 연동 포인트"로 문서화합니다.
   비실시간→실시간 전환점이므로 장애격리 전략도 논의합니다.

Q12: "바운디드 컨텍스트를 7개로 확정해도 되나요?"
A: MSA 후보 서비스가 7개로 이미 확정되어 있으므로,
   워크샵에서는 각 서비스가 독립 컨텍스트인지 "검증"합니다.
   독립배포/독립DB/독립팀 기준으로 확인하면 됩니다.

Q13: "워크샵 결과를 어떤 형태로 정리하나요?"
A: 결과물 템플릿(8.2절)을 활용하세요.
   핵심은: ① 서비스별 이벤트 목록
           ② 서비스 간 연동 매트릭스
           ③ 핫스팟 목록 (후속 논의 대상)

Q14: "이벤트 스토밍 결과를 개발에 바로 쓸 수 있나요?"
A: 이벤트 스토밍은 도메인 이해와 서비스 경계 확인이 목적입니다.
   개발 설계(API, DB 등)는 별도 단계에서 진행하며,
   이벤트 스토밍 결과를 입력으로 활용합니다.

Q15: "워크샵 후 변경사항이 생기면 어떻게 하나요?"
A: 이벤트 스토밍 결과는 "살아있는 문서"입니다.
   변경 시 결과물을 업데이트하고, 영향받는 서비스를
   확인하여 관련 담당자에게 공유합니다.
```

```
│                                                                 │

│                                                                 │
│  대응:                                                         │

│   🟦 커맨드는 '~하기' (명령) → 파란색                         │
```
