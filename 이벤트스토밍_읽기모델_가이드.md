## 2. 읽기 모델이란?

### 2.1 식당 비유로 이해하기

```
┌─────────────────────────────────────────────────────────────┐
│                    식당 비유로 이해하기                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  📋 주문서 (쓰기/Write)                                     │
│  → "김치찌개 1개 주문합니다" (행동/변경)                    │
│  → 주방에 전달됨 → 상태가 바뀜                             │
│                                                             │
│  📺 전광판 (읽기/Read)                                      │
│  → "현재 대기 3팀, 예상 20분"                              │
│  → 손님이 보는 화면 → 상태를 변경하지 않음                  │
│  → 여러 정보를 조합해서 보여줌                              │
│                                                             │
│  ─────────────────────────────────────────────────────────  │
│                                                             │
│  핵심: 주문서(쓰기)와 전광판(읽기)은                       │
│        목적이 다르니까 구조도 다르다!                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 이벤트 스토밍 요소와 비교

기존에 배운 포스트잇 요소들과 비교하면:

| 요소 | 색상 | 역할 | 방향 |
|------|------|------|------|
| 커맨드 🟦 | 파란색 | "주문을 접수해라" | **쓰기** (상태 변경) |
| 이벤트 🟧 | 오렌지 | "주문이 접수되었다" | 쓰기의 **결과** |
| 정책 💜 | 보라색 | "VIP면 자동 할인" | 자동 **반응** |
| **읽기 모델** 🟩 | 연두/초록 | "주문 현황 대시보드" | **읽기** (조회만) |

```
  🟦 커맨드          🟧 이벤트          🟩 읽기 모델
  (행동하기)    →    (결과 발생)    →    (화면에 보여주기)

  "주문 접수해라"    "주문이 접수됨"     "내 주문 목록"
                                         "배송 추적 화면"
```

### 2.3 읽기 모델이 필요한 이유

```
┌─────────────────────────────────────────────────────────────┐
│         왜 "읽기"를 따로 생각해야 하나요?                    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ❌ 읽기 모델 없이 생각하면:                                │
│                                                             │
│  "사용자가 주문 내역을 본다"                               │
│  → 그래서? 이벤트로 적어야 하나?                           │
│  → "조회됨"은 이벤트가 아닌데...                           │
│  → 혼란!                                                   │
│                                                             │
│  ✅ 읽기 모델로 정리하면:                                   │
│                                                             │
│  "사용자에게 주문 현황을 보여줘야 한다"                     │
│  → 🟩 읽기 모델: "주문 현황 뷰"                            │
│  → 어떤 데이터가 필요한지 명확해짐                         │
│  → 어떤 이벤트들이 이 화면을 갱신하는지 연결됨             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 3. 이벤트 스토밍에서의 표현법

### 3.1 이벤트 → 읽기 모델 → 커맨드 흐름도

```
  🟧 가격이         🟧 재고가         🟧 리뷰가
     변경되었다        차감되었다        등록되었다
       │                │                │
       └────────────────┼────────────────┘
                        │
                        ▼
              ┌──────────────────┐
              │  🟩 상품 상세     │
              │     페이지 뷰    │
              │                  │
              │  - 상품 정보     │
              │  - 재고 상태     │
              │  - 리뷰 요약     │
              └──────────────────┘
                        │
                        ▼
                  👤 사용자가
                     화면을 봄
                        │
                        ▼
              🟦 "장바구니에 담기"
                   (다음 커맨드)
```

### 3.2 실제 예시: 상품 상세 페이지

상품서비스팀에 친숙한 예시로 설명하면:

```
┌─────────────────────────────────────────────────────────────┐
│           🟩 읽기 모델: "상품 상세 페이지"                   │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  이 화면에 필요한 데이터:                                   │
│  ┌───────────────────────────────────────────┐              │
│  │ • 상품명, 가격, 설명        ← 상품 도메인 │              │
│  │ • 재고 수량 (품절 여부)     ← 재고 도메인 │              │
│  │ • 평균 별점, 리뷰 수        ← 리뷰 도메인 │              │
│  │ • 배송 예정일               ← 배송 도메인 │              │
│  │ • 적용 가능한 쿠폰          ← 프로모션    │              │
│  └───────────────────────────────────────────┘              │
│                                                             │
│  이 화면을 갱신하는 이벤트들:                               │
│  🟧 "가격이 변경되었다"       → 가격 표시 갱신             │
│  🟧 "재고가 차감되었다"       → 품절 여부 갱신             │
│  🟧 "리뷰가 등록되었다"       → 별점/리뷰 수 갱신          │
│  🟧 "쿠폰이 발행되었다"       → 적용 가능 쿠폰 갱신       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**핵심 포인트:** 하나의 읽기 모델(화면)에 **여러 도메인의 데이터**가 조합됩니다. 이것이 읽기 모델을 따로 표현하는 가장 큰 이유입니다.

## 4. 워크샵에서 읽기 모델 도출하기

### 4.1 도출 시점: Phase 5.5

```
⚠️ 중요: 읽기 모델은 Phase 5 이후에 추가합니다

  Phase 1~3: 이벤트 → 타임라인 → 커맨드
  Phase 4:   애그리게잇
  Phase 5:   정책
  ─────────────────────────────────────
  Phase 5.5: 👉 읽기 모델 (여기서 추가!)
  Phase 6:   바운디드 컨텍스트
```

> 읽기 모델부터 시작하면 **화면 설계**가 되어버립니다.
> 반드시 이벤트 흐름을 먼저 완성한 뒤에 진행하세요.

### 4.2 3단계 프로세스

```
┌─────────────────────────────────────────────────────────────┐
│              읽기 모델 도출 3단계 프로세스                    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Step 1: 이벤트 흐름 완성 후 시작                           │
│  ─────────────────────────────────                          │
│  → 이벤트(🟧), 커맨드(🟦), 정책(💜)을 먼저 다 붙인 뒤     │
│  → "사용자가 이 커맨드를 실행하려면 뭘 봐야 하지?"         │
│                                                             │
│  Step 2: 커맨드 앞에 "화면"을 질문                          │
│  ─────────────────────────────────                          │
│  → 각 🟦 커맨드 앞에서 멈추고                              │
│  → "이 행동을 하기 위해 어떤 정보를 보고 있었나요?"        │
│                                                             │
│  Step 3: 화면에 필요한 데이터 소스 연결                     │
│  ─────────────────────────────────                          │
│  → 🟩 읽기 모델에 어떤 이벤트들이 데이터를 공급하는지      │
│  → 화살표로 연결                                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### Step 1: 이벤트 흐름이 먼저!

이벤트(🟧), 커맨드(🟦), 정책(💜)을 모두 배치한 뒤에 시작합니다. 읽기 모델을 먼저 도출하면 UI 설계와 혼동되므로, 반드시 비즈니스 흐름 완성 후 진행하세요.

#### Step 2: 커맨드 앞에서 질문하기

퍼실리테이터가 각 커맨드(🟦) 앞에서 던지는 질문:

```
┌─────────────────────────────────────────────────────────────┐
│           퍼실리테이터 스크립트                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  🟦 "장바구니에 담기" 커맨드 앞에서:                        │
│                                                             │
│  "고객이 '장바구니에 담기'를 누르려면                       │
│   그 직전에 어떤 화면을 보고 있었나요?                      │
│   그 화면에는 어떤 정보들이 있었나요?"                      │
│                                                             │
│  참석자 답변 예시:                                          │
│  → "상품 상세 페이지요"                                    │
│  → "가격, 재고, 옵션 선택, 리뷰..."                       │
│                                                             │
│  → 🟩 읽기 모델: "상품 상세 뷰" 포스트잇 작성!             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### Step 3: 데이터 소스 연결

```
  🟧 상품이        🟧 가격이        🟧 리뷰가        🟧 쿠폰이
     등록되었다       변경되었다       등록되었다       발행되었다
       │               │               │               │
       │   ┌───────────┘               │               │
       │   │   ┌───────────────────────┘               │
       │   │   │   ┌───────────────────────────────────┘
       ▼   ▼   ▼   ▼
  ┌──────────────────────┐
  │  🟩 상품 상세 뷰      │         👤 사용자
  │                       │    ──────────────────
  │  • 상품 기본 정보     │    "이 화면을 보고"
  │  • 현재가 / 할인가    │    "결정을 내린다"
  │  • 리뷰 평점 (4.2)   │
  │  • 사용 가능 쿠폰 2장 │
  └──────────┬────────────┘
             │
             ▼
     🟦 "장바구니에 담기"
        (다음 커맨드)
```

### 4.3 퍼실리테이터 진행 스크립트

| 단계 | 스크립트 예시 |
|------|-------------|
| 도입 | "지금까지 '무엇이 일어나는가'를 정리했으니, 이제 '사용자가 무엇을 보는가'를 정리해봅시다." |
| 질문 | "이 커맨드를 실행하기 전에 사용자가 어떤 화면을 보고 있었나요?" |
| 확장 | "이 화면에 표시되는 정보는 어떤 도메인에서 오나요?" |
| 연결 | "그럼 이 이벤트들이 발생하면 이 화면이 갱신되겠네요. 화살표로 연결합시다." |
| 정리 | "이 읽기 모델을 보면 3개 도메인의 데이터가 필요하네요. 이건 MSA 전환 시 중요한 연동 지점입니다." |

## 5. 상품서비스팀 읽기 모델 예시

### 5.1 고객 접점 읽기 모델

| 🟩 읽기 모델 | 포함 데이터 | 공급하는 도메인 | 다음 커맨드 🟦 |
|-------------|------------|----------------|--------------|
| 상품 목록 뷰 | 상품명, 썸네일, 가격, 품절 여부 | 상품, 재고, 가격 | "상품 상세 보기" |
| 상품 상세 뷰 | 상세 정보, 옵션, 재고, 리뷰, 쿠폰 | 상품, 재고, 리뷰, 프로모션 | "장바구니에 담기" |
| 검색 결과 뷰 | 검색된 상품, 필터, 정렬 | 상품, 검색, 카테고리 | "상품 상세 보기" |
| 장바구니 뷰 | 담긴 상품, 수량, 가격 합계, 배송비 | 장바구니, 상품, 배송 | "주문하기" |

### 5.2 관리자/판매자 접점 읽기 모델

| 🟩 읽기 모델 | 포함 데이터 | 공급하는 도메인 | 다음 커맨드 🟦 |
|-------------|------------|----------------|--------------|
| 상품 관리 대시보드 | 등록 상품 목록, 상태, 재고 현황 | 상품, 재고 | "상품 수정하기" |
| 검수 대기 목록 | 검수 대기 상품, 제출일, 판매자 | 상품, 검수 | "검수 승인하기" |
| 재고 현황 뷰 | SKU별 재고, 입출고 이력 | 재고, 입고 | "재고 조정하기" |

### 5.3 보드 배치 예시: 상품 등록 → 판매 흐름

 👤 판매자                                      👤 검수자

    (카테고리 목록,      등록되었다      목록 뷰           완료되었다

       │                    │           판매자 정보)           │
       ▼                    ▼              │                   ▼
 🟦 상품 등록하기      💜 자동 검수       ▼             🟧 상품이
                          요청          🟦 검수             승인되었다
                                          승인하기              │
                                                               ▼
                                                         🟧 판매가
                                                            시작되었다
                                                               │
                                                               ▼
                                              ┌────────────────────────┐
                                              │ 🟩 상품 상세 뷰        │
                                              │    (고객에게 노출)      │
                                              │    • 상품 정보          │
                                              │    • 가격/재고          │
                                              │    • 리뷰              │
                                              └───────────┬────────────┘
                                                          │
                                                          ▼
                                                  🟦 장바구니에 담기
                                                     (👤 고객)
```

## 6. MSA 전환에서의 가치

### 6.1 바운디드 컨텍스트 경계 발견

읽기 모델을 붙이면 **바운디드 컨텍스트의 경계**가 자연스럽게 드러납니다.

```
┌─────────────────────────────────────────────────────────────┐
│       읽기 모델이 알려주는 것                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  🟩 읽기 모델 하나에 여러 도메인이 엮여 있다면?            │
│                                                             │
│  "상품 상세 뷰"                                             │
│    ← 상품 도메인                                           │
│    ← 재고 도메인                                           │
│    ← 리뷰 도메인                                           │
│    ← 프로모션 도메인                                       │
│                                                             │
│  → 이 도메인들 사이에 "데이터 연동"이 필요하다는 신호!     │
│  → MSA 전환 시 이 연동 지점이 API / 이벤트 통신이 됨       │
│                                                             │
│  ─────────────────────────────────────────────────────────  │
│                                                             │
│  즉, 읽기 모델은 "누가 누구의 데이터를 필요로 하는가"를   │
│  시각적으로 보여주는 도구입니다.                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 6.2 서비스 간 데이터 의존 관계 시각화

| 발견 | MSA 설계에 주는 의미 |
|------|---------------------|
| 🟩 뷰 하나에 3개 도메인 데이터 | → 3개 서비스 간 API 호출 또는 이벤트 구독 필요 |
| 🟩 뷰가 특정 도메인 데이터만 사용 | → 해당 서비스 내부에서 해결 가능 |
| 🟩 뷰에 실시간 갱신이 필요 | → 이벤트 기반 비동기 처리 고려 |
| 🟩 뷰가 과도하게 많은 도메인 참조 | → 🩷 핫스팟! 서비스 경계 재검토 필요 |

```

## 8. 요약

### Part 1 요약 — 개념 이해

| 질문 | 답변 |
|------|------|
| 읽기 모델이 뭔가요? | 사용자가 **보는 화면/조회 뷰**를 표현한 것 |
| 왜 필요한가요? | 여러 도메인 데이터가 **하나의 화면에 조합**되기 때문 |
| 포스트잇 색상은? | 🟩 연두/초록색 (외부 시스템과 같은 색, 또는 별도 표기) |
| 언제 사용하나요? | 이벤트 → 화면 → 다음 커맨드 흐름을 표현할 때 |

### Part 2 요약 — 워크샵 실전

| 항목 | 핵심 |
|------|------|
| **도출 시점** | Phase 5(정책) 이후, Phase 6(바운디드 컨텍스트) 이전 |
| **도출 질문** | "이 커맨드를 실행하기 전에 어떤 화면을 보고 있었나요?" |
| **MSA 가치** | 서비스 간 데이터 의존 관계가 시각적으로 드러남 |
| **권장 적용** | Option B — 핵심 화면 2~3개만 (15~20분) |
