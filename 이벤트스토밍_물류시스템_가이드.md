# 이벤트 스토밍 물류 시스템 가이드

## 1. 개요

### 1.1 이 가이드의 목적

물류 시스템을 대상으로 이벤트 스토밍 워크샵을 진행할 때, **퍼실리테이터와 참석자(협력사/계열사 담당자)가 자주 마주하는 특수한 상황**에 대한 실전 가이드입니다.

```
┌─────────────────────────────────────────────────────────────────┐
│              물류 시스템 워크샵의 핵심 도전 6가지                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 주문~배송~반품 전체 라이프사이클 이벤트 소유권 구분        │
│     "이 이벤트는 기간계인가요, 물류인가요?"                    │
│                                                                 │
│  2. 다중 택배사/다중 거점의 분기 처리 표현                     │
│     "택배사마다 다른 프로세스를 어떻게 통합하나요?"            │
│                                                                 │
│  3. 72개 EAI 인터페이스와 도메인 이벤트의 구분                 │
│     "EAI 인터페이스도 이벤트인가요?"                           │
│                                                                 │
│  4. 동시성/중복처리 이슈의 이벤트 표현                         │
│     "배송정보 동시요청 중복은 어떻게 표현하나요?"              │
│                                                                 │
│  5. WMS 외부 솔루션과의 이벤트 경계                            │
│     "WMS 출고확정은 우리 이벤트인가 WMS 이벤트인가?"           │
│                                                                 │
│  6. 반품/회수 프로세스의 복잡한 분기 처리                      │
│     "반품접수와 회수지시의 경계는?"                            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 물류 시스템의 특수성

```
┌─────────────────────────────────────────────────────────────────┐
│            일반 내부팀 vs 물류 시스템 비교                       │
├────────────────────────┬────────────────────────────────────────┤
│ 구분                   │ 물류 시스템                             │
├────────────────────────┼────────────────────────────────────────┤
│ 운영 주체              │ 협력사/계열사 (CJ대한통운 등)          │
├────────────────────────┼────────────────────────────────────────┤
│ 프로세스 범위          │ 주문 후 배송~반품까지 전체             │
├────────────────────────┼────────────────────────────────────────┤
│ 외부 시스템 의존       │ WMS, GLS, 6개 택배사, SAP 등           │
├────────────────────────┼────────────────────────────────────────┤
│ EAI 인터페이스         │ 72개 (물류 관련 약 30개)               │
├────────────────────────┼────────────────────────────────────────┤
│ 핵심 복잡성            │ 다중 택배사, 다중 거점, 동시성 이슈    │
├────────────────────────┼────────────────────────────────────────┤
│ 기간계 의존            │ 주문(OR), 상품(IMS)에서 데이터 수신    │
├────────────────────────┼────────────────────────────────────────┤
│ 정산 방식              │ 월별 배치 (SAP 연동)                   │
├────────────────────────┼────────────────────────────────────────┤
│ 특수 프로세스          │ 센터경유, 편의점배송, 가출고 관리      │
└────────────────────────┴────────────────────────────────────────┘
```

물류 시스템의 핵심 특수성:

- **주문~배송~반품 전체 라이프사이클**: 기간계에서 주문이 완료된 후부터 배송 완료, 반품/회수, 정산까지의 전체 프로세스를 다룸
- **다중 택배사/다중 거점**: CJ대한통운, 한진, 롯데, 우체국, 로젠, 일양로지스 등 6개 이상 택배사와 곤지암/안산/도심 거점 관리
- **72개 EAI 인터페이스**: 시스템 간 연동이 매우 복잡하며, 물류 관련 약 30개의 핵심 인터페이스가 존재
- **동시성/중복처리**: 배송정보 동시요청 시 중복 데이터 발생 방지를 위한 별도 프로세스 필요

### 1.3 6개 MSA 후보 서비스 개요

| 서비스 | 약어 | 프로세스 축 | 핵심 특성 |
|--------|------|------------|----------|
| 배송관리 (Fulfillment) | FU | 출하~배송완료 | 출하지시 생성, 배송구분 판단, 배송정보 전달 |
| 창고관리 (WMS) | WMS | 입고~출고 | WMS 출고처리, 입고확정, 재고관리, 양품화 |
| 택배사연동 (Courier) | COU | 배송지시~배송완료 | 택배사별 지시정보 전송, 배송정보 수신, 운송장 관리 |
| 반품/회수 (Returns) | RTN | 반품접수~환불 | 회수지시, 회수확정, 편의점회수, 반품비 처리 |
| 배송추적 (Tracking) | TRK | 출고~인수 | 배송상태 수신, 착지변경, 미자정보, 출고SMS |
| 물류정산 (Settlement) | STL | 정산 | 배송비/포장비 정산, 물류전표, SAP 연동 |

```
┌─────────────────────────────────────────────────────────────────┐
│                    배송 프로세스 흐름도                           │
└─────────────────────────────────────────────────────────────────┘

🟩 기간계(OR)                    물류 시스템 6개 서비스
주문완료 ──→ [FU] 배송관리 ──┬──→ [WMS] 창고관리 (센터배송)
                             │
                             ├──→ [COU] 택배사연동 ──→ [TRK] 배송추적
                             │
                             └──→ [STL] 물류정산 ──→ 🟩 SAP

🟩 기간계(OR)
반품접수 ──→ [RTN] 반품/회수 ──→ [FU] ──→ [COU] 택배사회수
                             └──→ [WMS] 반품입고 ──→ [STL] 정산
```

## 2. 물류 시스템 핵심 판단 기준

### 2.1 핵심 판단 기준

```
┌─────────────────────────────────────────────────────────────────┐
│                     핵심 판단 기준                               │
│                                                                 │
│  "배송 프로세스의 다른 단계에 영향을 주는가?"                  │
│                                                                 │
│  YES → 🟧 도메인 이벤트 (다른 서비스가 반응해야 함)            │
│  NO  → 서비스 내부 처리 (이벤트 아님)                          │
│                                                                 │
│  예시:                                                          │
│  ✅ "출하지시가 생성되었다" → WMS, COU가 반응 → 이벤트         │
│  ✅ "회수가 확정되었다" → 기간계 환불 처리 → 이벤트            │
│  ❌ "배송정보 임시 저장" → FU 내부 처리 → 이벤트 아님          │
│  ❌ "택배사 I/F 데이터 변환" → COU 내부 처리 → 이벤트 아님    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 판단 체크리스트 (4단계)

```
┌─────────────────────────────────────────────────────────────────┐
│                  4단계 판단 체크리스트                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Step 1: 비즈니스 의미가 있는가?                               │
│  ─────────────────────────────────────────────────────────────  │
│  □ 물류 담당자가 이해할 수 있는 용어인가?                      │
│  □ "출하지시가 생성되었다" ✅  vs "INSERT 실행됨" ❌            │
│                                                                 │
│  Step 2: 다른 서비스가 반응하는가?                              │
│  ─────────────────────────────────────────────────────────────  │
│  □ 이 사건 때문에 WMS/COU/TRK/RTN/STL이 뭔가 해야 하는가?     │
│  □ 기간계(OR)가 이 사건을 알아야 하는가?                       │
│                                                                 │
│  Step 3: 과거형으로 표현 가능한가?                              │
│  ─────────────────────────────────────────────────────────────  │
│  □ "~되었다" 형태로 자연스러운가?                              │
│  □ "배송상태가 변경되었다" ✅  vs "배송상태 조회" ❌            │
│                                                                 │
│  Step 4: 데이터 상태가 변경되는가?                              │
│  ─────────────────────────────────────────────────────────────  │
│  □ DB에 기록이 남는 변경인가?                                  │
│  □ 단순 조회나 화면 표시가 아닌가?                             │
│                                                                 │
│  4개 모두 ✅ → 🟧 도메인 이벤트                                │
│  1개라도 ❌ → 재검토 필요                                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 2.3 프로세스 축별 판단 플로우차트

```
┌─────────────────────────────────────────────────────────────────┐
│              프로세스 축별 이벤트 판단 플로우차트                │
└─────────────────────────────────────────────────────────────────┘

  [사건 발생]
      │
      ▼
  어느 프로세스 단계인가?
      │
      ├── 출하~배송 ──→ FU가 소유? ──┬─ Y → FU 이벤트 🟧
      │                              └─ N → WMS/COU 확인
      │
      ├── 입고~출고 ──→ WMS가 소유? ──┬─ Y → WMS 이벤트 🟧
      │                               └─ N → FU에서 지시한 것? → FU 이벤트
      │
      ├── 배송지시~완료→ COU가 소유? ──┬─ Y → COU 이벤트 🟧
      │                                └─ N → 택배사 외부 이벤트? → 🟩
      │
      ├── 반품~환불 ──→ RTN이 소유? ──┬─ Y → RTN 이벤트 🟧
      │                               └─ N → 기간계 반품접수? → 🟩
      │
      ├── 출고~인수 ──→ TRK가 소유? ──┬─ Y → TRK 이벤트 🟧
      │                               └─ N → 조회/읽기 모델? → 📖
      │
      └── 정산 ────→ STL이 소유? ──┬─ Y → STL 이벤트 🟧
                                    └─ N → SAP 시스템? → 🟩
```

### 2.4 서비스 간 이벤트 소유권 원칙

```
┌─────────────────────────────────────────────────────────────────┐
│              서비스 간 이벤트 소유권 3원칙                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  원칙 1: "데이터를 변경하는 서비스가 이벤트를 발행한다"        │
│  ─────────────────────────────────────────────────────────────  │
│  • 출하지시 데이터를 생성하는 FU가 "출하지시가 생성되었다" 발행│
│  • 출고 처리하는 WMS가 "출고처리가 완료되었다" 발행            │
│                                                                 │
│  원칙 2: "외부 시스템 이벤트는 수신 서비스가 변환 발행한다"    │
│  ─────────────────────────────────────────────────────────────  │
│  • 택배사가 보낸 배송정보 → COU가 수신 후                      │
│    "배송정보가 수신되었다" 발행                                 │
│  • WMS 솔루션의 출고완료 → WMS가 수신 후                       │
│    "출고처리가 완료되었다" 발행                                 │
│                                                                 │
│  원칙 3: "기간계와 물류의 경계는 EAI를 기준으로 나눈다"        │
│  ─────────────────────────────────────────────────────────────  │
│  • 기간계(OR)에서 EAI로 전달된 시점부터 물류 시스템 소유       │
│  • 물류에서 EAI로 전달한 시점에서 기간계로 소유 이전           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 2.5 외부 시스템(WMS, 택배사, GLS) 이벤트 경계

```
┌─────────────────────────────────────────────────────────────────┐
│           외부 시스템 이벤트 경계 판단                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  WMS (창고관리 솔루션):                                        │
│  • WMS 내부 처리 (피킹, 패킹 등) → WMS 서비스 내부             │
│  • WMS → 물류시스템 전달 (출고완료) → 🟧 WMS 이벤트           │
│  • 물류시스템 → WMS 전달 (출고지시) → 🟧 FU 이벤트            │
│                                                                 │
│  택배사 (CJ대한통운, 한진 등):                                 │
│  • 택배사 내부 처리 (집화, 간선 등) → 🟩 외부 시스템           │
│  • 택배사 → 물류시스템 전달 (배송상태) → 🟧 COU 이벤트        │
│  • 물류시스템 → 택배사 전달 (배송지시) → 🟧 COU 이벤트        │
│                                                                 │
│  GLS (CJ대한통운 통합):                                        │
│  • GLS 특화 API 연동 → COU 서비스의 어댑터로 처리             │
│  • GLS 고유 기능 → 🟩 외부 시스템으로 표시                     │
│                                                                 │
│  SAP (재무 시스템):                                            │
│  • SAP 전표 posting → 🟩 외부 시스템                           │
│  • 물류전표 생성 (posting 전) → 🟧 STL 이벤트                 │
│                                                                 │
│  편의점 (CVSnet/BGFpost):                                      │
│  • 편의점 접수/회수 → 🟩 외부 시스템                           │
│  • 편의점 회수 요청 → 🟧 RTN/COU 이벤트                       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 2.6 구체적 예시: 주문완료 → 출하지시 → 배송완료 → 반품 흐름

```
┌─────────────────────────────────────────────────────────────────┐
│           전체 흐름 이벤트 체인 예시                             │
└─────────────────────────────────────────────────────────────────┘

[고객 주문 완료]
   │
   ▼
🟩 기간계(OR): 주문이 완료되었다
   │
   ▼
🟧 FU: 배송정보가 전달되었다
   │
   ├──→ 💜 "배송구분 자동 판단" (센터/직송/편의점)
   │
   ▼
🟧 FU: 출하지시가 생성되었다
   │
   ├──→ [센터배송 Y] → 🟧 WMS: 출고지시가 접수되었다
   │                         │
   │                         ▼
   │                    🟧 WMS: 출고처리가 완료되었다
   │                         │
   │                         ▼
   ├──→ 🟧 COU: 택배사 지시정보가 전송되었다
   │         │
   │         ▼
   │    🟧 COU: 배송정보가 수신되었다
   │         │
   │         ▼
   │    🟧 TRK: 배송상태가 변경되었다 (집화→배달출발→배달완료)
   │         │
   │         ▼
   │    🟧 TRK: 배송이 완료되었다 → 🟩 기간계(OR) 상태 업데이트
   │
   │ (반품 발생 시)
   │
   ▼
🟩 기간계(OR): 반품이 접수되었다
   │
   ▼
🟧 RTN: 회수지시가 생성되었다
   │
   ├──→ 💜 "계약 택배사 설정 시 자동 회수지시"
   │

🟧 RTN: 회수가 확정되었다
   │
   ├──→ 🟧 WMS: 반품입고가 확정되었다
   └──→ 🟩 기간계(OR): 환불 처리
```
