# 이벤트 스토밍 프론트엔드팀 정책/규칙 분류 가이드

## 1. 개요

### 1.1 이 가이드의 목적

프론트엔드 팀 워크샵에서는 비즈니스 정책, 플랫폼 제약, UI/UX 규칙이 한꺼번에 쏟아져 나옵니다. 이 가이드는 **쏟아지는 "규칙"들을 체계적으로 분류**하여 스토밍의 효율을 높이기 위한 보충 자료입니다.

```
┌─────────────────────────────────────────────────────────────┐
│                    핵심 판단 원칙                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  "서버 상태가 바뀌면 정책, 화면 안에서만 영향이면 제외"    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 2. 3 레이어 분류 체계

프론트엔드 팀에서 나오는 "규칙"은 아래 3개 레이어로 분류합니다.

```
┌─────────────────────────────────────────────────────────────┐
│         프론트엔드에서 나오는 "규칙"의 3가지 레이어          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Layer 1: 💜 비즈니스 정책 (스토밍 대상)                    │
│  ─────────────────────────────────────                      │
│  → 백엔드와 무관하게 프론트에서 "먼저" 판단하더라도,       │
│    결국 비즈니스 상태를 바꾸는 규칙                         │
│  예) "재고 0이면 구매 버튼 비활성화"                        │
│  예) "미성년자는 주류 카테고리 접근 불가"                   │
│  예) "장바구니 50개 초과 시 추가 불가"                      │
│                                                             │
│  Layer 2: 🩷 핫스팟 (논의 대상, 정책은 아님)               │
│  ─────────────────────────────────────                      │
│  → 플랫폼/기술 제약으로 인한 규칙                          │
│  → 스토밍에서 "결정이 필요한 사항"으로 표시                 │
│  예) "iOS 인앱결제 수수료 정책"                             │
│  예) "오프라인 상태에서 찜하기 처리"                        │
│  예) "푸시 알림 권한 거부 시 대체 방안"                     │
│                                                             │
│  Layer 3: ❌ UI/UX 규칙 (스토밍 제외)                      │
│  ─────────────────────────────────────                      │
│  → 화면 표현에만 관련된 규칙                               │
│  예) "입력값 3자 이상일 때 검색 자동완성"                   │
│  예) "스크롤 80% 도달 시 다음 페이지 로드"                  │
│  예) "버튼 더블클릭 방지"                                   │
│  예) "토스트 메시지 3초 후 사라짐"                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.1 Layer 1: 💜 비즈니스 정책 (스토밍 대상)

**정의:** 프론트엔드에서 먼저 판단하더라도, 결국 **서버 상태를 변경**하거나 **백엔드 검증이 필수**인 규칙

**특징:**
- 서버에서도 동일하게 검증해야 함
- 위반 시 비즈니스에 직접적인 영향 (매출, 법적 문제 등)
- 다른 도메인/팀과 연동됨

| 예시 | 이유 |
|------|------|
| "재고 0이면 구매 버튼 비활성화" | 재고 도메인과 연동, 서버에서도 재고 검증 |
| "미성년자는 주류 카테고리 접근 불가" | 법적 규제, 서버에서 연령 검증 필수 |
| "장바구니 50개 초과 시 추가 불가" | 주문 도메인 제약, 서버에서도 검증 |
| "쿠폰 중복 사용 불가" | 프로모션 도메인 영향, 서버에서 쿠폰 상태 관리 |
| "3만원 이상 무료배송 안내 표시" | 배송비 계산 규칙과 연동 |
| "VIP 회원은 5% 추가 할인" | 회원 등급 도메인과 연동, 자동 적용 규칙 |
| "로그인 5회 실패 시 계정 잠금" | 보안 정책, 서버에서 시도 횟수 관리 |

**스토밍에서의 표현:**

```
  🟧 주문이 접수되었다  →  💜 VIP 할인 적용  →  🟧 할인이 적용되었다
  (트리거 이벤트)          (자동 정책)           (결과 이벤트)
```

### 2.2 Layer 2: 🩷 핫스팟 (논의 대상)

**정의:** 플랫폼/기술 제약으로 인한 규칙으로, **팀 간 협의가 필요**하지만 아직 정책으로 확정되지 않은 사항

**특징:**
- 플랫폼별 차이에서 비롯됨
- 결정 주체가 불명확 (보안팀? 백엔드? 기획?)
- 워크샵에서 바로 결론 내기 어려움

| 예시 | 협의 대상 |
|------|----------|
| "iOS 인앱결제 수수료 정책" | 사업팀 + 결제팀 |
| "오프라인 상태에서 찜하기 처리" | 백엔드 + 인프라팀 |
| "푸시 알림 권한 거부 시 대체 방안" | 기획팀 + 마케팅팀 |
| "앱 백그라운드 5분 후 세션 만료" | 보안팀 + 백엔드 |
| "검색 결과 캐싱 주기" | 검색팀 + 데이터팀 |
| "딥링크 미설치 사용자 처리" | 기획팀 + 마케팅팀 |

**스토밍에서의 표현:**

```
  🩷 "iOS 인앱결제 수수료 정책 — 사업팀과 협의 필요"
  🩷 "오프라인 찜하기 — 동기화 정책 미정"
```

> **핫스팟은 스토밍 보드에 붙여두되, 별도 목록으로 관리합니다.**
> 워크샵 후 해당 팀과 협의하여 💜 정책 또는 ❌ 제외로 확정합니다.

### 2.3 Layer 3: ❌ UI/UX 규칙 (스토밍 제외)

**정의:** **화면 표현에만 관련**된 규칙으로, 서버 상태나 비즈니스 로직에 영향을 주지 않는 사항

**특징:**
- 서버 상태와 무관
- 사용자 경험(UX)에만 영향
- 프론트엔드 내부에서 완결됨

| 예시 | 제외 이유 |
|------|----------|
| "입력값 3자 이상일 때 검색 자동완성" | UI 피드백, 서버 상태 무관 |
| "스크롤 80% 도달 시 다음 페이지 로드" | 페이지네이션 UX, 비즈니스 규칙 아님 |
| "버튼 더블클릭 방지" | UI 방어 로직 |
| "토스트 메시지 3초 후 사라짐" | 화면 표현 규칙 |
| "폼 입력 시 실시간 유효성 표시" | 즉각적 UI 피드백 |
| "무한스크롤 페이지 사이즈 20개" | 화면 표현 규칙 |
| "다크모드 활성화" | 로컬 UI 설정 |
| "애니메이션 전환 효과" | 시각적 표현 |

> **제외된 UI/UX 규칙은 별도 UX 백로그로 관리하세요.**
> 스토밍 보드에는 올리지 않습니다.

### 3.2 퍼실리테이터 핵심 질문

> **"이 규칙이 없으면 비즈니스에 문제가 생기나요, 아니면 사용자 경험만 나빠지나요?"**

| 대답 | 분류 |
|------|------|
| "비즈니스에 문제가 생깁니다" | 💜 정책으로 작성 |
| "UX만 나빠집니다" | ❌ 제외 (UX 백로그로 관리) |
| "잘 모르겠습니다" | 🩷 핫스팟으로 붙여두고 나중에 논의 |

## 4. 도메인별 정책/규칙 분류 예시

### 4.1 인증/회원 도메인

| 규칙 | 분류 | 판단 근거 |
|------|:---:|----------|
| "로그인 5회 실패 시 계정 잠금" | 💜 | 서버에서 시도 횟수 관리, 보안 정책 |
| "비밀번호 90일 경과 시 변경 권고" | 💜 | 서버에서 날짜 계산, 보안 규정 |
| "소셜 로그인 연동 해제 시 비밀번호 설정 필수" | 💜 | 인증 수단 변경, 서버 검증 필수 |
| "세션 만료 시간 (앱 백그라운드)" | 🩷 | 보안팀과 협의 필요 |
| "다중 기기 동시 로그인 허용 여부" | 🩷 | 보안팀 + 기획팀 협의 필요 |
| "자동 로그인 토큰 갱신 주기" | 🩷 | 백엔드 + 보안팀 협의 필요 |
| "비밀번호 입력 시 강도 표시 바" | ❌ | 즉각적 UI 피드백, 서버 상태 무관 |
| "로그인 폼 이메일 자동완성" | ❌ | 브라우저/OS 레벨 UI 기능 |

**혼동하기 쉬운 경계 사례:**

```
┌─────────────────────────────────────────────────────────────┐
│  "비밀번호 8자 이상 필수" → 어느 레이어?                    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  프론트에서 입력 시 실시간으로 글자 수를 검증하지만,       │
│  서버에서도 동일하게 검증해야 하는 규칙입니다.             │
│                                                             │
│  → 💜 비즈니스 정책                                        │
│  이유: 서버 검증이 필수이고, 위반 시 보안 문제 발생        │
│                                                             │
│  cf. "입력 시 실시간으로 글자 수 표시" 자체는 ❌ UI 규칙   │
│      하지만 "8자 이상이어야 한다"는 💜 비즈니스 정책       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 상품/카탈로그 도메인

| "품절 상품 구매 버튼 비활성화" | 💜 | 재고 도메인과 연동 |

## 5. 워크샵 실전 가이드

### 5.1 퍼실리테이터 핵심 질문 스크립트

#### 규칙이 쏟아질 때

```
상황: 참가자가 "이건 규칙인데요..." 하며 다양한 규칙을 나열할 때

대응:
"좋습니다! 규칙이 많이 나오고 있네요.
하나씩 분류해볼까요?

이 규칙에 대해 질문 하나만 드릴게요.

'이 규칙이 없으면 비즈니스에 문제가 생기나요,
아니면 사용자 경험만 나빠지나요?'

비즈니스 문제 → 💜 보라색 포스트잇
UX 문제만 → 별도 UX 목록에 기록
잘 모르겠다 → 🩷 핫스팟으로 붙여두죠"
```

#### 정책과 이벤트를 혼동할 때

```
상황: "VIP 할인 적용'은 이벤트인가요, 정책인가요?"

대응:
"좋은 질문입니다!

고객이 직접 행동한 것 → 커맨드 🟦 + 이벤트 🟧
시스템이 자동으로 적용하는 규칙 → 정책 💜

'쿠폰을 적용하기'는 고객의 커맨드, '쿠폰이 적용되었다'는 이벤트.
하지만 'VIP 5% 추가 할인'은 시스템이 자동 적용하니까 정책입니다.

보라색으로 적어볼까요?"
```

#### 이벤트 사이에 빠진 정책을 찾을 때

```
상황: 이벤트끼리 바로 연결하려고 할 때

대응:
"잠깐, 이 두 이벤트 사이에
자동 규칙이 있나요, 아니면 사람이 판단하나요?

'주문이 접수되었다' 다음에 바로 '배송이 시작되었다'?
그 사이에 자동으로 적용되는 규칙이 있을 수 있어요.

예를 들어 '결제 확인 후 자동 배송 접수' 같은
💜 정책이 빠진 건 아닌지 확인해봅시다."
```

### 5.2 흔한 실수 3가지

```
┌─────────────────────────────────────────────────────────────┐
│                  실수 1: UI 규칙을 정책으로 올림             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ❌ 잘못된 예:                                              │
│  💜 "버튼 더블클릭 방지"                                   │
│  💜 "토스트 메시지 3초 후 사라짐"                          │
│  💜 "무한스크롤 페이지 사이즈 20개"                        │
│                                                             │
│  → 이것들은 화면 안에서만 영향이 있는 UI 규칙              │
│  → 스토밍 보드에서 제거하고 UX 백로그로 이동               │
│                                                             │
│  교정 질문:                                                │
│  "이 규칙이 없어도 서버 데이터나 비즈니스에               │
│   문제가 생기지 않죠? 그렇다면 별도로 관리합시다."         │
│                                                             │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                  실수 2: 핫스팟을 정책으로 확정함            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ❌ 잘못된 예:                                              │
│  💜 "앱 백그라운드 5분 후 세션 만료"                       │
│  💜 "iOS 인앱결제 수수료 정책"                             │
│                                                             │
│  → 이것들은 아직 결정되지 않은 사항                        │
│  → 해당 팀과 협의 없이 정책으로 확정하면 안 됨             │
│  → 🩷 핫스팟으로 표시하고 협의 대상 팀을 명시             │
│                                                             │
│  교정 질문:                                                │
│  "이 규칙은 우리 팀이 단독으로 정할 수 있나요?            │
│   아니면 다른 팀과 협의가 필요한가요?"                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                  실수 3: 정책을 모두 핫스팟으로 미룸         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ❌ 잘못된 예:                                              │
│  🩷 "쿠폰 중복 사용 불가"  ← 이건 명확한 비즈니스 정책!   │
│  🩷 "재고 0이면 구매 불가"  ← 이것도 명확한 비즈니스 정책! │
│                                                             │
│  → 서버에서 반드시 검증하는 명확한 규칙까지               │
│    핫스팟으로 미루면 스토밍 결과가 빈약해짐                 │
│                                                             │
│  교정 질문:                                                │
│  "이 규칙은 서버에서도 검증하고 있죠?                      │
│   그렇다면 이건 확실한 비즈니스 정책입니다.                │
│   💜 보라색으로 바꿀까요?"                                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 5.3 정책 도출 타이밍

```
┌─────────────────────────────────────────────────────────────┐
│           워크샵에서 정책을 도출하는 최적 타이밍             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Phase 1~3: 이벤트 도출 + 타임라인 정렬 + 커맨드 추가      │
│  → 이 단계에서는 정책에 신경쓰지 않음                      │
│  → 이벤트와 커맨드에 집중!                                 │
│                                                             │
│  Phase 4: 애그리게잇 추가                                   │
│  → 이때 자연스럽게 "이건 자동 규칙인데..." 나옴            │
│  → 💜 정책으로 표시 시작                                   │
│                                                             │
│  Phase 5: 정책 도출 (핵심!)                                 │
│  → "이벤트 A → 이벤트 B 사이에 자동 규칙 있나요?"         │
│  → 이 질문으로 빠진 정책 집중 발굴                         │
│                                                             │
│  ※ 프론트엔드 팀은 Phase 4~5에서 UI 규칙이 많이 나옵니다  │
│     이때 3 레이어 분류법을 적극 활용하세요!                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 6. 자가 점검 체크리스트

워크샵 종료 후, 아래 항목을 확인하여 정책 분류가 올바르게 되었는지 점검합니다.

### 💜 비즈니스 정책 점검

```
□ 모든 💜 정책이 서버에서도 검증되는 규칙인가?
□ 💜 정책마다 트리거 이벤트(🟧)가 연결되어 있는가?
□ 💜 정책의 결과로 발생하는 이벤트(🟧)가 명시되어 있는가?
□ 프론트에서만 적용되는 UI 규칙이 💜에 섞여 있지 않은가?
□ "고객이 직접 행동" → 이벤트 vs "시스템 자동 적용" → 정책 구분이 명확한가?
```

### 🩷 핫스팟 점검

```
□ 모든 🩷 핫스팟에 "협의 대상 팀"이 명시되어 있는가?
□ 명확한 비즈니스 정책이 🩷로 잘못 분류된 건 없는가?
□ 워크샵 후 핫스팟 해소 일정이 잡혀 있는가?
□ 핫스팟이 해소되면 💜 또는 ❌로 재분류할 담당자가 정해져 있는가?
```

### ❌ 제외 항목 점검

```
□ ❌ 제외된 항목 중 실제로는 서버 연동이 필요한 것이 없는가?
□ 제외된 UI/UX 규칙이 별도 UX 백로그에 기록되었는가?
□ "서버에 저장 여부"가 불명확한 항목은 🩷 핫스팟으로 올렸는가?
```

### 전체 균형 점검

```
□ 💜 정책이 최소 5개 이상 도출되었는가?
  (너무 적으면 → 이벤트 사이에 빠진 자동 규칙 재탐색)
□ 🩷 핫스팟이 10개를 넘지 않는가?
  (너무 많으면 → 명확한 정책/제외로 재분류 시도)
□ ❌ 제외 항목이 별도 목록으로 관리되고 있는가?
```
