# 이벤트 스토밍 워크샵 스킬 사용 가이드

## 개요

이 프로젝트에는 이벤트 스토밍 워크샵의 **준비 → 분석 → 검증** 전 과정을 지원하는 3개의 Claude Code 스킬이 구성되어 있습니다.

| 스킬 | 슬래시 커맨드 | 역할 |
|------|-------------|------|
| **analyze-drawio** | `/analyze-drawio` | draw.io 파일을 분석하여 이벤트 스토밍 요소를 추출 |
| **workshop-prep** | `/workshop-prep` | 워크샵 준비 가이드 문서 + Mermaid 다이어그램 10개 생성 |
| **workshop-review** | `/workshop-review` | 작성된 준비 문서의 구조·포맷·정합성 검증 |

### 전체 워크플로우

```
┌──────────────────────────────────────────────────────────────────┐
│                     워크샵 준비 워크플로우                          │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ① /analyze-drawio        draw.io 파일 분석                      │
│         │                 색상 기반 요소 추출 + 오분류 감지          │
│         ▼                                                        │
│  ② /workshop-prep         준비 가이드 문서 생성                    │
│         │                 10개 섹션 + 10개 Mermaid 다이어그램       │
│         ▼                                                        │
│  ③ /workshop-review       문서 검증                               │
│                           파일·참조·포맷·섹션 정합성 확인            │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

## 빠른 시작

### 시나리오 1: draw.io 파일부터 시작

```
/analyze-drawio ./files/search-team.drawio
```
→ 분석 보고서를 확인한 후:
```
/workshop-prep 검색서비스팀 2차 (분석 결과를 붙여넣기)
```

### 시나리오 2: draw.io 파일 경로를 바로 전달

```
/workshop-prep 주문서비스팀 3차 ./files/order-team.drawio
```
→ 내부적으로 draw.io 분석을 수행한 뒤 문서를 생성합니다.

### 시나리오 3: 기존 문서 검증

```
/workshop-review ./이벤트스토밍_검색서비스팀_2차워크샵준비.md
```
→ 7개 카테고리의 검증 결과 보고서를 출력합니다.

## 스킬 1: `/workshop-prep` — 워크샵 준비 문서 생성

### 호출 방법

```
/workshop-prep [팀명] [N차] [draw.io 파일 경로 또는 분석 결과]
```

**인자 설명:**

| 인자 | 필수 | 예시 | 설명 |
|------|------|------|------|
| 팀명 | O | `검색서비스팀` | 한글 팀명 |
| N차 | O | `2차`, `3차` | 워크샵 차수 |
| draw.io 파일/분석 | O | `./files/team.drawio` | `.drawio` 파일 경로 또는 분석 결과 텍스트 |

### 5단계 워크플로우

| 단계 | 이름 | 내용 |
|------|------|------|
| Step 1 | draw.io 분석 | 색상 매핑으로 이벤트·커맨드·애그리게이트 등 추출 |
| Step 2 | 참조 문서 읽기 | 팀별 가이드, 도메인 예시, 이전 차수 문서 확인 |
| Step 3 | Mermaid 다이어그램 작성 | `.mmd` 파일 10개 생성 → PNG 렌더링 |
| Step 4 | 본문 작성 | 10개 섹션 구조에 따라 마크다운 문서 작성 |
| Step 5 | 검증 | workshop-review 체크리스트 기준 자가 검증 |

### 입출력 파일 규칙

**입력 (참조):**
- `이벤트스토밍_{팀명}_가이드.md` — 팀 특화 도전 과제
- `이벤트스토밍_{팀명}_도메인예시.md` — 도메인별 이벤트 예시
- `이벤트스토밍_{팀명}_워크샵실행.md` — 퍼실리테이터 스크립트
- `이벤트스토밍_읽기모델_가이드.md` — 읽기모델 프로세스
- `이벤트스토밍_시각화_가이드.md` — 포스트잇 색상·배치 패턴
- 이전 차수 문서: `이벤트스토밍_{팀명}_{N-1차}워크샵준비.md`

**출력:**
- `이벤트스토밍_{팀명}_{N차}워크샵준비.md` — 본문 문서
- `images/{팀명}_{N차}/diagram_01_현황요약.png` ~ `diagram_10_전체통합흐름.png` — 다이어그램 이미지 10개

### 10개 다이어그램

| # | 파일명 | Mermaid 타입 | 내용 |
|---|--------|-------------|------|
| 1 | `diagram_01_현황요약` | `pie` | 이전 차수 도출 요소 현황 (이벤트·커맨드·애그리게이트 비율) |
| 2 | `diagram_02_흐름영역맵` | `flowchart LR` | 흐름 영역 전체 맵 (영역 간 데이터 흐름) |
| 3 | `diagram_03_타임라인` | `gantt` | 워크샵 타임라인 (Phase별 시간 배분) |
| 4 | `diagram_04_애그리게이트_비포애프터` | `flowchart TB` | 애그리게이트 Before/After (오분류→정제) |
| 5 | `diagram_05_영역별매핑` | `flowchart TB` + subgraph | 영역별 애그리게이트-이벤트 매핑 |
| 6 | `diagram_06_정책후보맵` | `flowchart LR` | 정책 후보 맵 (이벤트→정책→결과 체인) |
| 7 | `diagram_07_핫스팟정책전환` 또는 `정책이벤트연결` | `flowchart TB/LR` | 차수에 따라 다름 (아래 참조) |
| 8 | `diagram_08_읽기모델후보` | `flowchart TB` | 읽기 모델 후보 (고객/운영자 분류) |
| 9 | `diagram_09_읽기모델이벤트연결` | `flowchart LR` | 읽기모델-이벤트 데이터 흐름 |
| 10 | `diagram_10_전체통합흐름` | `flowchart LR` + subgraph | 전체 요소 통합 흐름 |

### 10개 섹션 구조

| # | 섹션 제목 | 포함 다이어그램 | 핵심 포맷 |
|---|----------|--------------|----------|
| 1 | 개요 | — | ASCII 박스 (목표), 요약 테이블, 참조 문서 |
| 2 | 결과 정리 및 재검토 | #1 pie, #2 흐름맵 | 현황 분석, 영역별 설명 |
| 3 | 워크샵 타임라인 | #3 gantt | 상세 시간표 테이블 |
| 4 | Phase 1: 이벤트 정제 | — | 퍼실리테이터 스크립트, 통합/제외/재분류 가이드 |
| 5 | Phase 2: 애그리게이트 식별 | #4, #5 | 눈높이 설명, Before/After, 후보 상세 |
| 6 | Phase 3: 정책 도출 | #6, #7 | 눈높이 설명, 정책 테이블 |
| 7 | Phase 4: 읽기 모델 도출 | #8, #9 | 눈높이 설명, 모델 상세 |
| 8 | 전체 통합 및 정리 | #10 | 성과 체크리스트, 다음 단계 안내 |
| 9 | 퍼실리테이터 비상 대응 카드 | — | 상황-대응 카드 3~5개 |
| 10 | 결과물 템플릿 | — | 빈 기록 템플릿 |

### 차수별 차이

| 구분 | 2차 | 3차+ |
|------|-----|------|
| 이전 결과 참조 | 1차 결과 | 1~(N-1)차 전체 결과 |
| Phase 특성 | 이벤트 정제 + 애그리게이트 + 정책 + 읽기모델 | 정제 강화 + 오분류 교정 + BC 프리뷰 강조 |
| 다이어그램 7 | **정책-이벤트 연결** (`flowchart LR`) | **핫스팟→정책 전환** (`flowchart TB`) |
| 이벤트 정제 시간 | 20분 | 30분 (대폭 정제) |
| 섹션 1.2 제목 | `1차 요약 & 2차 목표` | `1~2차 요약 & 3차 목표` |
| 섹션 2 제목 | `1차 결과 정리 및 재검토` | `1~2차 결과 정리 및 재검토` |

## 스킬 2: `/analyze-drawio` — draw.io 분석

### 호출 방법

```
/analyze-drawio [draw.io 파일 경로]
```

**예시:**
```
/analyze-drawio ./files/search-team.drawio
```

### 실행 단계

| 단계 | 내용 |
|------|------|
| Step 1 | draw.io XML 파일 읽기 |
| Step 2 | `mxCell`의 `fillColor`에서 색상 추출 → 요소 분류 |
| Step 3 | 요소 유형별 집계 + x좌표 기준 타임라인 순서 추정 |
| Step 4 | 오분류 감지 (색상-이름 패턴 불일치 검사) |
| Step 5 | 구조화된 마크다운 보고서 출력 |

### 색상 → 요소 매핑 테이블

| 색상 범위 | 대표 코드 | 요소 | 포스트잇 |
|----------|----------|------|---------|
| 오렌지 계열 | `#FF8C00`, `#FF9800`, `#FFA500` | 도메인 이벤트 🟧 | 오렌지 |
| 파랑 계열 | `#4A90D9`, `#2196F3`, `#1565C0` | 커맨드 🟦 | 파란 |
| 노랑 계열 | `#FFD700`, `#FFEB3B`, `#FFC107` | 애그리게이트 🟨 | 노란 |
| 보라 계열 | `#9C27B0`, `#800080`, `#6A1B9A` | 정책 💜 | 보라 |
| 핑크 계열 | `#FF69B4`, `#E91E63`, `#C2185B` | 핫스팟 🩷 | 핑크 |
| 초록 계열 | `#2ECC71`, `#4CAF50`, `#2E7D32` | 외부 시스템 🟩 | 초록 |
| 하늘 계열 | `#E3F2FD`, `#87CEEB`, `#BBDEFB` | 읽기 모델 📖 | 하늘 |

> 정확한 코드 일치가 아니라 **유사 색상 범위**로 매칭합니다. 매핑 임계값(거리 50)을 초과하면 "미분류"로 표시됩니다.

### 오분류 감지 규칙

| 현재 분류 | 이름 패턴 | 의심 사유 | 제안 |
|----------|----------|----------|------|
| 🟨 애그리게이트 | `~됨`, `~실패`, `~완료됨` | 이벤트 이름 패턴 | 🟧 이벤트로 재분류 |
| 🟧 이벤트 | `~조회`, `~목록`, `~현황` | 읽기모델/커맨드 패턴 | 📖 읽기모델 또는 🟦 커맨드 |
| 🟨 애그리게이트 | `~검증`, `~확인`, `~판단` | 정책 패턴 | 💜 정책 |
| 🟧 이벤트 | `~하다`, `~요청하다` | 커맨드 패턴 (현재형) | 🟦 커맨드 |
| 🟦 커맨드 | `~됨`, `~되었다`, `~발생함` | 이벤트 패턴 (과거형) | 🟧 이벤트 |

**구조 기반 감지:**
- 같은 색상 포스트잇 10개 이상 → 세분화 필요 경고
- 이벤트 없이 커맨드만 있는 영역 → 이벤트 누락 경고
- 애그리게이트가 이벤트보다 많은 영역 → 과도한 분리 경고

### 출력 보고서 형식

```markdown
# draw.io 분석 결과: {파일명}

## 요소 현황 요약

| 요소 | 개수 | 비고 |
|------|------|------|
| 도메인 이벤트 🟧 | N개 | |
| 커맨드 🟦 | N개 | |
| ... | | |

## 요소별 목록

### 도메인 이벤트 🟧 (N개)
1. 이벤트명1
2. 이벤트명2
...

## ⚠️ 오분류 의심 항목

| # | 현재 분류 | 이름 | 의심 사유 | 제안 |
|---|----------|------|----------|------|
| 1 | 애그리게이트 🟨 | "주문이 취소됨" | "~됨"으로 끝남 | 이벤트로 재분류 |

## 흐름 영역 추정
1. 영역 1: ...
2. 영역 2: ...
```

## 스킬 3: `/workshop-review` — 문서 검증

### 호출 방법

```
/workshop-review [문서 파일 경로]
```

**예시:**
```
/workshop-review ./이벤트스토밍_주문서비스팀_3차워크샵준비.md
```

### 7개 검증 카테고리

| # | 카테고리 | 검증 항목 |
|---|---------|----------|
| 1 | **파일 존재** | 문서 파일, 이미지 디렉토리, PNG 10개 존재 확인 |
| 2 | **이미지 참조 정합성** | 문서 내 이미지 참조 ↔ 실제 파일 일치, 고아 파일 감지, 상대 경로 확인 |
| 3 | **details 태그** | `<details>` 10개, `📊 원본 Mermaid 코드 보기` summary, mermaid 코드 블록, 닫힘 태그 |
| 4 | **필수 섹션** | 10개 섹션 존재 및 필수 키워드 포함 여부 (아래 상세) |
| 5 | **포맷팅 규칙** | 퍼실리테이터 스크립트(`>`), 체크리스트(`- [ ]`), 테이블(5개+), ASCII 박스, 이모지 |
| 6 | **Mermaid 다이어그램** | 다이어그램 타입 순서, classDef 표준 색상, 노드 class 지정 |
| 7 | **내용 정합성** (선택) | draw.io 분석 결과 대비 이벤트·애그리게이트 수 일치, 수치 논리성 |

#### 필수 섹션 상세

| # | 섹션 | 필수 키워드 |
|---|------|-----------|
| 1 | `## 1. 개요` | 목적, 목표, 참조 문서 |
| 2 | `## 2. *차 결과 정리 및 재검토` | 현황 요약, 흐름 영역 |
| 3 | `## 3. *차 워크샵 타임라인` | gantt, Phase |
| 4 | `## 4. Phase 1: 이벤트 정제` | 퍼실리테이터 스크립트, 재검토 대상 |
| 5 | `## 5. Phase 2: 애그리게이트 식별` | 눈높이 설명, Before/After |
| 6 | `## 6. Phase 3: 정책 도출` | 정책 후보, 이벤트 연결 |
| 7 | `## 7. Phase 4: 읽기 모델 도출` | 읽기 모델 후보, 이벤트 연결 |
| 8 | `## 8. 전체 통합 및 정리` | 통합 흐름, 성과 체크리스트 |
| 9 | `## 9. 퍼실리테이터 비상 대응 카드` | 상황, 대응 |
| 10 | `## 10. 결과물 템플릿` | 일시, 참석자 |

### 출력 보고서 형식

```markdown
# 문서 검증 결과: {파일명}

## 요약
- ✅ 통과: N개
- ❌ 실패: N개
- ⚠️ 경고: N개

## 상세 결과

### 1. 파일 존재 — ✅/❌
(항목별 체크 결과)

### 2. 이미지 참조 — ✅/❌
(참조-실제 파일 매칭 결과)

...

## 권장 수정 사항
1. ...
2. ...
```

## FAQ / 트러블슈팅

### Mermaid 렌더링 실패 시

**증상:** `render-diagrams.sh` 실행 시 일부 다이어그램이 `❌ 실패`로 표시됨

**원인 및 해결:**
1. **npx 미설치** — Node.js 환경이 필요합니다. `node -v`로 확인하세요.
2. **mermaid 문법 오류** — 실패한 `.mmd` 파일을 열어 문법을 확인합니다. 한글 노드명에 따옴표(`"`)가 누락되면 파싱 에러가 발생합니다.
3. **Puppeteer 관련 오류** — mermaid-cli는 내부적으로 Puppeteer(headless Chrome)를 사용합니다. `npx -y @mermaid-js/mermaid-cli --help`로 설치 상태를 확인하세요.

**수동 렌더링:**
```bash
# 개별 파일 렌더링
npx -y @mermaid-js/mermaid-cli -i images/팀명_N차/diagram_01_현황요약.mmd -o images/팀명_N차/diagram_01_현황요약.png -w 1200 -b white -q

# 디렉토리 일괄 렌더링
bash .claude/skills/workshop-prep/scripts/render-diagrams.sh images/팀명_N차/
```

### 새 팀 추가 시 필요 작업

새로운 팀의 워크샵을 준비하려면 다음 파일들을 먼저 작성해야 합니다:

1. **`이벤트스토밍_{팀명}_가이드.md`** (필수) — 팀 도메인 특화 도전 과제, 판단 기준
2. **`이벤트스토밍_{팀명}_도메인예시.md`** (권장) — 도메인별 이벤트·커맨드·애그리게이트 예시
3. **`이벤트스토밍_{팀명}_워크샵실행.md`** (권장) — 퍼실리테이터 스크립트 기본안

> 이 파일들이 없어도 `/workshop-prep` 스킬은 동작하지만, 팀 맥락이 반영되지 않아 범용적인 문서가 생성됩니다.

### draw.io 파일이 여러 페이지인 경우

draw.io 파일은 여러 `<diagram>` 태그를 포함할 수 있습니다. `/analyze-drawio`는 모든 페이지를 순회하여 요소를 추출합니다. 특정 페이지만 분석하려면 draw.io에서 해당 페이지만 별도 파일로 내보내세요.

### 이전 차수 문서가 없는 경우 (1차 워크샵)

1차 워크샵은 이 스킬의 주요 대상이 아닙니다. 1차는 이벤트 발견(Discovery)이 중심이므로, 기존 가이드 문서(`이벤트스토밍_단계별_실행가이드.md`, `이벤트스토밍_워크샵_계획서.md`)를 직접 참조하세요. `/workshop-prep`은 **2차 이상**의 워크샵 준비에 최적화되어 있습니다.
