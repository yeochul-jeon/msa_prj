## 2. UI 이벤트 vs 비즈니스 이벤트 구분

### 2.1 핵심 원칙

```
┌─────────────────────────────────────────────────────────────┐
│                    핵심 판단 기준                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  "서버에 알려야 하는가?"                                   │
│                                                             │
│  예 → 비즈니스 이벤트 가능성 높음 🟧                       │
│  아니오 → UI/로컬 이벤트 (제외)                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 판단 기준 체크리스트

이벤트가 도메인 이벤트인지 판단할 때 아래 질문을 순서대로 확인하세요.

```
□ 1. 서버에 전달/저장되는가?
     → 아니오: UI 이벤트 (제외)
     → 예: 다음 질문으로

□ 2. 비즈니스적 의미가 있는가?
     → 아니오: 기술 이벤트 (제외)
     → 예: 다음 질문으로

□ 3. 다른 시스템/팀이 알아야 하는가?
     → 아니오: 로컬 이벤트일 가능성 (검토 필요)
     → 예: 도메인 이벤트! 🟧
```

### 2.5 퍼실리테이터 스크립트

#### "버튼 클릭을 이벤트로 적어도 되나요?"

```
대응:
"좋은 질문입니다. 버튼 클릭 자체보다는,
그 결과로 무슨 일이 '서버에서' 일어나는지 적어주세요.

예를 들어 '구매 버튼 클릭'이 아니라
'주문이 요청되었다'가 도메인 이벤트입니다.

버튼 클릭은 그 이벤트를 발생시키는 커맨드(🟦)에 가깝습니다."
```

#### "API 호출이 이벤트인가요?"

```
대응:
"API 호출은 커맨드(🟦)입니다.
이벤트는 그 API 호출의 '결과'로 일어난 일입니다.

예:
- 🟦 커맨드: '결제 요청하기'
- 🟧 이벤트: '결제가 승인되었다' 또는 '결제가 거부되었다'

API 호출 자체가 아니라, 그 결과 비즈니스적으로 무슨 일이
일어났는지를 적어주세요."
```

#### "화면 전환도 이벤트인가요?"

```
대응:
"화면 전환은 UI 네비게이션이라 도메인 이벤트가 아닙니다.

다만, 그 화면 전환이 '왜' 일어났는지 생각해보세요.
'결제가 완료되었다' → 결제 완료 화면으로 이동
여기서 이벤트는 '결제가 완료되었다'입니다.

화면 전환은 그 이벤트의 결과로 일어나는 UI 반응일 뿐입니다."
```

#### "로딩 상태도 이벤트인가요?"

```
대응:
"아니요, 로딩은 UI 상태입니다.

이벤트 스토밍에서는 '비즈니스적으로 의미 있는 상태 변화'만
다룹니다. 로딩 중/로딩 완료는 기술적 상태이지
비즈니스 이벤트가 아닙니다.

대신, 로딩이 끝난 후 무엇이 일어나는지가 이벤트입니다:
- '상품 목록이 조회되었다' ✅
- '로딩 완료' ❌"
```

## 3. 멀티 플랫폼 이벤트 추상화 (iOS, Android, Web)

### 3.1 핵심 원칙

```
┌─────────────────────────────────────────────────────────────┐
│                        핵심 원칙                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │

│  사용자가 iOS에서 결제하든, Web에서 결제하든               │
│  비즈니스 관점에서는 동일한 이벤트입니다:                  │
│                                                             │
│          🟧 "결제가 완료되었다"                            │
│                                                             │
```

### 3.2 공통 이벤트 vs 플랫폼 특화 이벤트

| 유형 | 예시 | 워크샵에서의 처리 |
|------|------|-------------------|
| **공통 이벤트** | 로그인 성공, 결제 완료, 장바구니 담기 | 🟧 하나의 이벤트로 표현 |
| **플랫폼 변형** | Face ID/지문/WebAuthn | 🟧 "생체인증 성공"으로 통합 |
| **플랫폼 특화** | Apple Pay만 지원하는 기능 | 🩷 핫스팟으로 표시 |

```
│  • 자연스럽게 공통 이벤트 도출                             │

│                                                             │

│                                                             │
│  피해야 할 구성: 플랫폼별 분리                             │

│  │ iOS 팀  │ │Android팀│ │ Web 팀  │  ← 각자 다른 이벤트   │

│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 플랫폼 차이 처리 방법

```
워크샵 중 플랫폼 차이 발견 시:

1단계: 즉시 핫스팟(🩷) 표시
   "iOS에서는 이렇게 동작해요" → 🩷 표시

2단계: 공통 결과에 집중하여 질문
   "결국 사용자가 얻는 결과는 무엇인가요?"

3단계: 추상화된 이벤트 도출
   "그러면 '인증이 성공했다'로 적어볼까요?"

4단계: 핫스팟에 플랫폼별 차이 메모
   🩷 "iOS: Face ID / Android: 지문 / Web: Passkey"
```

### 3.5 퍼실리테이터 스크립트

#### "iOS와 Android 플로우가 달라요"

```
대응:
"네, 플랫폼마다 UI 플로우가 다를 수 있습니다.
하지만 비즈니스 관점에서 '결과'는 같을 거예요.

예를 들어, iOS는 Face ID로, Android는 지문으로 인증하지만,
비즈니스 관점에서는 둘 다 '인증이 성공했다'입니다.

플랫폼별 차이점은 분홍색 핫스팟으로 표시해두고,
일단 공통 이벤트로 적어볼까요?"
```

#### "Web에는 이 기능이 없어요"

```
대응:
"좋은 지적입니다! 그 점은 핫스팟으로 표시해두죠.
```

## 8. FAQ 및 트러블슈팅

### 8.1 자주 묻는 질문

**Q: 프론트엔드 팀 워크샵에서 백엔드 API도 논의해야 하나요?**

A: 이벤트 스토밍에서는 **백엔드와의 연동 지점**을 식별하면 됩니다.
- "어떤 API를 호출하는가"가 아니라 "서버에서 어떤 일이 일어나는가"
- 연두색(🟩) 포스트잇으로 백엔드 시스템 연동 표시
- 상세 API 설계는 이후 단계에서 진행

---

**Q: 상태 관리(Redux, Recoil 등)와 관련된 것도 이벤트인가요?**

A: 아니요, 상태 관리는 **구현 상세**입니다.
- "Redux state가 업데이트됨"은 이벤트가 아님
- 그 결과로 비즈니스에서 무슨 일이 일어났는지가 이벤트
- 예: "장바구니에 상품이 담겼다"

---

**Q: Offline-first 기능은 어떻게 다루나요?**

A: 핫스팟(🩷)으로 표시하고 별도 논의합니다.
- 오프라인에서 발생한 것이 나중에 서버에 동기화되면 이벤트
- 예: "오프라인에서 찜한 상품이 동기화되었다"
- 🩷 "오프라인 동기화 충돌 처리 방법 검토 필요"

---

**Q: 프론트엔드에서만 처리되는 기능(예: 다크모드)도 포함해야 하나요?**

A: 서버에 저장/전달되지 않는 기능은 **제외**합니다.
- 다크모드, 폰트 크기 등 로컬 설정은 이벤트 아님
- 단, 서버에 저장되는 사용자 설정은 포함
- 예: "테마 설정이 서버에 저장되었다" (서버 동기화 시)

---

### 8.2 트러블슈팅

**상황 1: 참석자들이 계속 기술 용어를 사용할 때**

```
대응:
1. 부드럽게 리다이렉트:
   "좋습니다. 그 기술로 구현하면 결과적으로
    사용자에게 무슨 일이 일어나나요?"

2. 예시로 변환:
   "예를 들어, 'RxSwift로 구독'은
    '주문 상태가 실시간 업데이트됨'이 되겠네요."

3. 기술 논의는 핫스팟으로:
   "기술 선택에 대한 논의는 핫스팟으로 표시해두고,
    나중에 설계 단계에서 자세히 다루겠습니다."
```

**상황 2: 플랫폼팀 간 의견 충돌**

```
대응:
1. 양쪽 의견 모두 기록:
   "두 의견 모두 중요합니다.
    일단 핫스팟으로 표시해두죠."

2. 공통점 찾기:
   "구현 방식은 다르지만,
    사용자가 얻는 결과는 같지 않나요?"

3. 나중으로 미루기:
   "이 부분은 설계 단계에서 각 플랫폼팀과
    백엔드팀이 함께 논의하면 좋겠습니다."
```

**상황 3: UI 상태와 비즈니스 이벤트 구분이 안 될 때**

```
대응:
간단한 질문으로 확인:
1. "이 정보가 서버에 전달/저장되나요?"
   → 아니오: UI 상태 (제외)
   → 예: 다음 질문

2. "이 일이 일어난 것을 운영팀이 알아야 하나요?"
   → 예: 비즈니스 이벤트
   → 아니오: 아마도 UI 상태

3. 모르겠으면 핫스팟으로 표시
```

**상황 4: 이벤트가 너무 적게 나올 때**

```
대응:
유도 질문 활용:
1. "로그인 후 사용자가 할 수 있는 일을
    처음부터 끝까지 따라가 볼까요?"

2. "실패하는 경우는 어떤 것들이 있나요?"

3. "취소하거나 되돌리는 경우는요?"

4. "정기적으로 자동 실행되는 것은요?"
   (예: 세션 만료, 토큰 갱신)
```
