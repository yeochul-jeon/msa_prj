# 이벤트 스토밍 어드민/대고객 시스템 분류 가이드

## 1. 개요

### 1.1 이 가이드의 목적

이벤트 스토밍 워크샵 진행 시, **어드민(Back-Office) 시스템**과 **대고객(Customer-Facing) 시스템**을 어떻게 구분하고 스토밍 범위에 포함할지를 판단하기 위한 실전 가이드입니다.

```
┌─────────────────────────────────────────────────────────────┐
│           워크샵에서 자주 나오는 질문 TOP 5                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. "어드민에서 하는 상품 등록도 이벤트인가요?"            │
│                                                             │
│  2. "대시보드 조회 기능도 스토밍에 넣어야 하나요?"         │
│                                                             │
│  3. "어드민 승인 워크플로우는 별도 컨텍스트인가요?"        │
│                                                             │
│  4. "같은 기능인데 어드민/고객 화면이 다르면              │
│      이벤트를 따로 적나요?"                                │
│                                                             │
│  5. "CMS(콘텐츠 관리)는 어디에 넣어야 하나요?"            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 핵심 원칙

> **"시스템 종류(어드민/대고객)가 아닌, 비즈니스 이벤트 발생 여부가 기준이다."**

어드민 시스템이든 대고객 시스템이든, **비즈니스 상태를 변경**하는 행위라면 이벤트 스토밍의 대상입니다. 시스템의 UI가 다르다고 해서 도메인이 달라지는 것은 아닙니다.

```
┌──────────────────────────────────────────────────────────────┐
│                       핵심 원칙 3가지                         │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  ① 비즈니스 이벤트 중심 사고                                │
│     → "누가 했는가"보다 "무엇이 일어났는가"가 중요          │
│                                                              │
│  ② 액터 구분으로 충분한 경우가 많다                         │
│     → 같은 이벤트, 다른 액터 (👤 고객 vs 👨‍💼 운영자)        │
│                                                              │
│  ③ 분리 기준은 "고유 비즈니스 로직" 유무                    │
│     → 어드민만의 승인/검수/정산 로직이 있으면 별도 컨텍스트  │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

## 2. 핵심 판단 기준

### 2.1 비즈니스 상태 변경 여부 판단

어떤 기능이 이벤트 스토밍 대상인지를 판단하는 가장 중요한 기준은 **비즈니스 상태 변경(State Change)** 여부입니다.

```
┌─────────────────────────────────────────────────────────────┐
│              비즈니스 상태 변경 판단 기준                     │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ✅ 스토밍 대상 (비즈니스 상태가 변경됨)                    │
│  ─────────────────────────────────────                      │
│  • 데이터가 생성/수정/삭제되어 다른 도메인에 영향           │
│  • 승인/반려 등 워크플로우 상태가 전이                      │
│  • 외부 시스템과의 연동이 트리거                            │
│  • 사용자(고객/운영자)에게 알림이 발생                      │
│                                                             │
│  ❌ 스토밍 제외 (비즈니스 상태 변경 없음)                   │
│  ─────────────────────────────────────                      │
│  • 단순 조회/검색 (목록 조회, 상세 보기)                    │
│  • 리포팅/통계 대시보드                                     │
│  • UI 설정 변경 (테마, 레이아웃)                            │
│  • 단순 데이터 Export/Download                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 스토밍 대상 vs 제외 대상 분류

| 구분 | 어드민 기능 예시 | 대고객 기능 예시 | 스토밍 대상? |
|------|-----------------|-----------------|-------------|
| **비즈니스 상태 변경** | 상품 등록/수정 | 주문 생성 | ✅ 대상 |
| **승인 워크플로우** | 상품 승인/반려 | 환불 요청 | ✅ 대상 |
| **외부 연동 트리거** | 정산 실행 | 결제 요청 | ✅ 대상 |
| **단순 조회** | 주문 목록 조회 | 상품 목록 조회 | ❌ 제외 |
| **리포팅** | 매출 리포트 | - | ❌ 제외 |
| **설정 관리** | 시스템 설정 변경 | 알림 설정 | ⚠️ 상황에 따라 |

> **⚠️ "상황에 따라"인 경우:** 설정 변경이 비즈니스 로직에 영향을 준다면 대상입니다.
> 예: "무료배송 기준 금액 변경" → 주문 도메인에 영향 → ✅ 대상

## 3. 시스템 분류 방법 (3가지 패턴)

### 3.1 패턴 A: 같은 Bounded Context, 다른 액터

**가장 흔한 패턴.** 어드민과 대고객이 **동일한 비즈니스 이벤트**를 발생시키되, **액터(Actor)만 다른** 경우입니다.

```
┌─────────────────────────────────────────────────────────────┐
│              패턴 A: 같은 컨텍스트, 다른 액터                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  📦 상품 컨텍스트 (Product Context)                         │
│  ┌───────────────────────────────────────────────────┐      │
│  │                                                   │      │
│  │  👤 판매자 ──→ 🟦 상품 등록  ──→ 🟧 상품이       │      │
│  │  (대고객)       하기              등록되었다       │      │
│  │                     │                              │      │
│  │  👨‍💼 MD 담당 ──→ 🟦 상품 등록  ──→ 🟧 상품이     │      │
│  │  (어드민)       하기              등록되었다       │      │
│  │                                                   │      │
│  │  → 같은 이벤트, 같은 애그리게잇, 다른 액터        │      │
│  │  → 하나의 Bounded Context로 충분                  │      │
│  │                                                   │      │
│  └───────────────────────────────────────────────────┘      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**적용 조건:**

| 조건 | 설명 |
|------|------|
| 동일한 비즈니스 규칙 | 어드민/대고객 모두 같은 유효성 검증을 거침 |
| 동일한 결과 | 어느 쪽에서 실행해도 같은 도메인 이벤트 발생 |
| 권한 차이만 존재 | 기능은 같고 접근 권한만 다름 |

**워크샵에서의 표현 방법:**

```
┌───────────────────────────────────────────────────┐
│  포스트잇 배치 예시                                │
│                                                   │
│  👤 고객          🟦 주문하기    🟨 주문    🟧 주문이     │
│  👨‍💼 CS 담당자     (커맨드)      (Agg)     접수되었다   │
│  (액터 2명)                                       │
│                                                   │
│  → 액터를 2명 적되, 커맨드-이벤트는 하나만 작성   │
└───────────────────────────────────────────────────┘
```

**이 패턴에 해당하는 대표 예시:**

- 상품 등록/수정 (판매자 vs MD 담당자)
- 주문 취소 (고객 vs CS 담당자)
- 배송지 변경 (고객 vs CS 담당자)
- 회원 정보 수정 (고객 vs 운영자)

### 3.2 패턴 B: 별도 Bounded Context로 분리

어드민 시스템이 **고유한 비즈니스 로직**을 가지며, 대고객 시스템과 **독립적인 워크플로우**를 형성하는 경우입니다.

```
┌─────────────────────────────────────────────────────────────┐
│              패턴 B: 별도 Bounded Context 분리               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  📦 상품 컨텍스트               📦 상품 검수 컨텍스트       │
│  ┌─────────────────┐           ┌─────────────────┐         │
│  │                 │           │                 │         │
│  │ 🟧 상품이       │  ──────>  │ 🟧 검수가       │         │
│  │    등록되었다   │  (이벤트)  │    요청되었다   │         │
│  │                 │           │                 │         │
│  └─────────────────┘           │ 🟧 검수가       │         │
│                                │    승인되었다   │         │
│                                │                 │         │
│                                │ 🟧 검수가       │         │
│                                │    반려되었다   │         │
│                                │                 │         │
│                                │ 💜 검수 승인 시 │         │
│                                │    → 상품 활성화│         │
│                                │                 │         │
│                                └─────────────────┘         │
│                                                             │
│  → 검수 컨텍스트는 자체 워크플로우와 비즈니스 규칙 보유    │
│  → 별도 Bounded Context로 분리                             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**분리 조건 판단 테이블:**

| 분리 조건 | 설명 | 예시 |
|-----------|------|------|
| **승인 워크플로우** | 다단계 승인/반려 프로세스가 존재 | 상품 검수, 콘텐츠 승인, 프로모션 승인 |
| **고유 비즈니스 규칙** | 어드민에서만 적용되는 규칙이 3개 이상 | 정산 규칙, 수수료 계산, 판매자 등급 관리 |
| **독립적 생명주기** | 대고객 이벤트와 별개로 자체 상태 전이 | CMS 콘텐츠 발행 주기, 기획전 운영 주기 |
| **전문 액터** | 해당 기능만 담당하는 전문 역할 존재 | 정산 담당자, 검수 담당자, 콘텐츠 에디터 |
| **별도 SLA** | 대고객 시스템과 다른 응답시간/가용성 요구 | 정산 배치(일 1회) vs 주문(실시간) |

**별도 컨텍스트로 분리하는 대표 예시:**

| 어드민 전용 컨텍스트 | 핵심 이벤트 | 연관 대고객 컨텍스트 |
|---------------------|------------|---------------------|
| **상품 검수** | 검수 요청됨, 승인됨, 반려됨 | 상품 |
| **정산 관리** | 정산 생성됨, 확정됨, 지급됨 | 주문, 결제 |
| **콘텐츠 관리 (CMS)** | 콘텐츠 발행됨, 내려감 | 전시 |
| **프로모션 승인** | 프로모션 승인됨, 반려됨 | 프로모션 |
| **판매자 관리** | 판매자 승인됨, 정지됨 | 상품, 주문 |

### 3.3 패턴 C: 스토밍 제외 대상

비즈니스 상태를 변경하지 않는 **순수 조회/리포팅/설정** 기능은 이벤트 스토밍의 대상이 아닙니다.

```
┌─────────────────────────────────────────────────────────────┐
│              패턴 C: 스토밍 제외 대상                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ❌ 제외 항목                                               │
│  ──────────                                                 │
│  📊 대시보드/리포팅                                         │
│     • 매출 현황 대시보드                                    │
│     • 주문 통계 리포트                                      │
│     • 회원 분석 차트                                        │
│                                                             │
│  🔍 단순 조회 기능                                          │
│     • 주문 목록/상세 조회                                   │
│     • 상품 목록/상세 조회                                   │
│     • 회원 정보 조회                                        │
│                                                             │
│  ⚙️ 시스템 설정                                             │
│     • UI 테마 설정                                          │
│     • 알림 설정 (비즈니스 영향 없는 경우)                   │
│     • 관리자 계정 관리                                      │
│                                                             │
│  📥 데이터 Export                                           │
│     • CSV/Excel 다운로드                                    │
│     • 리포트 PDF 생성                                       │
│                                                             │
│  ⚠️ 예외: 아래의 경우 스토밍 대상이 될 수 있음             │
│  ──────────────────────────────────────────                  │
│  • "조회"지만 조회 기록이 비즈니스 규칙에 활용되는 경우     │
│    예: "고객이 상품을 조회했다" → 추천 알고리즘 입력        │
│  • "설정 변경"이 다른 도메인의 비즈니스 규칙을 바꾸는 경우  │
│    예: "무료배송 기준금액이 변경되었다" → 주문 도메인 영향  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 4. 분류 판단 플로우차트

워크샵에서 특정 기능의 스토밍 포함 여부와 분류 방법을 빠르게 판단하기 위한 플로우차트입니다.

```
                    ┌─────────────────────┐
                    │  이 기능은 비즈니스  │
                    │  상태를 변경하는가?  │
                    └─────────┬───────────┘
                              │
                   ┌──────────┴──────────┐
                   │                     │
                예 ▼                  아니오 ▼
         ┌─────────────┐          ┌──────────────┐
         │ 스토밍 대상  │          │ ❌ 스토밍     │
         │ 후보         │          │    제외       │
         └──────┬──────┘          │ (조회/리포팅/ │
                │                 │  설정 등)     │
                ▼                 └──────────────┘
     ┌────────────────────┐
     │ 어드민/대고객 모두  │
     │ 동일한 비즈니스     │
     │ 이벤트를 발생       │
     │ 시키는가?           │
     └─────────┬──────────┘
               │
    ┌──────────┴──────────┐
    │                     │
  예 ▼                아니오 ▼
┌──────────────┐   ┌──────────────────┐
│ 패턴 A       │   │ 어드민에 고유한   │
│ 같은 컨텍스트│   │ 비즈니스 로직이   │
│ 다른 액터    │   │ 있는가?           │
└──────────────┘   └────────┬─────────┘
                            │
                 ┌──────────┴──────────┐
                 │                     │
               예 ▼                아니오 ▼
        ┌──────────────┐    ┌──────────────┐
        │ 패턴 B       │    │ 패턴 A로     │
        │ 별도         │    │ 처리 가능    │
        │ 컨텍스트     │    │ (액터만 구분)│
        └──────────────┘    └──────────────┘
```

**플로우차트 사용 예시:**

## 6. 워크샵 권장 접근법

### 6.1 4단계 권장 프로세스

워크샵에서 어드민/대고객 시스템을 효과적으로 분류하기 위한 4단계 프로세스입니다.

```
┌─────────────────────────────────────────────────────────────┐
│              4단계 분류 프로세스                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  STEP 1: 이벤트 먼저, 시스템 구분은 나중에                  │
│  ────────────────────────────────────────                    │
│  • 처음에는 어드민/대고객 구분 없이 이벤트를 도출           │
│  • "누가"보다 "무엇이 일어났는가"에 집중                    │
│  • 🟧 오렌지 포스트잇에 이벤트만 나열                       │
│                                                             │
│                      ▼                                      │
│                                                             │
│  STEP 2: 액터 붙이기                                        │
│  ────────────────────                                       │
│  • 각 이벤트에 누가(액터) 트리거하는지 표시                 │
│  • 👤 고객 / 👨‍💼 운영자 / 🤖 시스템 구분                    │
│  • 하나의 이벤트에 여러 액터가 가능                         │
│                                                             │
│                      ▼                                      │
│                                                             │
│  STEP 3: 어드민 전용 이벤트 식별                            │
│  ──────────────────────────                                 │
│  • 어드민 액터만 트리거하는 이벤트를 별도 영역에 모음       │
│  • 승인/검수/정산 등 고유 워크플로우 확인                   │
│  • 🩷 핫스팟으로 "별도 컨텍스트?" 표시                      │
│                                                             │
│                      ▼                                      │
│                                                             │
│  STEP 4: 패턴 판단 및 컨텍스트 경계 확정                    │
│  ──────────────────────────────────────                     │
│  • 플로우차트(4장)를 활용하여 패턴 A/B/C 판단              │
│  • 패턴 B에 해당하면 별도 Bounded Context로 분리            │
│  • 컨텍스트 간 이벤트 흐름을 화살표로 연결                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 6.2 액터 구분 포스트잇 색상 안내

워크샵에서 액터를 시각적으로 구분하기 위한 색상 가이드입니다.

| 액터 유형 | 표시 방법 | 색상/아이콘 | 예시 |
|----------|----------|-----------|------|
| **고객 (Customer)** | 작은 포스트잇 | 👤 또는 연노랑 | 구매자, 방문자 |
| **운영자 (Admin)** | 작은 포스트잇 | 👨‍💼 또는 연파랑 | MD, CS 담당자, 운영팀 |
| **판매자 (Seller)** | 작은 포스트잇 | 🏪 또는 연초록 | 입점 판매자, 공급사 |
| **시스템 (System)** | 작은 포스트잇 | 🤖 또는 연회색 | 배치, 스케줄러, 외부 API |

```
┌─────────────────────────────────────────────────────────────┐
│  액터별 포스트잇 배치 예시                                   │

│                                                             │
│                 🟦 커맨드        🟨 Agg       🟧 이벤트     │

│                                                             │
```

## 7. 자가 점검 체크리스트

워크샵 종료 후, 어드민/대고객 시스템 분류가 적절한지 점검하기 위한 체크리스트입니다.

### 7.1 분류 적절성 점검

```
┌─────────────────────────────────────────────────────────────┐
│              자가 점검 체크리스트                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  □ 1. "어드민 컨텍스트"라는 이름의 거대 컨텍스트가         │
│       존재하지 않는가?                                      │
│       → 있다면 도메인별로 재분리 필요                       │
│                                                             │
│  □ 2. 모든 이벤트에 액터(고객/운영자/시스템)가              │
│       명시되어 있는가?                                      │
│       → 없다면 액터 표시 추가 필요                          │
│                                                             │
│  □ 3. 동일 이벤트가 어드민/대고객으로 중복 작성되지         │
│       않았는가?                                             │
│       → 중복이 있다면 하나로 통합하고 액터 2명 표시         │
│                                                             │
│  □ 4. 단순 조회/리포팅 기능이 이벤트에 포함되어             │
│       있지 않은가?                                          │
│       → 있다면 제거                                         │
│                                                             │
│  □ 5. 어드민 전용 컨텍스트(검수/정산/CMS 등)에             │
│       고유한 워크플로우(상태 전이)가 존재하는가?            │
│       → 없다면 패턴 A(같은 컨텍스트)로 통합 검토           │
│                                                             │
│  □ 6. 어드민 전용 컨텍스트와 대고객 컨텍스트 간의           │
│       이벤트 흐름(연동 지점)이 명확한가?                    │
│       → 불명확하면 컨텍스트 맵 보완 필요                    │
│                                                             │
│  □ 7. 비즈니스 상태를 변경하는 어드민 기능이 누락되지       │
│       않았는가? (승인, 정산, 콘텐츠 발행 등)               │
│       → 누락이 있다면 추가                                  │
│                                                             │
│  □ 8. 각 컨텍스트의 분리 근거가 "시스템 종류"가 아닌       │
│       "비즈니스 도메인"에 기반하는가?                       │
│       → 시스템 기준이라면 재검토                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 7.2 판단이 어려운 경계 사례

| 사례 | 판단 포인트 | 권장 분류 |
|------|-----------|----------|
| 어드민에서 고객 비밀번호 초기화 | 회원 상태 변경 → 비즈니스 이벤트 | 패턴 A (회원 컨텍스트, 운영자 액터) |
| 어드민 대량 상품 업로드 (엑셀) | 상품 등록과 동일 이벤트 발생 | 패턴 A (상품 컨텍스트, 배치 처리) |
| 어드민 프로모션 기간 설정 | 프로모션 상태 전이가 별도 존재 | 패턴 B (프로모션 관리 컨텍스트) |
| 어드민 로그 조회 | 비즈니스 상태 변경 없음 | 제외 |
| 어드민 배송비 정책 변경 | 주문 도메인 비즈니스 규칙에 영향 | 패턴 B (정책 관리 컨텍스트) |
| 어드민 쿠폰 일괄 발급 | 쿠폰 생성 이벤트 발생 | 패턴 A (쿠폰 컨텍스트, 운영자 액터) |

### 8.2 핵심 용어 정리

| 용어 | 정의 |
|------|------|
| **어드민 시스템 (Back-Office)** | 운영자/관리자가 사용하는 내부 관리 시스템 |
| **대고객 시스템 (Customer-Facing)** | 고객이 직접 사용하는 서비스 (웹, 앱) |
| **판매자 포털 (Seller Portal)** | 입점 판매자가 사용하는 시스템 (어드민과 대고객의 중간) |
| **Bounded Context** | 특정 도메인 모델이 일관성을 유지하는 경계 |
| **Context Map** | Bounded Context 간의 관계를 시각화한 다이어그램 |
| **액터 (Actor)** | 커맨드를 실행하는 주체 (고객, 운영자, 시스템 등) |

> **💡 이 가이드의 핵심 요약:** 어드민이냐 대고객이냐는 "시스템"의 문제이지 "도메인"의 문제가 아닙니다. 이벤트 스토밍은 도메인 중심으로 진행하되, 액터를 통해 시스템 차이를 표현하세요. 다만, 어드민에 고유한 비즈니스 워크플로우(승인/검수/정산 등)가 있다면 별도 Bounded Context로 분리하는 것이 MSA 설계에 유리합니다.
