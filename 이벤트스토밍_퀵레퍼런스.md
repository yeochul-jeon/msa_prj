# 이벤트 스토밍 퀵 레퍼런스

## 1분 요약

```
이벤트 스토밍이란?
→ 도메인 전문가와 개발자가 함께 비즈니스 프로세스를 시각화하는 워크샵

왜 필요한가?
→ MSA 전환을 위한 도메인 경계(바운디드 컨텍스트) 식별

어떻게?
→ 포스트잇으로 이벤트를 붙이고, 시간순 정렬하고, 그룹핑

결과물?
→ 마이크로서비스 후보 도출 + 부서 간 통합된 도메인 이해
```

## 30초 용어 정리

| 용어 | 뜻 | 포스트잇 | 예시 |
|------|-----|----------|------|
| **도메인 이벤트** | 시스템에서 일어난 의미있는 일 (과거형) | 🟧 | "주문이 접수되었다" |
| **커맨드** | 이벤트를 발생시키는 행동 (명령형) | 🟦 | "주문하기" |
| **애그리게잇** | 함께 변경되는 데이터 묶음 | 🟨 | "주문서" |

| **핫스팟** | 문제점, 불명확한 부분 | 🩷 | "배송 지연 문제" |

| **액터** | 커맨드를 실행하는 주체 | 아이콘 | 👤 고객, 🤖 시스템 |

## 5분 진행 순서

```
1. 이벤트 도출 (오렌지 포스트잇)
   - 생각나는 모든 이벤트 적기
   - 과거형으로 작성
   ↓

2. 타임라인 정렬
   - 시간 순서대로 배치
   - 왼쪽(시작) → 오른쪽(끝)
   ↓

3. 커맨드 추가 (파란 포스트잇)
   - 각 이벤트 왼쪽에 "무엇을 해서?"
   ↓

4. 애그리게잇 추가 (노란 포스트잇)
   - 어떤 데이터가 변경되었나?
   ↓

5. 정책 추가 (연보라 포스트잇)
   - 자동화된 규칙
   ↓

6. 바운디드 컨텍스트 식별
   - 관련 이벤트 묶기
   - 마이크로서비스 후보 도출
```

## 자주 하는 실수와 해결법

### ❌ 실수 1: 기술 용어로 이벤트 작성
```
나쁜 예:
"DB에 INSERT 되었다"
"API 호출했다"
"onClick 이벤트 발생"

좋은 예:
"주문이 저장되었다"
"결제가 요청되었다"
"주문 버튼이 클릭되었다"
```
**해결:** 비즈니스 관점에서 생각하기. "이것이 고객/업무에 무슨 의미인가?"

### ❌ 실수 2: 너무 세밀하거나 너무 추상적
```
너무 세밀:
"주문번호 생성됨" → "주문번호 검증됨" → "주문번호 저장됨"

너무 추상적:
"주문 처리됨"

적절한 수준:
"주문이 접수되었다" → "결제가 완료되었다" → "배송이 시작되었다"
```
**해결:** "이 이벤트가 다른 팀/시스템에 의미가 있나?" 질문하기

### ❌ 실수 3: 순서에 집착
```
초기에는 순서 무시!
처음부터 완벽하게 정렬하려 하지 마세요.
일단 많이 적고, 나중에 정렬하세요.
```

### ❌ 실수 4: 완벽주의
```
"이거 맞나요?"
"정답이 뭐죠?"
"완벽하게 해야 해요"

→ 정답은 없습니다!
→ 80% 완성도면 충분합니다
→ 틀려도 괜찮습니다 (핫스팟으로!)
```

### ❌ 실수 5: 논쟁 지속
```
"아니에요, 이건 이렇게 해야 해요!"
"우리 팀에서는 다르게 해요!"

→ 분홍 핫스팟으로 표시하고
→ 나중에 논의하기로
→ 일단 진행!
```

### 시간 관리 신호등

```
🟢 GREEN (여유)
- 참여도가 높고 진행 순조로움
- 계획보다 빠름
→ 더 깊은 논의 허용

🟡 YELLOW (주의)
- 시간이 예상보다 소요됨
- 토론이 길어짐
→ "이건 핫스팟으로 표시하고 넘어가죠"

🔴 RED (위험)
- 시간 초과 임박
- 핵심 단계 미완료 위험
→ "가장 중요한 프로세스에만 집중합시다"
```

### 에너지 관리

```
참석자 에너지 LOW 신호:
• 조용해짐
• 휴대폰 보기 시작
• 앉아있음
• 한숨

대응법:
→ 즉시 휴식 (5-10분)
→ 활동 변경 (앉아서 → 서서)
→ 성공 경험 공유 ("여기까지 잘하셨어요!")
→ 간단한 게임/아이스브레이킹
```

## 상황별 대응 스크립트

### 상황: "이거 왜 하는 거죠?"
```
답변:
"좋은 질문입니다. 우리가 MSA로 전환하려면 먼저
현재 시스템이 어떻게 동작하는지 명확히 이해해야 합니다.
이벤트 스토밍은 그것을 시각화하는 가장 효과적인 방법입니다."
```

### 상황: "너무 어려워요"
```
답변:
"괜찮습니다. 지금은 생각나는 것만 적어주세요.
완벽할 필요 없습니다. 함께하면서 점점 명확해질 거예요."

행동:
→ 옆에서 간단한 예시 보여주기
→ 질문으로 유도하기
```

### 상황: "우리 팀 프로세스와 다른데요?"
```
답변:
"매우 중요한 지적입니다. 분홍 포스트잇으로 표시해주세요.
부서 간 프로세스 차이를 발견하는 것이 이 워크샵의 핵심 목표입니다!"

→ 긍정적으로 받아들이기
→ 핫스팟으로 표시
```

### 상황: "시간이 부족한데요?"
```
답변:
"네, 시간을 체크해보니 조정이 필요하겠네요.
가장 중요한 [핵심 프로세스]에만 집중하고,
나머지는 다음 세션에서 다루겠습니다."

→ 우선순위 재조정
→ 2일차 계획 제안
```

## 성공 사례 패턴

### 성공한 워크샵의 공통점
```
✓ 경영진 지지
✓ 충분한 시간 확보 (조급하지 않음)
✓ 다양한 부서 참여
✓ 숙련된 퍼실리테이터
```

### 실패한 워크샵의 공통점
```
✗ 시간 부족 (2시간 이하)
✗ 일부 부서만 참여
✗ 결과물 방치
✗ 완벽주의 집착
✗ 기술 논의에 빠짐
✗ 후속 조치 없음
```

## 긴급 상황 대응

### 참석자가 절반만 온 경우
```
대응:
1. 온 사람들끼리라도 진행
2. 전체 범위 축소 (핵심 프로세스만)
3. 불참자 부서는 빈칸으로 남기고 표시
4. 후속 세션 계획
```

### 준비물을 깜빡한 경우
```
포스트잇 없음:
→ 종이 자르기, 디지털 도구 사용

마커 없음:
→ 볼펜, 연필로 대체

화이트보드 없음:
→ 벽에 큰 종이 붙이기
→ 바닥 활용
→ 온라인 도구로 전환
```

### 시간이 크게 부족한 경우
```
우선순위:
1. 핵심 프로세스 하나만 선택
2. 이벤트 도출 + 타임라인까지만
3. 사진 촬영
4. 다음 세션 약속

최소 목표:
"오늘 우리는 [핵심 프로세스]를 시각화했습니다"
```
