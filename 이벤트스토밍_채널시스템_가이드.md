# 이벤트 스토밍 채널 시스템 가이드

## 1. 개요

### 1.1 이 가이드의 목적

채널 시스템을 대상으로 이벤트 스토밍 워크샵을 진행할 때, **퍼실리테이터와 참석자(협력사 담당자)가 자주 마주하는 특수한 상황**에 대한 실전 가이드입니다.

```
┌─────────────────────────────────────────────────────────────────┐
│              채널 시스템 워크샵의 핵심 도전 7가지                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 7개 MSA 서비스 간 이벤트 소유권 구분                       │
│     "이 이벤트는 편성인가요, 온에어인가요?"                    │
│                                                                 │
│  2. 비실시간(편성) → 실시간(온에어) 전환점 표현                │
│     "편성 데이터가 실시간에 어떻게 반영되나요?"                │
│                                                                 │
│  3. Read 위주 서비스(온에어)의 이벤트 도출                     │
│     "조회만 하는데 이벤트가 있나요?"                           │
│                                                                 │
│  4. 기반 데이터(PDSS) 변경의 전파 범위                        │
│     "채널 코드 변경이 전체 서비스에 영향을 주나요?"            │
│                                                                 │
│  5. Fade out 가능성이 있는 서비스(매체, PGM) 범위 설정        │
│     "없어질 수 있는 서비스도 이벤트를 적어야 하나요?"          │
│                                                                 │
│  6. 내부 시스템(상품/주문)과의 이벤트 경계                    │
│     "방송 중 상품 정보는 누구의 이벤트인가요?"                │
│                                                                 │
│  7. 협력사(계열사) 참석자의 이벤트 스토밍 이해도 차이          │
│     "이벤트 스토밍을 처음 접하는 참석자가 많아요"             │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 채널 시스템의 특수성

> 채널 시스템은 일반 내부 팀과 달리, **외부 협력사가 관리**하며 **방송 시간축**을 따르는 특수한 도메인입니다.

```
┌─────────────────────────────────────────────────────────────────┐
│               일반 내부팀 vs 채널 시스템 비교                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  구분           │ 일반 내부팀        │ 채널 시스템              │
│  ───────────────────────────────────────────────────────────    │
│  운영 주체      │ 자사 개발팀        │ 외부 협력사(계열사)      │
│  시간 기준      │ 고객 행동 기반     │ 방송 시간축 기반         │
│  서비스 수      │ 1~2개              │ 7개 MSA 후보             │
│  연동 범위      │ 인접 팀 위주       │ 7개 서비스 + 내부 시스템 │
│  데이터 특성    │ CRUD 균등          │ Read 위주 (온에어)       │
│  Fade out       │ 거의 없음          │ 매체/PGM 가능성 있음     │
│  참석자 특성    │ 이벤트 스토밍 경험 │ 첫 경험 가능성 높음      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

```
┌─────────────────────────────────────────────────────────────────┐
│                    방송 시간축 개요도                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  시간축 무관     방송 전          방송 중          방송 후       │
│  (기반 데이터)  (비실시간)       (실시간)         (정산/분석)   │
│  ────────────────────────────────────────────────────────────   │
│                                                                 │
│  ┌──────┐    ┌──────┐┌──────┐   ┌──────┐     ┌──────┐┌──────┐ │
│  │ PDSS │    │ 편성 ││ 기획 │   │온에어│     │ 매체 ││ PGM  │ │
│  │      │    │      ││      │   │      │     │      ││실적  │ │
│  └──────┘    └──────┘└──────┘   └──────┘     └──────┘└──────┘ │
│                                                                 │
│                  ┌──────┐                                       │
│                  │ 타사 │ ← 독립 (시간축 무관)                  │
│                  │모니터│                                       │
│                  └──────┘                                       │
│                                                                 │

```

### 1.3 7개 MSA 후보 서비스 개요

| 서비스 | 약어 | 시간축 | 핵심 특성 | 핵심 키워드 |
|--------|------|--------|----------|------------|
| PDSS | BSS | 시간축 무관 | 기반 데이터, 채널 추가/확장성 | 마스터 데이터 |
| 편성 | BCS | 방송 전 | 비실시간, 편성팀 주관 | 편성표, 시간대 배치 |
| 방송제작 기획 | BCS | 방송 전 | 사전 프로세스, 별도 배포 가능 | 기획서, 출연자 |
| 온에어 | BOM | 방송 중 | Read 위주, 장애격리, 실시간 | 읽기 모델, 실시간 |
| 타사 모니터 | BCM | 독립 | 오프라인 생성, 낮은 연관성 | 경쟁사 분석, 이관 |
| 매체 | RFM | 방송 후 | 정산 메인, Fade out 가능성 | 매체비용, 정산 |
| PGM 실적/정산 | BCS | 방송 후 | 대량 데이터, 보안 | 방송실적, 정산 |

## 2. 채널 시스템 핵심 판단 기준

### 2.1 핵심 판단 기준

```
┌─────────────────────────────────────────────────────────────────┐
│                      핵심 판단 기준                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  "방송 프로세스의 다른 단계에 영향을 주는가?"                   │
│                                                                 │
│  예 → 도메인 이벤트 가능성 높음 🟧                              │
│  아니오 → 내부 처리 또는 단순 데이터 변경 (제외)                │
│                                                                 │
│  ─────────────────────────────────────────────────────────────  │
│                                                                 │
│  추가 판단 기준:                                                │
│  • 다른 서비스(7개 중)가 이 변화에 반응해야 하는가?             │
│  • 방송 시간축에서 다음 단계로 전달되어야 하는가?               │
│  • 내부 시스템(상품/주문)이 알아야 하는가?                      │

│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 판단 체크리스트 (4단계)

이벤트 후보가 도메인 이벤트인지 판단할 때 아래 질문을 순서대로 확인하세요.

```
□ 1. 방송 프로세스의 상태가 변경되는가?
     (편성 확정, 방송 시작, 실적 집계 등)
     → 아니오: 내부 처리 가능성 높음
     → 예: 다음 질문으로

□ 2. 다른 서비스가 이 변화에 반응해야 하는가?
     (편성→온에어, 온에어→PGM 등)
     → 아니오: 서비스 내부 이벤트 (제외 가능)
     → 예: 다음 질문으로

□ 3. 시간축 경계를 넘어가는가?
     (방송 전→방송 중, 방송 중→방송 후)
     → 예: 핵심 도메인 이벤트! 🟧
     → 아니오: 다음 질문으로

□ 4. 이 변화가 방송 운영이나 정산에 영향을 주는가?
     → 예: 도메인 이벤트 🟧
     → 아니오: 내부 데이터 변경 (제외)
     → 모르겠다: 🩷 핫스팟으로 표시
```

### 2.4 서비스 간 이벤트 소유권 원칙

```
┌─────────────────────────────────────────────────────────────────┐
│                   이벤트 소유권 원칙                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  원칙 1: "데이터를 변경하는 서비스가 이벤트를 발행한다"         │
│  ─────────────────────────────────────────────────────────────  │
│  • 편성표를 변경하는 건 편성 서비스                             │
│    → "편성표가 확정되었다" = 편성 서비스의 이벤트               │
│  • 방송 시작/종료를 처리하는 건 온에어 서비스                   │
│    → "방송이 시작되었다" = 온에어 서비스의 이벤트               │
│                                                                 │
│  원칙 2: "시간축 경계의 이벤트는 상류 서비스가 발행한다"        │
│  ─────────────────────────────────────────────────────────────  │
│  • 편성(방송 전) → 온에어(방송 중) 경계:                       │
│    "편성표가 확정되었다" = 편성 서비스 발행, 온에어가 구독      │
│  • 온에어(방송 중) → PGM(방송 후) 경계:                        │
│    "방송이 종료되었다" = 온에어 발행, PGM이 구독                │
│                                                                 │
│  원칙 3: "참조 데이터 변경은 원본 소유자의 이벤트"              │
│  ─────────────────────────────────────────────────────────────  │
│  • PDSS의 채널 코드 변경 → PDSS 이벤트                         │
│  • 내부 상품팀의 가격 변경 → 상품팀 이벤트                     │
│  • 온에어는 이들을 구독해서 읽기 모델 갱신                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

│  "채널에서 편성하고 방송하고 정산한다"                          │
│  → 하나의 큰 흐름으로 이벤트를 나열                            │
│  → 서비스 간 경계가 불명확                                     │
│                                                                 │
│  ✅ 올바른 예                                                  │
│  "편성 서비스에서 편성표를 확정한다"                            │
│  "온에어 서비스에서 방송을 시작한다"                            │
│  "PGM 서비스에서 실적을 집계한다"                              │
│  → 서비스별 경계를 명확히 하고 연동 이벤트를 도출              │
│                                                                 │
│  교정 방법:                                                     │
│  "이 이벤트를 처리하는 서비스가 어디인가요?                     │
│   편성인가요? 온에어인가요? 서비스 이름을 붙여주세요."          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 실수 2: 온에어의 Read를 Write 이벤트로 표현

│  ❌ 잘못된 예                                                  │

│  🟧 "시청률이 업데이트되었다"                                  │
│  → 조회(Read)를 이벤트(Write 결과)로 혼동                      │
│                                                                 │
│  ✅ 올바른 예                                                  │
│  📖 "방송상품정보 읽기모델" (편성/상품 이벤트에서 생성)        │
│  📖 "실시간주문현황 읽기모델" (주문 이벤트에서 생성)           │
│  📖 "실시간시청률 읽기모델" (시청률 데이터에서 생성)           │
│  → 읽기 모델로 표현하고, 원본 이벤트와 연결                    │
│                                                                 │
│  교정 방법:                                                     │
│  "이건 조회(보여주기)인가요, 변경(상태 바꾸기)인가요?           │
│   조회라면 읽기 모델로 표현합시다."                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```
